Что нужно построить

React Native (Expo) приложение для просмотра и фильтрации 259 футбольных команд. Данные приходят с HackerRank API. Пользователь может искать команды, фильтровать по стоимости и титулам, а также вычислять расстояние до стадиона через GPS.

Шаг 1: TypeScript Interfaces

Первое, что нужно сделать -- определить типы данных. В iOS это был бы Codable struct. В TypeScript -- interface.

День 2: TypeScript patterns
Swift (Codable)
struct Team: Codable { let name: String let league: String let estimatedValue: String let estimatedValueNumeric: Int let numberOfLeagueTitlesWon: Int let stadiumName: String let stadiumCapacity: Int }
TypeScript (interface)
export interface Team { name: string; league: string; estimated_value: string; estimated_value_numeric: number; number_of_league_titles_won: number; stadium_name: string; stadium_capacity: number; // ... 20+ полей от API }
Все типы в одном файле types/team.ts -- Team, TeamsApiResponse, FilterState, NominatimResult, DistanceResult. Это чистая архитектура: типы отдельно от логики.

Шаг 2: Загрузка данных (useTeams)

Custom hook который делает то же, что URLSession + @Published в iOS. Загружает данные при mount, управляет loading/error/retry.

День 1: useState + useEffect + Custom Hooks
// hooks/useTeams.ts export function useTeams(): UseTeamsResult { const [teams, setTeams] = useState<Team[]>([]); // @State var teams: [Team] = [] const [loading, setLoading] = useState(true); // @State var loading = true const [error, setError] = useState<string | null>(null); const load = useCallback(async () => { setLoading(true); setError(null); try { const data = await fetchAllTeams(); // URLSession.shared.data(from:) setTeams(data); } catch (e) { setError(e instanceof Error ? e.message : "Failed to load teams"); } finally { setLoading(false); } }, []); useEffect(() => { load(); }, [load]); // .onAppear { load() } return { teams, loading, error, retry: load }; }
Паттерн useCallback + useEffect -- стандартный способ загрузки данных в React. Хук useCallback мемоизирует load(), а useEffect вызывает его при mount (пустой массив зависимостей). Retry -- просто повторный вызов load().

Шаг 3: API Layer (fetch + parallel)

259 команд на 26 страницах. Стратегия: загрузить первую страницу, узнать total_pages, параллельно загрузить остальные.

День 1: async/await
// utils/api.ts export async function fetchAllTeams(): Promise<Team[]> { const firstPage = await fetchTeamsPage(1); const { total_pages, data } = firstPage; if (total_pages <= 1) return data; // Параллельная загрузка всех страниц const remaining = Array.from( { length: total_pages - 1 }, (_, i) => fetchTeamsPage(i + 2) ); const pages = await Promise.all(remaining); // DispatchGroup / TaskGroup const allTeams = [...data, ...pages.flatMap(p => p.data)]; return allTeams.sort( (a, b) => b.estimated_value_numeric - a.estimated_value_numeric ); }
Promise.all -- аналог Swift TaskGroup. Все 25 оставшихся страниц загружаются одновременно. В iOS ты бы использовал async let или withTaskGroup.

Шаг 4: FlatList (основной список)

Аналог UITableView/UICollectionView. Виртуализирует элементы -- рендерит только видимые.

День 1: FlatList
// app/index.tsx <FlatList data={filteredTeams} renderItem={renderItem} // cellForRowAt keyExtractor={keyExtractor} // reuseIdentifier ListHeaderComponent={<FilterBar />} // tableHeaderView ListEmptyComponent={<EmptyState />} // backgroundView при пустом списке ListFooterComponent={...} // tableFooterView contentContainerStyle={styles.listContent} initialNumToRender={10} // prefetchDataSource maxToRenderPerBatch={10} keyboardDismissMode="on-drag" // .scrollDismissesKeyboard />
renderItem и keyExtractor обёрнуты в useCallback! Без мемоизации FlatList будет пересоздавать ячейки при каждом ре-рендере родителя. Это как если бы UITableView вызывал dequeueReusableCell каждый кадр.

Шаг 5: Компоненты (TeamCard)

Каждый компонент -- это View в SwiftUI. Props -- это @Binding. Callbacks -- это closures.

Шаг 6: Фильтрация (useFilters + useMemo)

Клиентская фильтрация с debounce поиска. useMemo пересчитывает только когда меняются зависимости -- как @State + computed property.

День 1: useMemo + useCallback
// hooks/useFilters.ts export function useFilters(teams: Team[]): UseFiltersResult { const [filters, setFilters] = useState<FilterState>(DEFAULT_FILTERS); const [searchText, setSearchTextRaw] = useState(""); const debounceRef = useRef<ReturnType<typeof setTimeout>>(); // Debounced search (300ms) const setSearchText = useCallback((text: string) => { setSearchTextRaw(text); // мгновенно обновить TextInput if (debounceRef.current) clearTimeout(debounceRef.current); debounceRef.current = setTimeout(() => { setFilters(prev => ({ ...prev, searchQuery: text })); }, 300); // отложить фильтрацию }, []); // Мемоизированная фильтрация -- пересчёт ТОЛЬКО при изменении teams или filters const filteredTeams = useMemo(() => { const query = filters.searchQuery.toLowerCase(); return teams.filter(team => team.name.toLowerCase().includes(query) && team.estimated_value_numeric >= filters.minValue && team.number_of_league_titles_won >= filters.minTitles ); }, [teams, filters]); return { filteredTeams, filters, searchText, ... }; }
Debounce = Combine's .debounce(for:). Пользователь печатает -- TextInput обновляется мгновенно, но фильтрация срабатывает через 300ms после последнего нажатия. useRef хранит timeout ID между рендерами (не вызывая ре-рендер).

Шаг 7: Filter Modal (Local State Pattern)

Модальное окно с локальным состоянием. Изменения применяются только по нажатию "Apply" -- не при каждом тапе на chip.

День 2: State Management
// components/FilterModal.tsx export function FilterModal({ visible, onClose, minValue, minTitles, onApply }: Props) { const [localValue, setLocalValue] = useState(minValue); const [localTitles, setLocalTitles] = useState(minTitles); const handleOpen = () => { setLocalValue(minValue); // Синхронизация при открытии setLocalTitles(minTitles); }; const handleApply = () => { onApply(localValue, localTitles); // Применяем только при нажатии Apply onClose(); }; return ( <Modal visible={visible} onShow={handleOpen}> {/* Value chips */} {VALUE_STEPS.map((step, i) => ( <Pressable onPress={() => setLocalValue(step)} style={[styles.chip, localValue === step && styles.chipActive]}> <Text>{VALUE_STEP_LABELS[i]}</Text> </Pressable> ))} {/* Title stepper (- / value / +) */} {/* Apply button */} </Modal> ); }
Local state в модалке -- важный паттерн! Без него список пере-фильтровался бы при каждом тапе на chip. Это было бы медленно (259 элементов * 3 фильтра) и визуально дёрганым. В iOS аналог: sheet с @State внутри, .onAppear для синка.

Шаг 8: Геолокация (expo-location)

По кнопке: запросить GPS разрешение → получить координаты → геокодировать стадион → посчитать расстояние.

День 3: Location Services
// app/index.tsx const userLocationRef = useRef<{ lat: number; lon: number } | null>(null); const handleDistancePress = useCallback(async (team: Team) => { setDistances(prev => ({ ...prev, [team.name]: { value: null, loading: true, error: null } })); // 1. Получить GPS (кэшируется в ref) if (!userLocationRef.current) { const { status } = await Location.requestForegroundPermissionsAsync(); if (status !== "granted") { /* set error */ return; } const loc = await Location.getCurrentPositionAsync({}); userLocationRef.current = { lat: loc.coords.latitude, lon: loc.coords.longitude }; } // 2. Геокодировать стадион (кэш в Map) const coords = await geocodeStadium(team.stadium_name); // 3. Посчитать расстояние (haversine) const dist = haversine( userLocationRef.current.lat, userLocationRef.current.lon, coords.lat, coords.lon ); setDistances(prev => ({ ...prev, [team.name]: { value: dist, loading: false, error: null } })); }, []);
Три уровня кэширования: userLocationRef (GPS -- один запрос за сессию), geocodeCache (Map в api.ts -- один запрос на стадион), distances state (UI -- результат сохраняется). В iOS: CLLocationManager + NSCache.

Шаг 10: Accessibility

Onward оценивает accessibility. В assessment каждый интерактивный элемент имеет роль и label.

// TeamCard -- accessibilityRole + dynamic label <View accessibilityRole="summary" accessibilityLabel={`${team.name}, valued at ${team.estimated_value}, ${team.number_of_league_titles_won} league titles`} > // Distance button -- состояние меняет label <Pressable accessibilityRole="button" accessibilityLabel={ distanceText ? `${team.stadium_name} is ${distanceText} away` : `Calculate distance to ${team.stadium_name}` } disabled={distanceResult?.loading} > // FilterModal -- chips с accessibilityState <Pressable accessibilityRole="button" accessibilityState={{ selected: localValue === step }} accessibilityLabel={`Minimum value ${VALUE_STEP_LABELS[i]}`} >
iOS аналоги: accessibilityRole = .accessibilityTraits, accessibilityLabel = .accessibilityLabel, accessibilityState = .isSelected. Минимум 44pt tap targets (minHeight: 44 / minWidth: 44 в StyleSheet). WCAG AA контраст на всех текстах.

Шаг 11: Error & Empty States

Три состояния экрана: Loading → Error | Content (Empty | List). Каждое должно быть красивым и информативным.

День 2: Error Boundaries
// app/index.tsx -- три ветки render if (loading) { return ( <View style={styles.center}> <ActivityIndicator size="large" color="#4A90D9" /> <Text>Loading teams...</Text> </View> ); } if (error) { return <ErrorState message={error} onRetry={retry} />; } return ( <FlatList ListEmptyComponent={ <EmptyState hasFilters={activeFilterCount > 0} onReset={resetFilters} /> } /> );
EmptyState умный: если фильтры активны -- показывает "Try adjusting your search" + кнопку Reset. Если данных нет вообще -- "Check your connection". Это production-level UX.

Шаг 13: Trade-offs (что спросят на review)

Assessment оценивает не только код, но и мышление. Будь готов объяснить:

"Почему fetch all upfront, а не pagination?"
259 команд = ~100KB. API не поддерживает серверную фильтрацию. При клиентской фильтрации нужны все данные. Pagination добавила бы сложность без выгоды. При 10K+ элементов -- да, нужен серверный фильтр.
"Почему chips, а не slider для value?"
Диапазон $3M-$2.5B нелинейный. На линейном слайдере 90% значений заняли бы 2% полоски. Chips интуитивнее и accessible (44pt targets, VoiceOver friendly).
"Почему нет Redux/Zustand?"
Один экран, 3 фильтра, один массив. useState + useMemo достаточно. Добавлять state management library -- over-engineering. Показывает, что ты знаешь, когда НЕ использовать инструмент.
"Почему distance on-demand?"
Geocoding 259 стадионов при mount: ~4 минуты при 1 req/sec (Nominatim policy). Плохой UX. Per-button-press + cache -- правильный баланс.
