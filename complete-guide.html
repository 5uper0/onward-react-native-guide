Главное отличие React от SwiftUI — это модель ре-рендеринга. В SwiftUI система использует property wrappers (@State, @Binding) чтобы точечно отслеживать, какие View зависят от каких данных, и обновлять только их. React работает иначе: при изменении состояния компонент-функция вызывается целиком заново, возвращает новое дерево JSX, и React сравнивает его с предыдущим (reconciliation). Это означает, что каждый рендер — это полный вызов функции компонента сверху донизу.

Для iOS разработчика это ключевой сдвиг в мышлении. В UIKit ты явно мутировал view (label.text = "new"). В SwiftUI фреймворк сам решал, что перерисовать через diffing some View. В React ты описываешь UI как чистую функцию от состояния: UI = f(state). Каждый раз когда state меняется, функция вызывается снова и React эффективно обновляет DOM через Virtual DOM diffing.

Понимание этого цикла рендеринга — фундамент всего остального. Все хуки (useState, useEffect, useMemo, useCallback) — это способы управлять поведением внутри этого цикла. Если ты понимаешь, что функция компонента вызывается при каждом изменении состояния, все остальное встанет на свои места.

🍎 SwiftUI / iOS
@State → состояние View @Binding → передача состояния вниз @Published → реактивные данные ObservableObject → shared state View → компонент UI ViewModifier → HOC / styled component onAppear → lifecycle hook Combine → реактивные потоки
⚛️ React Native
useState → состояние компонента props → передача данных вниз Context/Redux → shared state React.FC → функциональный компонент styled-components → стилизация useEffect → lifecycle hook RxJS / React Query → реактивные потоки
React — это декларативный UI, как SwiftUI. Ты описываешь ЧТО показать, не КАК. Но в отличие от SwiftUI, где diffing происходит на уровне View struct, в React diffing происходит на уровне Virtual DOM — легковесного JavaScript-представления UI дерева. React сравнивает предыдущее и новое дерево, вычисляет минимальный набор изменений и применяет их к реальному UI.
Частая ошибка iOS разработчиков: думать, что React компонент "живет" между рендерами как объект. Нет — функция вызывается заново каждый раз. Все локальные переменные пересоздаются. Только useState и useRef сохраняют значения между рендерами (React хранит их в отдельной структуре, привязанной к позиции компонента в дереве).

📝 JavaScript за 10 минут (для Swift разработчика)

JavaScript и Swift внешне похожи, но фундаментально отличаются в системе типов. Swift — строго типизированный язык с compile-time проверками. JavaScript — динамически типизированный с неявным приведением типов (type coercion). Это значит, что "5" + 3 в JS даст "53" (строка!), а "5" - 3 даст 2 (число!). Такие сюрпризы — главная причина, почему в React Native проектах всегда используют TypeScript.

TypeScript добавляет статическую типизацию поверх JavaScript, но она стирается при компиляции — в runtime типов нет. Это значит, что TypeScript ловит ошибки на этапе написания кода (как Swift), но не защищает от невалидных данных из API в runtime (в отличие от Swift Codable, который бросает ошибку при несовпадении типов). Для runtime валидации в TS-проектах используют библиотеки вроде Zod или io-ts.

Ещё одно важное отличие: в Swift nil — это отсутствие значения в Optional, и компилятор заставляет тебя обработать этот случай. В JavaScript есть два "пустых" значения: null (явное отсутствие) и undefined (переменная не инициализирована). TypeScript помогает, но к этой двойственности нужно привыкнуть.

🍎 Swift
// Переменные let name = "Oleh" // immutable var count = 0 // mutable // Типы let age: Int = 30 let names: [String] = ["A", "B"] let user: User? = nil // Функции func greet(name: String) -> String { return "Hello, \(name)" } // Замыкания let double = { (x: Int) in x * 2 } names.map { $0.uppercased() } // Optional chaining user?.profile?.name
⚛️ JavaScript/TypeScript
// Переменные const name = "Oleh"; // immutable (почти) let count = 0; // mutable // Типы (TypeScript) const age: number = 30; const names: string[] = ["A", "B"]; const user: User | null = null; // Функции function greet(name: string): string { return `Hello, ${name}`; } // Arrow functions (замыкания) const double = (x: number) => x * 2; names.map(n => n.toUpperCase()); // Optional chaining (ES2020) user?.profile?.name
const как let в Swift, let как var. Да, наоборот. Но есть нюанс: const в JS запрещает переприсваивание переменной, но не делает объект иммутабельным. const arr = [1,2,3]; arr.push(4); — это валидно! В Swift let arr = [1,2,3] действительно запрещает мутацию массива. Это фундаментальное отличие: в JS const защищает ссылку, не содержимое.
Главные ловушки type coercion для Swift-разработчика: [] == false (true!), "" == false (true!), 0 == "" (true!). Всегда используй === (строгое сравнение) вместо ==. В TypeScript линтер ESLint с правилом eqeqeq поймает это автоматически. Также: typeof null === "object" — это известный баг JavaScript с 1995 года, который никогда не починят.
Template literals (`Hello, ${name}`) — аналог Swift string interpolation ("Hello, \(name)"). Обрати внимание на обратные кавычки ` вместо обычных кавычек. Template literals также поддерживают многострочность без специального синтаксиса.

🎁 Деструктуризация — твой новый друг

В JS/TS это используется ВЕЗДЕ. Привыкай.

Деструктуризация — это не просто синтаксический сахар, а основной паттерн передачи данных в React. Каждый компонент получает один аргумент — объект props. Вместо того чтобы писать props.team, props.onSelect по всему компоненту, ты сразу извлекаешь нужные поля в параметрах функции: function Card({ team, onSelect }). Это делает код чище и сразу показывает, какие данные компонент использует.

В Swift ближайший аналог — pattern matching в switch/case для enum-ов: case .success(let data). Но в JS деструктуризация работает для любых объектов и массивов, а не только для enum-ов. Деструктуризация массивов (const [a, b] = array) особенно важна для хуков: useState возвращает массив из двух элементов, и ты деструктуризуешь его в [value, setValue]. Имена переменных при деструктуризации массива ты выбираешь сам (в отличие от объектов, где имена должны совпадать с ключами).

Ещё один частый паттерн — spread и rest оператор (...). В параметрах функции function Button({ title, ...rest }) означает "возьми title отдельно, а все остальные props собери в объект rest". Это как variadic parameters в Swift, но для именованных свойств. Этот паттерн используется повсеместно для проброса props в дочерние компоненты.

// Объекты const user = { name: "Oleh", age: 30, city: "Valencia" }; const { name, age } = user; // name = "Oleh", age = 30 // С переименованием const { name: userName } = user; // userName = "Oleh" // Массивы const coords = [40.4168, -3.7038]; const [lat, lng] = coords; // lat = 40.4168 // В параметрах функций (ОЧЕНЬ часто в React) function TeamCard({ team, onSelect }) { return <Text>{team.name}</Text>; } // Аналог в Swift: // func teamCard(team: Team, onSelect: () -> Void)
Когда видишь { something } в параметрах функции — это деструктуризация props. Когда видишь { something } внутри JSX — это JavaScript-выражение. Контекст определяет значение фигурных скобок. Это одна из причин, почему начинающие путаются — одинаковый синтаксис, разная семантика.
Деструктуризация с значениями по умолчанию: function Card({ title = "Untitled", count = 0 }). Это как default parameter values в Swift. Но осторожно: значение по умолчанию применяется только при undefined, не при null. Если передать title={null}, title будет null, а не "Untitled". В Swift Optional с nil и отсутствие параметра — разные вещи, здесь похоже.
Вложенная деструктуризация тоже работает: const { user: { name, age } } = data. Но не злоупотребляй — глубокая деструктуризация ухудшает читаемость. Если объект глубоко вложен, лучше деструктуризовать в два шага.

⏳ Async/Await — почти как в Swift

Async/await в JavaScript появился позже, чем Promises (аналог Swift Concurrency поверх GCD). В основе лежит объект Promise — это контейнер для будущего значения. Promise может быть в трёх состояниях: pending (выполняется), fulfilled (успех), rejected (ошибка). async/await — это синтаксический сахар поверх Promise, как Swift async/await поверх continuation-based concurrency.

Ключевое отличие от Swift: в JavaScript нет ключевого слова throws. Любая async функция может бросить ошибку, и TypeScript не заставит тебя её обработать. В Swift компилятор требует try перед каждым вызовом throwing-функции. В JS/TS ты должен сам помнить, что fetch может упасть, и обернуть вызов в try/catch. Это одна из самых частых причин необработанных ошибок в React-приложениях.

Ещё одно важное отличие: try/catch в JavaScript оборачивает блок кода, а не отдельные выражения. В Swift ты пишешь let data = try await fetch() — try привязан к конкретному вызову. В JS ты оборачиваешь весь блок в try { ... } catch { ... }, и любая ошибка внутри блока попадёт в один catch. Это менее гранулярно, но на практике достаточно — обычно ты хочешь обработать ошибку на уровне всей операции, а не отдельного шага.

Promise chains (.then().catch().finally()) — это старый стиль, но ты встретишь его в существующем коде. fetch(url).then(res => res.json()).then(data => setData(data)).catch(err => setError(err)). Async/await делает то же самое, но читается линейно, как синхронный код.

🍎 Swift async/await
func fetchTeams() async throws -> [Team] { let url = URL(string: "api/teams")! let (data, _) = try await URLSession .shared.data(from: url) return try JSONDecoder() .decode([Team].self, from: data) } // Использование Task { do { let teams = try await fetchTeams() self.teams = teams } catch { print(error) } }
⚛️ JavaScript async/await
async function fetchTeams(): Promise<Team[]> { const response = await fetch("api/teams"); if (!response.ok) { throw new Error("Network error"); } return response.json(); // автоматический JSON parse } // Использование try { const teams = await fetchTeams(); setTeams(teams); } catch (error) { console.error(error); }
fetch — это как URLSession.shared.data(from:), только проще. Возвращает Promise (как Task). Но есть подвох: fetch считает HTTP 404 и 500 успешным ответом (Promise resolved), потому что сервер ответил. Ошибкой считается только сетевой сбой. Поэтому всегда проверяй response.ok (true для статусов 200-299). В Swift URLSession тоже не бросает ошибку на 404, но это чаще проверяют явно через HTTPURLResponse.statusCode.
В React нельзя сделать компонент async: async function MyComponent() — это ошибка. Асинхронная логика всегда живёт внутри useEffect или обработчиков событий. Это отличается от Swift, где View.body может неявно ждать @Published данные через ObservableObject. В React ты явно управляешь состояниями loading/error/data.
Promise.all([fetch1(), fetch2()]) — аналог Swift async let для параллельного выполнения. Promise.race() — кто первый, тот и результат (полезно для timeout). Promise.allSettled() — ждёт все, даже если часть упала (нет прямого аналога в Swift).

⚛️ Твой первый компонент: View vs Component

В React Native компоненты — это функции, а не структуры или классы. Исторически React начинал с классовых компонентов (похожих на UIViewController), но с 2019 года индустрия полностью перешла на функциональные компоненты с хуками. Ты не встретишь классы в новом коде. Это как переход iOS-мира от UIKit к SwiftUI — но более радикальный, потому что в React старый и новый подход могут сосуществовать в одном проекте.

Что значит "рендеринг" компонента? Когда React вызывает функцию компонента, она возвращает JSX — описание того, как UI должен выглядеть прямо сейчас. React сравнивает это описание с предыдущим и обновляет реальные нативные View только там, где есть различия. Это аналог body: some View в SwiftUI — ты описываешь текущее состояние UI, а не последовательность мутаций.

Жизненный цикл компонента проще, чем у UIViewController. Функциональный компонент не имеет методов viewDidLoad, viewWillAppear, viewDidDisappear и т.д. Вместо этого есть один универсальный хук useEffect, который покрывает все эти случаи. Компонент "рождается" (mount), "обновляется" (re-render при изменении props или state), и "умирает" (unmount). Всё управление жизненным циклом — через useEffect с правильным dependency array.

Props (входные параметры) в React — это read-only. Компонент никогда не модифицирует свои props, как SwiftUI View не модифицирует let-свойства. Если дочернему компоненту нужно сообщить что-то родителю — передаётся callback-функция через props (аналог closure в Swift).

🍎 SwiftUI View
struct TeamCard: View { let team: Team let onTap: () -> Void var body: some View { VStack(alignment: .leading) { Text(team.name) .font(.headline) Text(team.city) .foregroundColor(.gray) } .padding() .background(Color.white) .cornerRadius(12) .onTapGesture { onTap() } } }
⚛️ React Native Component
interface TeamCardProps { team: Team; onTap: () => void; } function TeamCard({ team, onTap }: TeamCardProps) { return ( <TouchableOpacity onPress={onTap}> <View style={styles.card}> <Text style={styles.title}> {team.name} </Text> <Text style={styles.subtitle}> {team.city} </Text> </View> </TouchableOpacity> ); }
body: some View → return JSX. Структура та же: входные данные (props) → UI. Но важное отличие: в SwiftUI body — это computed property, которая вызывается фреймворком только когда @State изменяется. В React функция компонента вызывается при любом изменении state или props, включая изменения в родительском компоненте. Поэтому оптимизация (React.memo, useMemo) так важна.
В Swift ты привык к строгой типизации props через struct. В React TypeScript выполняет ту же роль через interface. Определи interface TeamCardProps { team: Team; onTap: () => void; } и TypeScript будет проверять, что все обязательные props переданы. Опциональные props помечаются ? — как Optional в Swift.
Компонент должен возвращать один корневой элемент. Если нужно вернуть несколько элементов рядом — оберни в <View> или используй <Fragment> (сокращённо <>...</>). Fragment не создаёт реального View — это аналог Group в SwiftUI.

🏷️ JSX — это не HTML, это Swift DSL

JSX выглядит как HTML, но это просто синтаксический сахар для вызовов функций.

Когда Babel/TypeScript компилирует JSX, каждый тег превращается в вызов React.createElement(type, props, children). Это значит, что JSX — это JavaScript-выражения, а не шаблоны. Ты можешь сохранить JSX в переменную, вернуть из функции, положить в массив. Это как SwiftUI ViewBuilder — ты конструируешь дерево UI-элементов через обычный код.

Важное ограничение: внутри {} в JSX можно писать только выражения (expressions), но не инструкции (statements). Выражение возвращает значение: 2 + 2, isOk ? "yes" : "no", array.map(fn). Инструкция — нет: if/else, for, switch. Поэтому для условного рендеринга используют тернарный оператор ? : или логическое И &&, а для списков — .map() вместо for. В SwiftUI ты можешь использовать if/else внутри body благодаря @ViewBuilder — в JSX такой магии нет.

В React Native, в отличие от React для веба, ты не используешь HTML-теги (<div>, <span>). Вместо них — нативные компоненты: <View> (как UIView), <Text> (как UILabel), <Image> (как UIImageView), <ScrollView>, <TextInput>. Весь текст обязательно оборачивается в <Text> — нельзя просто написать строку внутри <View>.

// Это JSX: <View style={styles.container}> <Text>Hello</Text> </View> // Компилируется в: React.createElement(View, { style: styles.container }, React.createElement(Text, null, "Hello") ); // Аналог в Swift (если бы был такой DSL): View(style: .container) { Text("Hello") }
В JSX: {} = "вставь JavaScript выражение". Но осторожно с &&: {count && <Text>{count}</Text>} при count = 0 отрендерит "0" на экране, а не ничего! Потому что 0 — falsy, но React рендерит числа. Используй явное сравнение: {count > 0 && ...}. Это одна из самых частых ошибок, которую ловят на code review.
// Условный рендеринг {isLoading && <ActivityIndicator />} {error ? <ErrorView /> : <Content />} // Маппинг списка {teams.map(team => <TeamCard key={team.id} team={team} />)}

Атрибут key при рендеринге списков — обязательный. Это подсказка React для алгоритма reconciliation: по key он понимает, какой элемент переместился, добавился или удалился. Без key (или с index в качестве key) React будет пересоздавать DOM-элементы при любом изменении порядка, что убивает производительность и вызывает баги с состоянием.

Стили в React Native — это не CSS, а JavaScript-объекты. Вместо .cornerRadius(12) пишешь borderRadius: 12. Вместо .foregroundColor(.gray) — color: 'gray'. Используй StyleSheet.create({}) для оптимизации — React Native валидирует стили один раз при создании, а не при каждом рендере.

🔄 useState = @State

useState — это самый базовый хук React. Он делает то же, что @State в SwiftUI: хранит значение между рендерами и вызывает перерисовку при изменении. Но механика отличается. В SwiftUI @State — это property wrapper, который хранит данные вне struct (во внутреннем хранилище фреймворка) и автоматически трекает изменения через willSet. В React useState — это функция, которая регистрирует "слот" состояния в связанном списке хуков компонента.

Критически важный нюанс: замыкания захватывают стейт на момент рендера. Если ты вызвал setCount(count + 1) три раза подряд в одном обработчике, ты получишь +1, а не +3, потому что count в каждом вызове — это одно и то же значение из текущего рендера. Решение — функциональное обновление: setCount(prev => prev + 1). Тогда React гарантированно использует актуальное значение.

React батчит обновления состояния. Если ты вызываешь setCount(1); setName("Oleh"); setLoading(false); — будет один ре-рендер, а не три. С React 18+ батчинг работает везде: в обработчиках событий, в setTimeout, в промисах. Это как SwiftUI, который тоже объединяет изменения @State в один цикл обновления UI.

🍎 SwiftUI @State
struct Counter: View { @State private var count = 0 var body: some View { VStack { Text("Count: \(count)") Button("Increment") { count += 1 // автоматический re-render } } } }
⚛️ React useState
function Counter() { const [count, setCount] = useState(0); return ( <View> <Text>Count: {count}</Text> <Button title="Increment" onPress={() => setCount(count + 1)} /> </View> ); }
useState возвращает [текущее значение, функция-сеттер]. Это деструктуризация массива — имена ты выбираешь сам. Конвенция: [thing, setThing]. Аргумент useState — начальное значение. Можно передать функцию: useState(() => expensiveComputation()) — она вызовется только при первом рендере (lazy initialization). Полезно если начальное значение дорого вычислять.
Нельзя напрямую менять count = 5. ТОЛЬКО через setCount(5). Иначе React не узнает об изменении и UI не обновится. В Swift с @State ты пишешь count = 5 и property wrapper вызывает willSet → перерисовку. В React такой магии нет — setter-функция явная.
Ловушка "stale closure": если у тебя setTimeout или interval, внутри callback count будет значением на момент создания замыкания, а не текущим. Используй setCount(prev => prev + 1) вместо setCount(count + 1) в любых callback-ах, которые могут вызваться позже. Это не проблема для простых onClick, но критично для таймеров и подписок.

🔄 useState с объектами — immutability!

Это, пожалуй, самый контринтуитивный момент для Swift-разработчика. В Swift struct — это value type: когда ты пишешь filter.country = "Spain", ты создаёшь новую копию struct, и @State автоматически замечает изменение. В JavaScript объекты — это reference types (как class в Swift). Если ты мутируешь объект напрямую (filter.country = "Spain"), ссылка на объект не меняется, и React не видит изменения — UI не обновится.

Поэтому в React принят паттерн иммутабельных обновлений: создаёшь новый объект с нужными изменениями через spread-оператор { ...old, changed: newValue }. Это создаёт новую ссылку, React сравнивает ссылки (===) и понимает: "объект изменился, нужен ре-рендер". По сути, ты вручную делаешь то, что Swift struct делает автоматически — copy-on-write.

Для вложенных объектов это становится verbose: setUser({ ...user, address: { ...user.address, city: "Valencia" } }). Для сложных случаев используют библиотеку Immer (через useImmer хук), которая позволяет писать мутабельный код, а под капотом создаёт иммутабельные обновления. Также useReducer (аналог Redux reducer) часто удобнее для объектов с множеством полей.

🍎 SwiftUI (struct = value type)
struct TeamListView: View { @State private var filter = Filter( country: nil, minPlayers: 0 ) var body: some View { // Можно напрямую менять Button("Set Spain") { filter.country = "Spain" } } }
⚛️ React (объекты мутабельны!)
function TeamListView() { const [filter, setFilter] = useState({ country: null, minPlayers: 0 }); // ❌ НЕПРАВИЛЬНО (мутация!) // filter.country = "Spain"; // ✅ ПРАВИЛЬНО (новый объект) const handleSetSpain = () => { setFilter({ ...filter, country: "Spain" }); }; }
...filter — spread operator. Копирует все поля объекта в новый объект. Всегда создавай НОВЫЙ объект! Но помни: spread делает только shallow copy (поверхностную копию). Вложенные объекты копируются по ссылке. Это как NSMutableDictionary copy в Objective-C — верхний уровень копируется, внутренние — нет.
Для массивов тот же принцип. Не используй push(), splice() — они мутируют массив. Используй иммутабельные операции: [...arr, newItem] (добавить), arr.filter(x => x.id !== id) (удалить), arr.map(x => x.id === id ? { ...x, name: "new" } : x) (обновить элемент). В Swift с value types ты не думаешь об этом — компилятор сам создаёт копии. В React ты должен делать это явно.
Правило: если вызываешь setState — передавай новую ссылку. Для примитивов (number, string, boolean) это происходит автоматически. Для объектов и массивов — используй spread или immutable-методы. Это фундаментальный контракт React: Object.is(oldState, newState) === true → skip render, === false → re-render.

⚡ useEffect = onAppear + Combine

useEffect — самый мощный и одновременно самый сложный хук. Правильная ментальная модель: useEffect — это механизм синхронизации компонента с внешним миром, а не "lifecycle callback". Ты говоришь React: "после каждого рендера, если эти зависимости изменились, выполни этот код". Этот сдвиг в мышлении (от lifecycle к синхронизации) — ключевой для понимания React.

В SwiftUI ты привык думать категориями: onAppear (появился на экране), onDisappear (ушёл с экрана), .onChange(of:) (значение изменилось). В React useEffect покрывает все три случая в зависимости от dependency array. Пустой массив [] = mount/unmount (как onAppear + onDisappear). Массив с зависимостями [value] = onChange(of: value). Без массива = каждый рендер (почти никогда не нужно).

Частая ошибка — использовать useEffect для вещей, которым он не нужен. Если ты вычисляешь derived state (filteredList из list + searchQuery), не используй useEffect — используй useMemo или просто вычисляй в теле компонента. useEffect нужен для side effects: сетевые запросы, подписки, таймеры, логирование, DOM-манипуляции. Если код не взаимодействует с чем-то вне React — скорее всего, useEffect не нужен.

Race condition — классическая проблема с fetch в useEffect. Если пользователь быстро меняет фильтр, предыдущий запрос может вернуться после нового. Решение: cleanup-функция с флагом let ignore = false; return () => { ignore = true; } или AbortController для отмены fetch. В Swift ты решал это через Task.cancel() или Combine cancellable.

useEffect выполняется после рендера (не до, не во время). Это значит, что пользователь сначала увидит UI без данных, потом данные загрузятся и произойдёт повторный рендер. Это нормально — используй loading-состояние. В SwiftUI onAppear тоже вызывается после первого рендера, поведение идентично.
В SwiftUI ты привык к .task { await loadData() } — async-контексту привязанному к lifecycle View. В React useEffect не может быть async-функцией напрямую. Паттерн: useEffect(() => { async function load() { ... } load(); }, []) или useEffect(() => { loadData().then(setData); }, []).

📦 Dependency Array — самое важное

Dependency array — это механизм, через который ты говоришь React: "перезапусти этот эффект, только если эти значения изменились". React сравнивает каждый элемент массива с его предыдущим значением через Object.is() (по сути ===). Для примитивов (числа, строки, boolean) это работает интуитивно: Object.is(5, 5) === true. Но для объектов и массивов — сравнивается ссылка, а не содержимое.

Это источник самой распространённой ошибки с useEffect: бесконечный цикл. Если ты создаёшь объект или массив в теле компонента и передаёшь в dependency array, при каждом рендере создаётся новая ссылка → React считает что зависимость изменилась → запускает эффект → эффект вызывает setState → новый рендер → новая ссылка → снова эффект → бесконечный цикл. В Swift с value types (struct) такой проблемы нет, потому что Equatable сравнивает содержимое.

Практическое правило: в dependency array кладут только примитивы, стабильные ссылки (из useState, useRef) и мемоизированные значения (из useMemo, useCallback). Если нужно зависеть от свойства объекта — передавай само свойство: [filter.country] вместо [filter].

// 1. Пустой массив = только при mount (как viewDidLoad / onAppear) useEffect(() => { fetchInitialData(); }, []); // 2. С зависимостями = при изменении зависимостей (как Combine .sink) useEffect(() => { fetchTeams(filter.country); // перезагрузка при смене страны }, [filter.country]); // 3. Без массива = КАЖДЫЙ рендер (почти никогда не нужно!) useEffect(() => { console.log('Каждый рендер'); // ❌ Обычно ошибка });
Думай о dependency array как о .onChange(of:) в SwiftUI. Но с ключевым отличием: в SwiftUI onChange срабатывает только при фактическом изменении значения (Equatable comparison). В React — при изменении ссылки (Object.is). Для примитивов разницы нет, но для объектов и массивов — огромная.
Забыл зависимость? ESLint правило react-hooks/exhaustive-deps подскажет. Никогда не подавляй это правило через // eslint-disable без глубокого понимания последствий. Если ESLint просит добавить зависимость, а ты не хочешь — проблема в дизайне эффекта, а не в правиле. Обычно решение: useMemo/useCallback для стабилизации ссылки, или разделение эффекта на несколько меньших.
Функции, объявленные в теле компонента, создаются заново при каждом рендере. Если ты добавишь такую функцию в dependency array — получишь бесконечный цикл. Решение: либо переместить функцию внутрь useEffect, либо обернуть в useCallback, либо вынести за пределы компонента (если функция не зависит от props/state).

🧹 Cleanup в useEffect

Cleanup-функция в useEffect — это аналог deinit и cancellable.cancel() из Swift. Но она вызывается не только при unmount компонента, а ещё и перед каждым повторным запуском эффекта. Это критически важно для предотвращения утечек памяти и race conditions.

Когда cleanup вызывается? Два случая: (1) компонент удаляется из дерева (unmount) — как deinit; (2) зависимости эффекта изменились и эффект перезапускается — сначала вызывается cleanup предыдущего эффекта, потом новый эффект. Второй случай часто забывают, но он критичен. Представь: ты подписываешься на WebSocket для чата комнаты X. Пользователь переключился на комнату Y. Без cleanup ты останешься подписан на обе комнаты.

Типичные паттерны cleanup: (1) clearInterval(id) / clearTimeout(id) — для таймеров, иначе они продолжают работать после unmount; (2) subscription.unsubscribe() — для EventEmitter, WebSocket, Firebase listeners; (3) controller.abort() — для AbortController, отменяющего fetch-запросы; (4) ignore = true — флаг для игнорирования результатов устаревших промисов.

В Swift ты привык, что ARC и deinit автоматически чистят ресурсы. В React нет автоматического управления подписками — если забыл cleanup, подписка живёт вечно (memory leak). React Strict Mode в development-режиме вызывает эффект дважды (mount → unmount → mount), чтобы помочь найти отсутствующий cleanup.

🍎 Swift Combine
class TeamViewModel: ObservableObject { private var cancellables = Set<AnyCancellable>() func subscribe() { locationService.updates .sink { [weak self] location in self?.updateLocation(location) } .store(in: &cancellables) } deinit { cancellables.forEach { $0.cancel() } } }
⚛️ React useEffect cleanup
function LiveLocationView() { useEffect(() => { // Subscribe const subscription = locationService.subscribe( location => updateLocation(location) ); // Cleanup (как deinit) return () => { subscription.unsubscribe(); }; }, []); }
Return функция в useEffect = cleanup. Вызывается при unmount ИЛИ перед следующим запуском эффекта (если зависимости изменились). Это гарантирует, что предыдущая подписка/таймер будут очищены перед созданием новых. В Swift Combine ты хранил cancellables в Set — здесь cleanup выполняет ту же роль.
Если ты используешь setInterval внутри useEffect с пустым dependency array, переменные внутри callback будут "заморожены" на значениях первого рендера (stale closure). Таймер будет вечно видеть count = 0. Решение: setCount(prev => prev + 1) (функциональное обновление) или useRef для хранения актуального значения.
Паттерн для safe async в useEffect: useEffect(() => { let cancelled = false; fetchData().then(data => { if (!cancelled) setData(data); }); return () => { cancelled = true; }; }, [id]). Это предотвращает обновление state unmounted компонента — частая причина warning "Can't perform a React state update on an unmounted component".

📋 FlatList = UITableView

FlatList — это аналог UITableView, но с принципиально другим подходом к виртуализации. UITableView использует пул ячеек (cell reuse): создаёт ~15 ячеек и переиспользует их при скролле через dequeueReusableCell. FlatList использует виртуализацию: рендерит только видимые элементы плюс буфер (windowSize), а остальные удаляет из памяти полностью. При скролле React создаёт новые компоненты и уничтожает невидимые.

Это важное отличие с точки зрения производительности. В UIKit cell reuse сохраняет UIView объекты в памяти — ты только обновляешь содержимое. В React Native виртуализация пересоздаёт компоненты — это дороже, поэтому оптимизация (React.memo, getItemLayout, windowSize) критически важна для больших списков. Без оптимизации список из 1000 элементов может заметно подтормаживать.

Для группированных данных (как UITableView с секциями) используй SectionList — он принимает массив секций с заголовками. Для горизонтальных списков — FlatList с пропсом horizontal={true}. Для сеток — FlatList с numColumns={2}. Один компонент покрывает UITableView, UICollectionView с FlowLayout, и горизонтальный UIScrollView.

⚡ FlatList Performance Tips

Производительность FlatList — одна из главных тем в React Native. В UIKit ты получаешь 60fps "бесплатно" благодаря нативному cell reuse. В React Native каждый scroll-кадр может вызвать JavaScript → Native мост (bridge), что создаёт задержки. Новая архитектура (Fabric + JSI) значительно улучшает это, но оптимизация остаётся важной.

Каждый проп ниже решает конкретную проблему. getItemLayout убирает необходимость React Native измерять высоту каждого элемента при скролле — без него при быстром скролле видны пустые области (как при неправильном estimatedRowHeight в UITableView). windowSize контролирует количество элементов в памяти — меньше значение = меньше памяти, но больше шанс увидеть blank areas. maxToRenderPerBatch ограничивает количество элементов, рендерящихся за один JS-цикл — меньше = плавнее скролл, но медленнее заполнение.

Для реально больших списков (1000+ элементов) рассмотри @shopify/flash-list — drop-in замена FlatList, которая использует cell recycling (как UITableView!) вместо виртуализации. Benchmarks показывают 5-10x улучшение производительности. Shopify разработали его специально потому что стандартный FlatList не справлялся с их каталогами.

<FlatList data={teams} keyExtractor={(item) => item.id} // ✅ Уникальный ID, не index! renderItem={renderTeam} // ✅ Вынеси функцию наружу // Для фиксированной высоты (как estimatedRowHeight) getItemLayout={(data, index) => ({ length: ITEM_HEIGHT, offset: ITEM_HEIGHT * index, index })} // Виртуализация (сколько экранов рендерить) windowSize={10} // Начальное количество initialNumToRender={10} // Максимум элементов в памяти maxToRenderPerBatch={10} />

Результат: неправильный state в дочерних компонентах, мерцание, потеря введённых данных. В UIKit это как если бы dequeueReusableCell возвращал ячейку с данными другого элемента.
getItemLayout — как задать rowHeight вместо estimatedRowHeight в UITableView. Без него FlatList должен рендерить каждый элемент чтобы узнать его высоту, что убивает производительность при scrollToIndex и initialScrollIndex. Используй только если все элементы одной высоты. Для динамической высоты — рассмотри @shopify/flash-list с estimatedItemSize.
Вынеси renderItem в отдельную функцию вне компонента или мемоизируй через useCallback. Если определить renderItem inline (renderItem={({ item }) => ...}), при каждом рендере создаётся новая функция → FlatList считает что все элементы обновились. Это как если бы UITableView вызывал cellForRowAt для всех видимых ячеек при каждом скролле.

🎯 useCallback — мемоизация функций

В Swift функции и замыкания — это reference types, но при передаче в SwiftUI View они не вызывают перерисовку (SwiftUI сравнивает по данным, не по замыканиям). В React всё наоборот: каждый рендер создаёт новые экземпляры всех функций в теле компонента. Это значит, что если ты передаёшь функцию как prop дочернему компоненту, обёрнутому в React.memo, memo считает что prop изменился (новая ссылка на функцию) — и дочерний компонент перерендеривается без необходимости.

useCallback решает эту проблему: он возвращает ту же ссылку на функцию между рендерами, пока зависимости не изменились. По сути, это useMemo для функций: useCallback(fn, deps) === useMemo(() => fn, deps). Но useCallback полезен ТОЛЬКО если дочерний компонент обёрнут в React.memo или функция используется в dependency array другого хука.

Частая ошибка — оборачивать в useCallback всё подряд "на всякий случай". У useCallback есть overhead: React хранит функцию и dependency array в памяти, сравнивает зависимости при каждом рендере. Если дочерний компонент не обёрнут в React.memo, useCallback бесполезен — компонент всё равно перерендерится из-за изменения любого другого пропса или потому что родитель перерендерился.

💾 useMemo — мемоизация вычислений

useMemo — это кеширование результата вычисления между рендерами. Ближайший аналог в Swift — lazy var + manual invalidation: значение вычисляется один раз и кешируется, пока зависимости не изменятся. В отличие от computed property в Swift, которая пересчитывается при каждом обращении, useMemo пересчитывает только при изменении зависимостей.

Когда использовать useMemo? Два основных случая: (1) дорогие вычисления — фильтрация/сортировка большого массива, сложные математические операции; (2) стабилизация ссылки — когда результат (объект/массив) передаётся в dependency array другого хука или в React.memo компонент. Без useMemo новый массив (даже с теми же элементами) создаётся при каждом рендере.

Когда НЕ использовать: для простых вычислений (конкатенация строк, простая арифметика). useMemo сам имеет overhead — React хранит предыдущий результат и dependency array, сравнивает зависимости. Для простых операций этот overhead может быть дороже, чем пересчёт. Правило: если вычисление занимает меньше 1ms — useMemo не нужен.

Важно: React не гарантирует что useMemo сохранит кеш вечно. В будущем React может сбрасывать кеш для экономии памяти. Поэтому код должен работать корректно и без useMemo — это оптимизация, не гарантия поведения.

⚛️ React useMemo

Не злоупотребляй! useMemo имеет overhead: хранит предыдущее значение + dependency array + сравнивает зависимости при каждом рендере. Используй для: фильтрации/сортировки массивов 100+ элементов, сложных трансформаций данных, создания объектов, передаваемых в useEffect deps или React.memo компоненты. НЕ используй для: простых строк, чисел, boolean значений.
В Swift ты мог использовать lazy var для отложенного вычисления. Разница: lazy var вычисляется один раз и никогда не инвалидируется. useMemo перевычисляется при изменении зависимостей. Это ближе к "умному кешу" чем к lazy initialization. Для one-time initialization в React используй useState(() => expensiveInit()) или useRef.
Триада оптимизации React: useMemo (кешируй значения) + useCallback (кешируй функции) + React.memo (пропускай ре-рендер компонента). Они работают вместе: React.memo сравнивает props по ссылке, useMemo/useCallback обеспечивают стабильные ссылки. Без useMemo/useCallback React.memo бесполезен (props всегда "новые").

🛡️ React.memo — предотвращение re-render

По умолчанию в React, когда родительский компонент перерендеривается, все дочерние компоненты тоже перерендериваются, даже если их props не изменились. Это отличается от SwiftUI, где фреймворк автоматически определяет, что View не изменился (через Equatable), и пропускает перерисовку. В React эту оптимизацию нужно делать вручную через React.memo.

React.memo — это Higher-Order Component (HOC), который оборачивает твой компонент и добавляет shallow comparison props перед рендером. Shallow comparison означает: для каждого пропса проверяется Object.is(prevProp, nextProp). Если все props равны — рендер пропускается. "Shallow" потому что вложенные объекты сравниваются по ссылке, не по содержимому.

Когда использовать React.memo: (1) компонент рендерится часто (например, элемент списка в FlatList); (2) компонент тяжёлый (содержит сложный UI или вычисления); (3) props стабильны (используешь useMemo/useCallback для объектов и функций). Не оборачивай каждый компонент в memo — для простых компонентов overhead сравнения может быть дороже самого рендера.

Для отладки используй React DevTools Profiler: он показывает, сколько раз и почему каждый компонент перерендерился. Вкладка "Why did this render?" покажет, какой именно prop изменился. Это незаменимый инструмент, аналог Instruments Time Profiler в Xcode.

// Без memo — рендерится при ЛЮБОМ изменении родителя function TeamRow({ team, onSelect }) { console.log('TeamRow render', team.id); // Вызовется часто! return (...); } // С memo — рендерится только если props изменились const TeamRow = React.memo(function TeamRow({ team, onSelect }) { console.log('TeamRow render', team.id); // Только при изменении team/onSelect return (...); }); // С кастомным сравнением (как Equatable) const TeamRow = React.memo( function TeamRow({ team, onSelect }) { ... }, (prevProps, nextProps) => { // return true = НЕ рендерить (props равны) return prevProps.team.id === nextProps.team.id; } );
React.memo похож на Equatable для SwiftUI View. Если props не изменились — skip render. Второй аргумент React.memo — кастомная функция сравнения (как кастомный == в Equatable). Возвращай true если props равны (НЕ рендерить), false если изменились (рендерить). Обрати внимание: это обратная логика shouldComponentUpdate из классов.
React.memo сравнивает только props, НЕ state и НЕ context. Если компонент использует useContext или свой useState — он перерендерится при их изменении даже с React.memo. Это частый источник путаницы: "я обернул в memo, почему всё ещё рендерится?" — потому что внутри useState или context изменился.
Порядок оптимизации списков: (1) сначала оберни элемент списка в React.memo; (2) затем стабилизируй callback-props через useCallback; (3) затем добавь getItemLayout если высота фиксированная; (4) профилируй через React DevTools. Не делай шаг 4 до шагов 1-3 — без мемоизации профилирование покажет "всё рендерится", что неинформативно.

📜 Правила Hooks — выучи наизусть!

Правила хуков выглядят как произвольные ограничения, но у них есть техническая причина. React хранит state каждого хука в связанном списке (linked list), привязанном к позиции компонента в дереве. При каждом рендере React проходит этот список по порядку: первый вызов useState — первый элемент списка, второй — второй, и так далее. React идентифицирует хуки по порядковому номеру вызова, а не по имени.

Если ты поместишь хук внутрь условия if, то при одном рендере может быть 3 вызова хуков, а при следующем — 2. React потеряет соответствие между хуками и их данными: useState для count вернёт значение, предназначенное для name. Это приведёт к катастрофическим и трудно-отлавливаемым багам.

В Swift такой проблемы нет: property wrappers (@State, @Binding) — это свойства struct, и компилятор гарантирует их наличие. В React хуки — это runtime-вызовы, и компилятор (TypeScript) не может проверить их порядок. Поэтому существует ESLint плагин eslint-plugin-react-hooks, который ловит нарушения на этапе написания кода.

Правило простое: все хуки — на верхнем уровне функции компонента. Условную логику размещай внутри хука (в callback useEffect, в теле useState callback), а не вокруг вызова хука.

// ❌ НЕЛЬЗЯ: Hooks внутри условий function BadComponent({ show }) { if (show) { const [value, setValue] = useState(0); // 💥 Error! } } // ❌ НЕЛЬЗЯ: Hooks внутри циклов function BadComponent({ items }) { items.forEach(item => { useEffect(() => {}); // 💥 Error! }); } // ❌ НЕЛЬЗЯ: Hooks во вложенных функциях function BadComponent() { const handleClick = () => { const [x, setX] = useState(0); // 💥 Error! }; } // ✅ ПРАВИЛЬНО: Hooks только на верхнем уровне function GoodComponent({ show }) { const [value, setValue] = useState(0); // ✅ useEffect(() => { if (show) { // логика здесь } }, [show]); }
React отслеживает hooks по порядку вызова. Если порядок меняется — всё ломается. Два правила: (1) вызывай хуки только на верхнем уровне (не в if, for, nested functions); (2) вызывай хуки только из React-компонентов или из других custom hooks (не из обычных функций). ESLint правило react-hooks/rules-of-hooks обязательно — включай в каждый проект.
Нарушение правил хуков не всегда падает с ошибкой — иногда приложение молча работает неправильно. Например, хук может вернуть state другого хука. Это крайне сложно отладить. В development-режиме React Strict Mode помогает выявить некоторые проблемы, вызывая компоненты и эффекты дважды.
В Swift ты мог создать ObservableObject в любом месте — в init, в вычисляемом свойстве, в замыкании. В React хуки строго ограничены верхним уровнем компонента. Если тебе нужна условная логика — выноси её в отдельный компонент: вместо if (show) { const [x] = useState(0); } создай <ConditionalComponent /> который рендерится условно и содержит свой useState.

🔧 Custom Hooks — переиспользуемая логика

Custom hooks — это главный паттерн архитектуры в React. Они заменяют ViewModel (MVVM), Interactor (VIPER), Presenter (MVP) и другие паттерны, которые ты использовал в iOS. По сути, custom hook — это функция, которая инкапсулирует stateful логику и может использовать другие хуки. Она начинается с префикса use — это не просто конвенция, ESLint использует этот префикс для проверки правил хуков.

В iOS мире ты бы создал TeamsViewModel: ObservableObject с @Published var teams, @Published var loading, func fetch(). В React тот же паттерн выражается через custom hook useTeams(), который возвращает { teams, loading, error, fetch }. Преимущество хука: он не привязан к конкретному компоненту. ViewModel в iOS обычно создаётся как свойство View и живёт вместе с ним. Hook можно вызвать в любом компоненте, и каждый компонент получит свой собственный независимый экземпляр state.

Паттерн композиции: custom hooks могут использовать другие custom hooks. useTeams() внутри может использовать useFetch(), который внутри использует useState и useEffect. Это как протоколы с default implementation в Swift, но гибче — ты комбинируешь поведение через вызовы функций, а не через наследование или протоколы.

Тестирование: custom hooks легко тестировать изолированно через @testing-library/react-hooks. Тест вызывает хук, эмулирует действия (act), проверяет результат. Это значительно проще, чем тестировать ObservableObject в Swift, где нужно подписываться на @Published и ждать обновлений.

🍎 Swift ViewModel
class TeamsViewModel: ObservableObject { @Published var teams: [Team] = [] @Published var loading = false @Published var error: Error? func fetch() async { loading = true do { teams = try await api.fetchTeams() } catch { self.error = error } loading = false } }
⚛️ React Custom Hook
// Конвенция: имя начинается с "use" function useTeams() { const [teams, setTeams] = useState([]); const [loading, setLoading] = useState(false); const [error, setError] = useState(null); const fetch = useCallback(async () => { setLoading(true); try { const data = await api.fetchTeams(); setTeams(data); } catch (e) { setError(e); } setLoading(false); }, []); return { teams, loading, error, fetch }; }
Custom Hook — это просто функция, которая использует другие hooks. Как ViewModel, но проще. Важное отличие: ViewModel в iOS — один экземпляр на View (обычно через @StateObject). Custom hook создаёт новый изолированный state при каждом вызове. Два компонента, вызывающие useTeams(), получат два разных списка teams. Для shared state используй Context или state manager (Redux, Zustand).
Именование критично: если функция не начинается с use, ESLint не будет проверять правила хуков внутри неё. fetchTeams() — обычная функция, хуки внутри неё вызовут runtime-ошибку. useTeams() — custom hook, хуки внутри работают корректно. Это техническое ограничение, не стилистическое предпочтение.
Типичные custom hooks в проекте: useFetch(url) — generic загрузка данных, useDebounce(value, delay) — задержка для поиска, useLocalStorage(key) — persistent state, useMediaQuery(query) — responsive design, useForm(config) — управление формами. Библиотека react-use содержит десятки готовых хуков — не изобретай велосипед.

Использование Custom Hook

Этот слайд показывает главную идею custom hooks — separation of concerns (разделение ответственности). Компонент TeamsScreen не знает ничего о сетевых запросах, обработке ошибок или преобразовании данных. Он просто вызывает useTeams() и получает готовые данные. Это тот же принцип, что MVVM в iOS: View не знает о сети, ViewModel инкапсулирует логику.

Преимущество перед ViewModel: переиспользуемость и композиция. В iOS ViewModel обычно привязан к конкретному экрану через @StateObject. Custom hook можно вызвать в любом компоненте — экран, модальное окно, виджет. Более того, можно комбинировать хуки: const { teams } = useTeams(); const { favorites } = useFavorites(); const display = useMemo(() => mergeTeamsWithFavorites(teams, favorites), [teams, favorites]);

Обрати внимание на структуру файлов: useTeams.ts — отдельный файл для хука, TeamsScreen.tsx — компонент. Конвенция: один хук = один файл, имя файла совпадает с именем хука. Папка hooks/ для shared hooks, или хук рядом с компонентом, если используется только там. В iOS мире это аналог раздельных файлов ViewModel и View.

Паттерн if (loading) return; if (error) return; — это "early return" для разных состояний. В SwiftUI ты бы использовал switch viewModel.state { case .loading: ... case .error: ... case .loaded: ... }. В React нет встроенного enum для состояний загрузки, но паттерн тот же: проверяй состояния по порядку и возвращай соответствующий UI. Библиотеки вроде React Query / TanStack Query делают это автоматически.

// useTeams.ts export function useTeams() { const [teams, setTeams] = useState<Team[]>([]); const [loading, setLoading] = useState(true); const [error, setError] = useState<string | null>(null); useEffect(() => { fetchTeams() .then(setTeams) .catch(e => setError(e.message)) .finally(() => setLoading(false)); }, []); return { teams, loading, error }; } // TeamsScreen.tsx function TeamsScreen() { const { teams, loading, error } = useTeams(); // Вся логика скрыта! if (loading) return <Loading />; if (error) return <Error message={error} />; return <TeamList teams={teams} />; }
Custom Hooks позволяют переиспользовать stateful логику между компонентами. Каждый вызов хука создаёт изолированный state — два компонента, использующие useTeams(), будут иметь два разных набора данных, двух разных loading-состояний. Это НЕ singleton как Redux store — это фабрика state, как создание нового ViewModel для каждого экрана.
В production-проектах вместо ручных useTeams() хуков часто используют TanStack Query (React Query) — библиотеку для server state management. Она автоматически добавляет: кеширование, дедупликацию запросов, background refetch, retry, optimistic updates, pagination. Это как URLSession с NSURLCache + Combine, но гораздо мощнее и проще. Рекомендуется для любого проекта с API-вызовами.
Антипаттерн: не создавай "God Hook" — один хук, который делает всё (useApp() с 20 полями state). Каждый хук должен отвечать за одну задачу (Single Responsibility Principle). useTeams() для загрузки команд, useSearch() для поиска, useAuth() для авторизации. Именно так разделяй — ты уже знаешь этот принцип из iOS.

🗺️ Navigation в React Native: Overview

Самое важное отличие от iOS: в React Native нет встроенной навигации. В UIKit/SwiftUI навигация — часть фреймворка (UINavigationController, NavigationStack). В RN навигация — это сторонняя библиотека, которая эмулирует нативное поведение через JavaScript. Под капотом React Navigation использует нативные экраны (react-native-screens), но управление стеком происходит в JS. Это значит, что переходы между экранами — это не настоящий push/pop ViewController, а управление деревом React-компонентов с нативными анимациями.

Это фундаментальное архитектурное отличие. В iOS ты привык, что UINavigationController владеет стеком ViewController-ов, каждый со своим жизненным циклом (viewDidLoad, viewWillAppear). В RN "экран" — это просто React-компонент, который монтируется/размонтируется. Вместо viewWillAppear ты используешь хук useFocusEffect, вместо viewDidLoad — useEffect при монтировании.

iOS: React Native: ──────────────────────────────────────────── UINavigationController → Stack Navigator UITabBarController → Tab Navigator UISplitViewController → Drawer Navigator present(vc, animated:) → Stack.Screen (modal) push/pop → navigation.navigate/goBack Coordinator pattern → Expo Router (file-based)

В React Native есть два подхода:

  • React Navigation — классический, императивный. Ты описываешь навигационное дерево в коде (как Coordinator pattern). Гибко, но много boilerplate. Используется в большинстве production-приложений.
  • Expo Router — file-based, как Next.js (рекомендуется). Файловая структура определяет routes автоматически. Проще для новых проектов, встроенный deep linking, типизированные маршруты. Основан на React Navigation, поэтому все его возможности доступны.

Выбор между ними: если начинаешь новый проект на Expo — используй Expo Router. Если интегрируешь RN в существующее нативное приложение (brownfield) — React Navigation напрямую. Expo Router — это обёртка над React Navigation, поэтому переход от одного к другому не требует переписывания всего.

Expo Router = SwiftUI NavigationStack + файловая структура. Создал файл — получил route. В отличие от Coordinator pattern, где ты вручную описываешь каждый переход, здесь маршруты генерируются автоматически из структуры папок. Это устраняет целый класс багов, связанных с рассинхронизацией маршрутов и кода.
В iOS при push/pop ViewController-ы сохраняют состояние в стеке. В RN экраны по умолчанию тоже остаются в памяти при push, но при navigate к уже существующему экрану — он может быть переиспользован, а не создан заново. Это поведение отличается от iOS, где каждый push создаёт новый экземпляр.

📁 Expo Router: Структура файлов = Routes

File-based routing — это конвенция из мира веб-разработки (Next.js, Nuxt, SvelteKit), перенесённая в мобильную разработку. Идея проста: структура папок и файлов в директории app/ автоматически определяет навигационные маршруты приложения. Не нужен Coordinator, не нужен Router — файловая система и есть твой роутер.

Для iOS-разработчика это радикальное упрощение. Вспомни, сколько кода уходит на Coordinator pattern: протоколы, фабрики, методы навигации, связывание ViewController-ов. В Expo Router всего этого нет. Создал файл teams.tsx — маршрут /teams готов. Создал папку team/ с файлом [id].tsx — динамический маршрут /team/:id готов. Вся навигационная логика выводится автоматически.

🍎 iOS Coordinator

⚛️ Expo Router (file-based)
app/ ├── _layout.tsx // Root layout ├── (tabs)/ // Tab group │ ├── _layout.tsx // Tab bar config │ ├── index.tsx // "/" (Home tab) │ ├── teams.tsx // "/teams" tab │ └── profile.tsx // "/profile" tab ├── team/ │ └── [id].tsx // "/team/123" (dynamic) └── modal.tsx // "/modal" (modal screen) // Навигация: router.push('/team/123'); router.back();

Обрати внимание на ключевые отличия в поведении: в iOS Coordinator явно создаёт экземпляры ViewController-ов и управляет их жизненным циклом. В Expo Router экран создаётся автоматически при навигации на соответствующий URL. Параметры передаются через URL (как в вебе), а не через инициализатор. Это делает deep linking бесплатным — любой маршрут доступен по URL, что в iOS требует отдельной настройки Universal Links.

Имя файла = URL path. [id].tsx — dynamic segment (как :id в web). Круглые скобки (tabs) создают "группу" — она не появляется в URL. Файл index.tsx — это корневой маршрут группы (как index.html в вебе). Если ты привык к Coordinator pattern, представь, что файловая система — это и есть твой Coordinator, только декларативный.
Распространённая ошибка iOS-разработчиков: пытаться передать сложные объекты через навигационные параметры (как через segue prepare). В Expo Router параметры — это строки в URL. Передавай только ID, а данные загружай на целевом экране через React Query или Zustand. Это отличается от iOS, где ты часто передаёшь целый model object при push.

📦 _layout.tsx — Root Layout

Файлы _layout.tsx — это ключевая концепция Expo Router. Каждый layout оборачивает все дочерние маршруты в своей директории. Это как UINavigationController, который "владеет" всеми экранами внутри себя. Layout определяет тип навигатора (Stack, Tabs, Drawer) и общие настройки для всех дочерних экранов.

Layouts вкладываются друг в друга: root layout оборачивает всё приложение, layout в (tabs)/ оборачивает только вкладки. Это создаёт иерархию навигаторов — точно как в iOS, где UITabBarController содержит UINavigationController-ы. Разница в том, что в iOS ты строишь эту иерархию императивно в коде, а здесь она выводится из структуры папок.

// app/_layout.tsx import { Stack } from 'expo-router'; export default function RootLayout() { return ( <Stack screenOptions={{ headerStyle: { backgroundColor: '#1a1a2e' }, headerTintColor: '#fff', headerTitleStyle: { fontWeight: 'bold' }, }} > <Stack.Screen name="(tabs)" options={{ headerShown: false }} /> <Stack.Screen name="team/[id]" options={{ title: 'Team Details' }} /> <Stack.Screen name="modal" options={{ presentation: 'modal', title: 'Settings' }} /> </Stack> ); }

В этом коде screenOptions — это глобальные настройки, аналог UINavigationBar.appearance(). Они применяются ко всем экранам в этом Stack. Каждый Stack.Screen регистрирует маршрут и его опции. Обрати внимание на presentation: 'modal' — это аналог present(vc, animated: true) в iOS. Stack автоматически добавляет кнопку "назад" и жест свайпа, как UINavigationController.

Stack — это UINavigationController. screenOptions — глобальные настройки navigation bar. Важное отличие: в iOS ты можешь настроить navigation bar из самого ViewController (в viewWillAppear). В Expo Router ты тоже можешь переопределить options прямо из экрана через компонент <Stack.Screen options={{...}} />, размещённый внутри компонента экрана.
Root Layout — это точка входа приложения. Здесь размещают глобальные providers (QueryClient, Context, темы). Если ты забудешь обернуть Stack в необходимый Provider, все дочерние экраны потеряют доступ к контексту. Это аналог того, как в iOS SceneDelegate конфигурирует root window и root ViewController.

📑 Tab Bar Layout

Tab Bar в Expo Router — это layout-файл внутри группы (tabs). Круглые скобки в имени папки создают "группу маршрутов" — они не влияют на URL. То есть файл app/(tabs)/teams.tsx доступен по адресу /teams, а не /(tabs)/teams. Это чисто организационная структура для группировки экранов под одним навигатором.

Каждый файл внутри (tabs)/ автоматически становится вкладкой. Порядок вкладок определяется порядком Tabs.Screen в layout. Если файл существует в папке, но не описан в layout — он всё равно будет доступен как маршрут, но появится как дополнительная вкладка (что обычно нежелательно).

Ключевые отличия от UITabBarController: в iOS каждая вкладка обычно содержит свой UINavigationController. В Expo Router ты можешь вложить Stack внутрь каждой вкладки, но часто этого не нужно — навигация из вкладки просто пушит экран поверх всего tab bar (через root Stack). Иконки — это React-компоненты (не UIImage), что даёт больше гибкости: можно использовать анимированные иконки, бейджи, кастомные SVG.

В iOS для кастомного tab bar ты создаёшь подкласс UITabBarController или подменяешь UITabBar. В Expo Router можно полностью заменить tab bar через tabBar prop: <Tabs tabBar={(props) => <MyCustomTabBar {...props} />}>. Это значительно проще, чем наследование от UITabBar.
Частая ошибка: создать файл в (tabs)/ и забыть описать его в layout. Expo Router создаст для него вкладку с дефолтными настройками. Чтобы скрыть маршрут из tab bar, используй options={{ href: null }} — экран останется доступным программно, но не покажется как вкладка.

🔗 Navigation Hooks

В iOS для навигации ты обращаешься к navigationController — объекту, который управляет стеком. В React Native навигация доступна через хуки — функции, которые дают доступ к навигатору из любого компонента. Это важное отличие: не нужно передавать ссылку на навигатор через dependency injection или делегаты — любой компонент внутри навигатора может навигировать.

Expo Router предоставляет три ключевых хука: useRouter() для навигации, useLocalSearchParams() для чтения параметров текущего маршрута, и usePathname() для определения текущего URL. Эти хуки типизированы — TypeScript подскажет, какие параметры доступны на каждом маршруте.

import { useRouter, useLocalSearchParams, usePathname } from 'expo-router'; function TeamListScreen() { const router = useRouter(); const handleTeamPress = (teamId: string) => { // Push new screen (как navigationController.push) router.push(`/team/${teamId}`); // Или с параметрами router.push({ pathname: '/team/[id]', params: { id: teamId, tab: 'players' } }); }; const handleSettings = () => { // Modal (как present(vc, animated:)) router.push('/modal'); }; const handleBack = () => { router.back(); // Как navigationController.popViewController }; } // В team/[id].tsx function TeamDetailScreen() { const { id } = useLocalSearchParams<{ id: string }>(); // id = "123" если URL был /team/123 }

Разница между router.push() и router.replace(): push добавляет экран в стек (как pushViewController), replace заменяет текущий (как setViewControllers с последним элементом). Есть также router.navigate() — умная навигация, которая не создаёт дубликат, если экран уже в стеке. Это отличается от iOS, где каждый push всегда создаёт новый ViewController.

useLocalSearchParams — как получить параметры из segue в iOS. Но есть важное отличие: параметры всегда строковые (это URL). Если передал число, получишь строку. Привычка из Swift, где параметр может быть любого типа, здесь не работает. Используй parseInt(id) или валидацию через Zod, если нужны числа.
Есть тонкая разница между useLocalSearchParams и useGlobalSearchParams: local возвращает параметры, актуальные для данного экрана (даже если URL изменился), global — всегда текущие параметры URL. На практике почти всегда используй local, иначе компонент будет перерисовываться при навигации на другие экраны.
Deep linking работает автоматически: поскольку навигация основана на URL, любой маршрут вроде /team/123 доступен из push-уведомлений или внешних ссылок без дополнительной настройки. В iOS для этого нужно настраивать Universal Links, обрабатывать URL в SceneDelegate, и маршрутизировать вручную.

🗄️ State Management: Когда useState мало

Управление состоянием — это, пожалуй, самое большое концептуальное отличие React от iOS. В UIKit ты привык, что каждый ViewController владеет своим состоянием, и передача данных между экранами происходит через делегаты, замыкания, NotificationCenter или shared singleton. В SwiftUI появились @State, @Binding, @EnvironmentObject — но даже там нет такого чёткого разделения типов состояния, как в React-экосистеме.

В React есть фундаментальное правило: данные текут только вниз (unidirectional data flow). Родительский компонент передаёт данные детям через props. Ребёнок не может напрямую изменить состояние родителя — только вызвать callback, переданный через props. Это отличается от iOS, где delegate pattern позволяет ребёнку вызвать метод родителя напрямую, а @Binding создаёт двустороннюю связь. В React двусторонней привязки нет — есть только передача данных вниз и callback-ов вверх.

Типы состояния: 1. LOCAL STATE (useState) Скоуп: один компонент Пример: isExpanded, inputValue iOS: @State в SwiftUI, свойства ViewController 2. LIFTED STATE (props drilling) Скоуп: несколько компонентов в дереве Пример: filter в родителе, используется в детях iOS: @Binding, delegate pattern 3. GLOBAL STATE (Context/Redux/Zustand) Скоуп: всё приложение Пример: user, theme, favorites iOS: @EnvironmentObject, Singleton, AppDelegate 4. SERVER STATE (React Query/SWR) Скоуп: кэшированные данные с бэкенда Пример: teams, players, matches iOS: URLSession + ViewModel + кэш (нет стандарта)

Ключевой принцип: "поднимай состояние" (lift state up). Если два соседних компонента используют одни и те же данные, перенеси состояние в их общего родителя. Если состояние нужно далеко по дереву — используй Context или Zustand. Если данные приходят с сервера — используй React Query. Неправильный выбор типа состояния — главная причина проблем с производительностью и запутанного кода в React-приложениях.

В iOS это: @State = useState, @Binding = callback props, @EnvironmentObject = Context, ViewModel + Combine = React Query. Главное отличие: в React чёткое разделение client state и server state. В iOS ViewModel обычно хранит и то, и другое — это источник сложности.
В iOS ты привык к reference-type моделям (class) и мутации по месту. В React всё состояние immutable — ты не мутируешь массив, а создаёшь новый. setItems([...items, newItem]) вместо items.append(newItem). Это нужно для того, чтобы React мог определить, что состояние изменилось, и перерисовать компонент.

🌐 React Context = @EnvironmentObject

React Context — это механизм "проброса" данных через дерево компонентов без передачи через props на каждом уровне. Концептуально это аналог @EnvironmentObject в SwiftUI: ты размещаешь Provider наверху дерева, и любой дочерний компонент может "подписаться" на эти данные через useContext.

Но есть критическое отличие от SwiftUI: в SwiftUI, @EnvironmentObject работает с ObservableObject, и SwiftUI умно перерисовывает только те View, которые используют изменённые @Published свойства. В React Context нет такой гранулярности: когда значение Context изменяется, ВСЕ компоненты, использующие useContext для этого контекста, перерисовываются. Даже если они используют только часть данных, которая не изменилась.

🍎 SwiftUI Environment
// Создание class UserStore: ObservableObject { @Published var user: User? @Published var favorites: Set<String> = [] func toggleFavorite(_ id: String) { if favorites.contains(id) { favorites.remove(id) } else { favorites.insert(id) } } } // Провайдер @main struct MyApp: App { @StateObject var userStore = UserStore() var body: some Scene { WindowGroup { ContentView() .environmentObject(userStore) } } } // Использование struct TeamRow: View { @EnvironmentObject var store: UserStore var body: some View { ... } }

// Создание контекста interface FavoritesContextType { favorites: Set<string>; toggleFavorite: (id: string) => void; } const FavoritesContext = createContext<FavoritesContextType>(null!); // Провайдер function FavoritesProvider({ children }) { const [favorites, setFavorites] = useState(new Set<string>()); const toggleFavorite = useCallback((id: string) => { setFavorites(prev => { const next = new Set(prev); if (next.has(id)) next.delete(id); else next.add(id); return next; }); }, []); return ( <FavoritesContext.Provider value={{ favorites, toggleFavorite }}> {children} </FavoritesContext.Provider> ); } // Использование function TeamRow() { const { favorites, toggleFavorite } = useContext(FavoritesContext); }

Обрати внимание на useCallback в провайдере. Без него функция toggleFavorite пересоздавалась бы при каждом рендере провайдера, и это считалось бы "новым значением" контекста, провоцируя перерисовку всех потребителей. useCallback мемоизирует функцию — она создаётся один раз и не меняется. В SwiftUI такой проблемы нет, потому что методы ObservableObject не пересоздаются.

Context подходит для данных, которые меняются редко: тема, авторизация, настройки. НЕ используй Context для часто обновляемых данных (позиция скролла, ввод текста, анимации) — каждое обновление перерисует всё поддерево. Для таких случаев используй Zustand с селекторами или локальный state. Это главная ловушка Context, в которую попадают разработчики с опытом SwiftUI, где @EnvironmentObject более оптимизирован.

🪝 Custom Hook для Context

Обёртка useContext в кастомный хук — это не просто удобство, а best practice в React-разработке. Этот паттерн решает три проблемы: (1) скрывает детали реализации (потребитель не знает, что под капотом Context), (2) добавляет проверку на наличие Provider, (3) обеспечивает единую точку рефакторинга — если захочешь заменить Context на Zustand, изменишь только этот хук.

В iOS эквивалент — это протокол, за которым скрывается конкретная реализация. Компонент вызывает useFavorites() и получает интерфейс, не зная, хранятся ли данные в Context, Zustand, или AsyncStorage. Это инверсия зависимости (Dependency Inversion) на уровне React-хуков.

// contexts/FavoritesContext.tsx import { createContext, useContext, useState, useCallback, ReactNode } from 'react'; interface FavoritesContextType { favorites: Set<string>; isFavorite: (id: string) => boolean; toggleFavorite: (id: string) => void; } const FavoritesContext = createContext<FavoritesContextType | null>(null); export function FavoritesProvider({ children }: { children: ReactNode }) { const [favorites, setFavorites] = useState<Set<string>>(new Set()); const isFavorite = useCallback((id: string) => favorites.has(id), [favorites]); const toggleFavorite = useCallback((id: string) => { setFavorites(prev => { const next = new Set(prev); next.has(id) ? next.delete(id) : next.add(id); return next; }); }, []); return ( <FavoritesContext.Provider value={{ favorites, isFavorite, toggleFavorite }}> {children} </FavoritesContext.Provider> ); } // Custom hook для удобства export function useFavorites() { const context = useContext(FavoritesContext); if (!context) { throw new Error('useFavorites must be used within FavoritesProvider'); } return context; }

Проверка if (!context) — это эквивалент fatalError() в Swift: если компонент пытается использовать useFavorites() вне FavoritesProvider, приложение падает с понятным сообщением. Без этой проверки ты получишь загадочный Cannot read property 'favorites' of null — и будешь долго искать причину. В SwiftUI @EnvironmentObject делает то же самое — крэшится, если объект не предоставлен.

Всегда проверяй null в custom hook! Иначе получишь cryptic errors. Типичная ошибка: используешь хук в компоненте, который рендерится ДО монтирования Provider. Например, в _layout.tsx до обёртки в Provider. Порядок имеет значение — Provider должен быть выше в дереве, чем любой потребитель.
Обрати внимание на паттерн: Context + тип + Provider + custom hook — это всегда один файл. Этот файл экспортирует только FavoritesProvider и useFavorites. Сам FavoritesContext не экспортируется — он деталь реализации. Это как private/public в Swift: потребителю не нужен доступ к сырому Context.

Подключение Provider

Provider tree ("дерево провайдеров") — это паттерн, который может удивить iOS-разработчика. В SwiftUI ты цепляешь .environmentObject() к View, и все дочерние View получают доступ. В React аналогичная концепция, но провайдеры вкладываются друг в друга как матрёшки. Root layout — это место, где ты выстраиваешь эту "матрёшку".

Порядок вложенности важен: внутренний провайдер может использовать данные из внешнего, но не наоборот. Например, если AuthProvider зависит от QueryClient (для загрузки профиля пользователя), то QueryClientProvider должен быть снаружи. Типичный порядок: ErrorBoundary > QueryClient > Auth > Theme > Navigation.

// app/_layout.tsx import { Stack } from 'expo-router'; import { FavoritesProvider } from '@/contexts/FavoritesContext'; import { QueryClient, QueryClientProvider } from '@tanstack/react-query'; const queryClient = new QueryClient(); export default function RootLayout() { return ( <QueryClientProvider client={queryClient}> <FavoritesProvider> <Stack> <Stack.Screen name="(tabs)" options={{ headerShown: false }} /> <Stack.Screen name="team/[id]" /> </Stack> </FavoritesProvider> </QueryClientProvider> ); } // В любом компоненте внутри: function TeamRow({ team }) { const { isFavorite, toggleFavorite } = useFavorites(); return ( <View style={styles.row}> <Text>{team.name}</Text> <TouchableOpacity onPress={() => toggleFavorite(team.id)}> <Ionicons name={isFavorite(team.id) ? "heart" : "heart-outline"} color="red" /> </TouchableOpacity> </View> ); }

В этом примере QueryClientProvider оборачивает всё — значит React Query доступен везде, включая внутри FavoritesProvider. FavoritesProvider оборачивает Stack — значит все экраны навигации имеют доступ к favorites. Если бы FavoritesProvider был снаружи QueryClientProvider, то favorites-логика не могла бы использовать React Query для синхронизации с сервером.

В iOS ты часто используешь AppDelegate или SceneDelegate для инициализации глобальных сервисов. В React root layout выполняет ту же роль. Если у тебя много провайдеров (5+), создай отдельный компонент Providers, который их все оборачивает — это снижает "лесенку" вложенности и делает root layout чище.
Распространённая ошибка: создание new QueryClient() внутри компонента (в теле функции). При каждом рендере будет создаваться новый экземпляр, что сбрасывает весь кэш. Объявляй QueryClient вне компонента (на уровне модуля) или используй useState / useRef для однократного создания.

🐻 Zustand — простой global state

Zustand (немецкое слово "состояние") — это минималистичная библиотека для глобального состояния. Если Context — это встроенный инструмент React с известными ограничениями (перерисовка всех потребителей), то Zustand решает эти проблемы и добавляет удобства. В мире iOS ближайший аналог — это синглтон с @Published свойствами в Combine, но без boilerplate.

Почему Zustand, а не Redux или MobX? Redux требует actions, reducers, middleware — огромный boilerplate для простых задач. MobX использует observable/proxy-паттерн, похожий на SwiftUI, но добавляет "магию", которую трудно отлаживать. Zustand — золотая середина: минимальный API (create — и всё), нет Provider-обёрток, нативная поддержка TypeScript, встроенная персистенция. Для большинства мобильных приложений это оптимальный выбор.

Context хорош, но Zustand ещё проще и быстрее.

// stores/useFavoritesStore.ts import { create } from 'zustand'; import { persist, createJSONStorage } from 'zustand/middleware'; import AsyncStorage from '@react-native-async-storage/async-storage'; interface FavoritesStore { favorites: string[]; toggleFavorite: (id: string) => void; isFavorite: (id: string) => boolean; } export const useFavoritesStore = create<FavoritesStore>()( persist( (set, get) => ({ favorites: [], toggleFavorite: (id) => set((state) => ({ favorites: state.favorites.includes(id) ? state.favorites.filter(f => f !== id) : [...state.favorites, id] })), isFavorite: (id) => get().favorites.includes(id), }), { name: 'favorites-storage', storage: createJSONStorage(() => AsyncStorage), } ) ); // Использование (без Provider!) function TeamRow({ team }) { const { isFavorite, toggleFavorite } = useFavoritesStore(); // ... }

Ключевое преимущество Zustand перед Context — селекторы. Ты можешь подписаться только на часть store: const favorites = useFavoritesStore(state => state.favorites). Компонент перерисуется только когда favorites изменится, а не при любом изменении store. В iOS это как @Published в Combine — подписка на конкретное свойство, а не весь объект. В Context такой гранулярности нет.

Zustand = useState + persist + devtools. Без boilerplate. Без Provider wrapper. Ещё одно преимущество: store можно использовать вне React-компонентов (в утилитах, middleware). Вызови useFavoritesStore.getState() из обычной функции — это невозможно с Context, который работает только внутри компонентов.
Встроенная персистенция через persist middleware — это аналог UserDefaults в iOS. Данные автоматически сериализуются в AsyncStorage и восстанавливаются при запуске. Не нужно вручную писать save/load логику. Для чувствительных данных (токены) используй expo-secure-store вместо AsyncStorage — аналог Keychain в iOS.

💾 Storage в React Native: Полный гайд

В iOS у тебя UserDefaults, Keychain, CoreData, FileManager — и ты точно знаешь, когда что использовать. В React Native ситуация аналогичная, но с другими именами. Главное правило: выбор storage зависит от типа данных, объёма и требований безопасности.

AsyncStorage — дефолтный выбор для большинства случаев. Асинхронный key-value store, данные хранятся как строки (нужен JSON.stringify/parse для объектов). Ограничение: 6MB на Android. Zustand persist middleware использует его по умолчанию.

MMKV — если нужна скорость. Синхронный (не нужен await!), в 30x быстрее AsyncStorage на чтении. Создан WeChat для миллиардов пользователей. Используй как drop-in замену AsyncStorage в Zustand: persist(store, { storage: createMMKVStorage() }).

Главное отличие от iOS: в React Native нет синхронного key-value store из коробки (AsyncStorage — async). Это значит, что при старте приложения ты не можешь мгновенно прочитать настройки — нужен loading state. MMKV решает эту проблему, но требует нативной зависимости. На интервью: начни с AsyncStorage (встроенный), упомяни MMKV как production-оптимизацию.
На интервью часто спрашивают: "Где хранить auth token?" Правильный ответ: expo-secure-store (аналог Keychain). НИКОГДА не AsyncStorage — данные не зашифрованы, на Android доступны через root. Для offline-first приложения с большим объёмом данных: SQLite или WatermelonDB. Для простых настроек/фильтров: AsyncStorage или MMKV.

🔄 React Query = Server State Management

React Query (TanStack Query) — это, вероятно, самая важная библиотека для React-разработки после самого React. Она решает проблему, которую каждый iOS-разработчик решал десятки раз: загрузка данных с сервера, кэширование, обработка ошибок, повторные попытки, обновление в фоне, отмена запросов. В iOS для этого ты пишешь ViewModel с Combine, URLSession, кэш-слой — React Query делает это в 5 строк.

Ключевая идея: серверные данные (teams, players, matches) — это не "состояние приложения", а "кэш удалённых данных". Они принадлежат серверу, а у тебя только копия. React Query управляет этой копией: знает, когда она "свежая" (staleTime), когда нужно обновить (refetch), когда показать кэшированное значение пока грузится новое (stale-while-revalidate). Этот паттерн устраняет 90% кода, который ты писал в ViewModel.

🍎 Swift ViewModel
class TeamsViewModel: ObservableObject { @Published var teams: [Team] = [] @Published var isLoading = false @Published var error: Error? private var refreshTask: Task<Void, Never>? func load() async { isLoading = true do { teams = try await api.fetchTeams() } catch { self.error = error } isLoading = false } func refresh() async { // Отменить предыдущий refreshTask?.cancel() refreshTask = Task { await load() } } }
⚛️ React Query
// Автоматически: loading, error, caching, // refetch, background updates, deduplication function TeamsScreen() { const { data: teams, isLoading, error, refetch, isRefetching } = useQuery({ queryKey: ['teams'], queryFn: () => api.fetchTeams(), staleTime: 5 * 60 * 1000, // 5 min cache }); // Всё остальное React Query делает сам: // - Кэширование // - Дедупликация запросов // - Background refetch // - Retry on error }

Сравни объём кода: Swift ViewModel — 20+ строк ручного управления состоянием (и это ещё без кэширования и дедупликации!). React Query — одна декларация. queryKey: ['teams'] — уникальный ключ кэша. Если два экрана одновременно запросят ['teams'], React Query отправит только один HTTP-запрос и раздаст результат обоим. staleTime — как долго данные считаются "свежими": 5 минут не будет повторного запроса. Это как NSCache + URLSessionConfiguration.requestCachePolicy, но декларативно.

React Query — это как URLSession + NSCache + Combine publisher + error handling + retry logic + request deduplication — всё в одном. В iOS ты бы строил Repository layer, кэш-менеджер, и координировал это через Combine. Здесь это один вызов useQuery. Stale-while-revalidate паттерн означает: показать кэшированные данные мгновенно, одновременно загрузить свежие в фоне — пользователь не видит спиннер при повторном заходе на экран.
Не храни серверные данные в Zustand или Context! Типичная ошибка при переходе с iOS: создать глобальный store для teams/players, загрузить данные при старте, и вручную обновлять. React Query делает это лучше: автоматический refetch при возвращении на экран, при восстановлении сети, по таймеру. Zustand — для client state (favorites, настройки). React Query — для server state (API-данные).

✏️ Mutations в React Query

Если useQuery — это чтение данных (GET), то useMutation — это запись (POST, PUT, DELETE). В iOS ты бы написал метод в ViewModel, который отправляет запрос, обрабатывает ответ, обновляет локальное состояние, показывает спиннер и обрабатывает ошибки. Mutation делает то же самое декларативно: ты описываешь "что мутировать" и "что делать при успехе/ошибке".

Ключевая концепция — cache invalidation. После мутации (например, добавления игрока) кэш становится устаревшим. Вместо того чтобы вручную обновлять локальные данные (как ты бы сделал с @Published var players в iOS), ты вызываешь invalidateQueries — React Query автоматически перезагрузит данные с сервера. Это проще и надёжнее, чем ручная синхронизация.

// Mutation = POST/PUT/DELETE (изменение данных) function TeamDetailScreen({ teamId }) { const queryClient = useQueryClient(); // Получение данных const { data: team } = useQuery({ queryKey: ['team', teamId], queryFn: () => api.getTeam(teamId), }); // Mutation для добавления игрока const addPlayerMutation = useMutation({ mutationFn: (player: NewPlayer) => api.addPlayer(teamId, player), onSuccess: () => { // Инвалидировать кэш — данные устарели queryClient.invalidateQueries({ queryKey: ['team', teamId] }); // React Query автоматически перезагрузит }, onError: (error) => { Alert.alert('Error', error.message); }, }); const handleAddPlayer = (player: NewPlayer) => { addPlayerMutation.mutate(player); }; return ( <View> {addPlayerMutation.isPending && <ActivityIndicator />} <Button title="Add Player" onPress={() => handleAddPlayer({ name: 'New Player' })} disabled={addPlayerMutation.isPending} /> </View> ); }

Для ещё лучшего UX существуют optimistic updates — ты обновляешь UI сразу, не дожидаясь ответа сервера. Если сервер вернул ошибку — откатываешь изменения. React Query поддерживает это через onMutate (обновить кэш оптимистично), onError (откатить), onSettled (синхронизировать). В iOS аналог — Core Data с pending changes, которые потом синхронизируются.

invalidateQueries — говорит React Query "эти данные устарели, перезагрузи". Можно инвалидировать по частичному ключу: invalidateQueries({ queryKey: ['team'] }) инвалидирует ВСЕ team-запросы (список, детали). Это как NSNotification, которая говорит всем подписчикам "данные изменились" — но автоматически, без ручных Observer-ов.
Не вызывай mutate() внутри рендера! Mutation — это side effect, его нужно вызывать в обработчиках событий (onPress, onSubmit). Если нужна автоматическая мутация при изменении данных, используй useEffect с mutate. Также помни: mutate не возвращает Promise. Если нужен await — используй mutateAsync.

🏗️ Architecture: iOS patterns → React

Одна из главных ловушек для iOS-разработчика — попытка перенести MVVM/VIPER/Clean Architecture буквально в React. Это не работает, потому что React построен на другой парадигме. В iOS ты разделяешь View, ViewModel, Model как отдельные классы с чёткими зависимостями. В React основной примитив декомпозиции — хук (hook), а не класс. Хуки — это функции, которые инкапсулируют логику и могут использовать другие хуки. Они заменяют и ViewModel, и Use Case, и Repository — но в виде composable функций, а не объектов.

Почему MVVM не переводится напрямую? В iOS ViewModel — это ObservableObject, который живёт дольше View (через @StateObject). В React "ViewModel" — это custom hook, который вызывается внутри компонента и привязан к его жизненному циклу. Нет отдельного объекта-ViewModel — есть набор хуков, которые компонент вызывает для получения данных и действий. Это более плоская архитектура: нет слоёв абстракции ради абстракции.

iOS Pattern React Equivalent ───────────────────────────────────────────── MVC → Component + hooks MVVM → Component + custom hooks Coordinator → Expo Router Repository → API layer + React Query Use Cases → Custom hooks Dependency Injection → Context / Zustand Singleton Services → Global stores

На практике архитектура React-приложения сводится к трём уровням: (1) Components — чистый UI, рендерит JSX на основе props, (2) Hooks — бизнес-логика, загрузка данных, вычисления, (3) Services/Stores — API-клиент, глобальное состояние, утилиты. Это проще, чем VIPER с 5 слоями, но достаточно для масштабирования. Большинство продовых React Native приложений используют именно эту структуру.

В React нет строгого MVC/MVVM. Есть компоненты + hooks + state management. Вместо архитектурных паттернов с классами — функциональная композиция: маленькие хуки собираются в большие, компоненты используют хуки. Если ты захочешь Clean Architecture — custom hook = Use Case, API-модуль = Repository, Context/Zustand = Dependency Injection container.
Опытные iOS-разработчики часто создают избыточные абстракции в React: отдельный ViewModel-класс, протоколы для каждого сервиса, фабрики. Это overeengineering. В React custom hook УЖЕ является ViewModel. useTeamDetail(id) — это и ViewModel, и Use Case одновременно. Не создавай дополнительные классы-обёртки, если хук справляется.

📁 Рекомендуемая структура проекта

Структура проекта в React Native отличается от iOS-проектов. В Xcode ты привык к группам по слоям: Models/, Views/, ViewModels/, Services/. В React-мире тоже есть такое деление, но с другими именами. Важный принцип — colocation (размещение рядом): файлы, которые используются вместе, хранятся в одной папке. Hook useTeams лежит рядом с типами Team и API-вызовами для teams.

Папка app/ — единственная "магическая" директория: Expo Router сканирует её для построения маршрутов. Всё остальное — обычные папки, организованные по конвенции. Нет файла вроде .xcodeproj, который определяет структуру — ты сам решаешь, как организовать код.

src/ ├── app/ # Expo Router (screens) │ ├── _layout.tsx │ ├── (tabs)/ │ │ ├── _layout.tsx │ │ ├── index.tsx │ │ └── ... │ └── team/[id].tsx │ ├── components/ # Переиспользуемые UI компоненты │ ├── TeamRow.tsx │ ├── LoadingSpinner.tsx │ └── ErrorView.tsx │ ├── hooks/ # Custom hooks (бизнес-логика) │ ├── useTeams.ts │ ├── useTeamDetail.ts │ └── useAuth.ts │ ├── stores/ # Zustand stores (global state) │ ├── useFavoritesStore.ts │ └── useAuthStore.ts │ ├── services/ # API, networking │ ├── api.ts │ └── types.ts │ ├── utils/ # Helpers │ └── formatters.ts │ └── constants/ # Config, theme ├── colors.ts └── config.ts

Разберём каждую папку: app/ содержит только экраны и layouts — минимум логики, максимум UI. components/ — переиспользуемые UI-элементы (как UIView-подклассы). hooks/ — бизнес-логика, привязанная к домену (как ViewModel-ы). stores/ — глобальное состояние (как Singleton-сервисы). services/ — API-клиент и типы (как Networking layer). utils/ — чистые функции без побочных эффектов. constants/ — конфигурация, цвета, размеры (как Assets.xcassets + Constants.swift).

По мере роста проекта можно перейти к feature-based структуре: features/teams/ содержит компоненты, хуки, типы и сервисы для teams. Это как модули в iOS (SPM packages), но без формальных границ. Когда стоит переходить? Когда в hooks/ скопилось 20+ файлов и становится трудно найти нужный. Начинай с плоской структуры и рефактори по мере необходимости.
Не помещай бизнес-логику в app/ (экраны). Экран должен только вызывать хуки и рендерить UI. Если видишь useState + useEffect + вычисления в файле экрана — пора выносить в custom hook. Правило: файл экрана не должен превышать 50-80 строк. Вся сложность — в хуках.

🧹 Clean Architecture: Custom Hook как Use Case

Это центральный паттерн React-архитектуры, и он элегантно решает задачу Clean Architecture без лишних слоёв. Custom hook объединяет все источники данных (сервер, локальное состояние, глобальный store) и предоставляет компоненту единый интерфейс. В Clean Architecture терминах это Use Case / Interactor — он координирует получение данных из разных репозиториев и предоставляет результат Presenter-у (компоненту).

Преимущество перед MVVM: в iOS ViewModel — это класс, который нужно инициализировать, держать ссылку (@StateObject), управлять его жизненным циклом. Custom hook — это просто вызов функции. Его можно вызвать условно, можно вызвать несколько раз, можно передать другому хуку. Хуки composable: useTeamDetail внутри использует useQuery, useState и useFavoritesStore — каждый из них может быть заменён независимо.

// hooks/useTeamDetail.ts // Это как Use Case / Interactor в Clean Architecture export function useTeamDetail(teamId: string) { // Server state (React Query) const { data: team, isLoading, error } = useQuery({ queryKey: ['team', teamId], queryFn: () => api.getTeam(teamId), enabled: !!teamId, // Не запускать если нет ID }); // Local state const [isEditing, setIsEditing] = useState(false); // Global state (Zustand) const { isFavorite, toggleFavorite } = useFavoritesStore(); // Derived state const favorite = useMemo(() => isFavorite(teamId), [teamId, isFavorite]); // Actions const handleFavoriteToggle = useCallback(() => { toggleFavorite(teamId); }, [teamId, toggleFavorite]); // Return ViewModel-like interface return { // State team, isLoading, error, isEditing, favorite, // Actions setIsEditing, toggleFavorite: handleFavoriteToggle, }; } // Использование в компоненте — чистый UI! function TeamDetailScreen() { const { id } = useLocalSearchParams(); const { team, isLoading, favorite, toggleFavorite } = useTeamDetail(id); if (isLoading) return <Loading />; return <TeamDetailView team={team} favorite={favorite} onToggle={toggleFavorite} />; }

Обрати внимание на тестируемость: useTeamDetail можно протестировать отдельно от UI через renderHook. API мокается, Zustand store мокается — и ты проверяешь всю логику без рендеринга компонентов. В iOS для этого ты бы создал протоколы для каждой зависимости ViewModel и инжектил моки. Здесь зависимости (api, store) можно мокнуть через jest.mock на уровне модулей — проще и менее церемониально.

Dependency injection в React происходит через модульную систему (import) и React Context, а не через инициализаторы. Для тестов ты мокаешь модули через jest.mock('./api'). Для runtime DI — передаёшь зависимости через Context. Не нужны фабрики, контейнеры, Swinject — модульная система JS и есть твой DI-контейнер.
Правило хорошего хука: он возвращает объект с данными (state) и действиями (actions), как ViewModel. Компонент деструктурирует результат и рендерит UI. Если хук возвращает больше 7-8 полей — пора разбить на два хука. Композиция хуков — главный инструмент управления сложностью в React.

🛡️ Error Boundaries — ловим crashes

В iOS необработанная ошибка в UI-коде приводит к крэшу всего приложения. В React аналогично: если компонент выбросит исключение при рендере, всё приложение "белеет" (white screen of death). Error Boundary — это компонент-обёртка, который ловит ошибки в дочерних компонентах и показывает fallback UI вместо крэша. Это как do-catch для целого поддерева компонентов.

Важная деталь: Error Boundary обязан быть class-компонентом. Это единственное место в современном React, где нужен class вместо function. Хуков для Error Boundary не существует (пока). Метод getDerivedStateFromError вызывается при ошибке рендера, а componentDidCatch — для логирования (как Crashlytics/Sentry). На практике большинство проектов используют библиотеку react-error-boundary, которая предоставляет готовый компонент с хуками.

// components/ErrorBoundary.tsx import { Component, ReactNode } from 'react'; interface Props { children: ReactNode; fallback?: ReactNode; } interface State { hasError: boolean; error?: Error; } class ErrorBoundary extends Component<Props, State> { state: State = { hasError: false }; static getDerivedStateFromError(error: Error): State { return { hasError: true, error }; } componentDidCatch(error: Error, info: React.ErrorInfo) { // Логирование в Sentry, Crashlytics и т.д. console.error('ErrorBoundary caught:', error, info); } render() { if (this.state.hasError) { return this.props.fallback || ( <View style={styles.container}> <Text>Something went wrong</Text> <Button title="Try again" onPress={() => this.setState({ hasError: false })} /> </View> ); } return this.props.children; } } // Использование в _layout.tsx export default function RootLayout() { return ( <ErrorBoundary> <QueryClientProvider client={queryClient}> <Stack /> </QueryClientProvider> </ErrorBoundary> ); }
Error Boundary — это try/catch для React компонентов. Ловит ошибки рендеринга. В iOS аналог — это NSSetUncaughtExceptionHandler + fallback UI. Стратегия: размести ErrorBoundary на корневом уровне (ловит всё) и на уровне отдельных экранов/фич (изоляция ошибки в одном разделе не убивает всё приложение).
Error Boundary НЕ ловит: (1) ошибки в event handlers (onPress, onChange) — для них используй try/catch, (2) асинхронные ошибки (setTimeout, fetch) — для них React Query.onError или собственный try/catch, (3) ошибки на сервере (SSR). Error Boundary ловит только ошибки в render-фазе. Это важное ограничение — многие ожидают, что он ловит ВСЁ, как глобальный exception handler.
Хорошая стратегия: Error Boundary + React Query error handling + try/catch в event handlers. Вместе они покрывают все случаи. React Query уже встроенно обрабатывает сетевые ошибки (retry, error state). Error Boundary ловит ошибки рендера. Try/catch — для остального. Это более гранулярно, чем глобальный crash handler в iOS.

📝 TypeScript: Типы для React Native

Если ты привык к Swift, TypeScript покажется знакомым, но с важными отличиями. Swift — это language с типами первого класса: struct, class, enum, protocol. TypeScript — это надстройка над JavaScript, и типы существуют только при компиляции (стираются в runtime). Это значит: нет is / as для runtime type checking, нет Codable для автоматической десериализации. Но TypeScript даёт то же, что Swift — автодополнение, ошибки при компиляции и документацию через типы.

В React Native типизация особенно важна в четырёх местах: (1) модели данных (Team, Player), (2) props компонентов, (3) параметры навигации, (4) возвращаемые значения хуков. Ниже — конвенции для каждого.

// types/team.ts export interface Team { id: string; name: string; country: string; founded: number; stadium: string; players?: Player[]; } export interface Player { id: string; name: string; position: 'GK' | 'DF' | 'MF' | 'FW'; number: number; } // API responses export interface ApiResponse<T> { data: T; meta?: { total: number; page: number; }; } // Component props export interface TeamRowProps { team: Team; onPress: (team: Team) => void; showFavorite?: boolean; } // Screen params (Expo Router) export type TeamDetailParams = { id: string; tab?: 'info' | 'players' | 'matches'; }; // Hook return types export interface UseTeamsResult { teams: Team[]; isLoading: boolean; error: Error | null; refetch: () => void; }

Обрати внимание на interface vs type: в TypeScript они почти взаимозаменяемы. Конвенция: interface для объектных форм (props, модели), type для union-ов и сложных типов. ApiResponse<T> — generic, как Result<T, Error> в Swift. position: 'GK' | 'DF' | 'MF' | 'FW' — это string literal union, аналог enum Position: String в Swift, но без отдельного типа.

В Swift ты привык к Codable для маппинга JSON → struct. В TypeScript нет runtime проверки типов: если API вернул неправильный JSON, TypeScript НЕ выдаст ошибку — данные просто будут не того типа. Для runtime-валидации используй библиотеку Zod: const TeamSchema = z.object({ id: z.string(), ... }). Это как Codable, но явный и с кастомными ошибками.
Частая ошибка: определять generic компоненты без правильной типизации событий. Нативные события в React Native типизированы: onPress: (event: GestureResponderEvent) => void, onChangeText: (text: string) => void. Не используй any для event handlers — это убивает весь смысл TypeScript. Также: для платформо-специфичного кода используй Platform.OS с типами 'ios' | 'android'.

🎯 Discriminated Unions — типизация состояний

Discriminated unions — это ответ TypeScript на Swift enums with associated values. В Swift enum LoadingState<T> с case-ами — это единый тип, где каждый вариант может нести свои данные. В TypeScript аналогичная концепция, но реализована через union объектных типов с общим "дискриминантом" — полем, по которому TypeScript различает варианты.

Почему это важно для React? Потому что типичный анти-паттерн — хранить loading, error и data как отдельные boolean/nullable поля: isLoading: boolean, error: Error | null, data: T | null. Это позволяет невозможные состояния: isLoading: true, error: someError — одновременно загрузка и ошибка. Discriminated union делает невозможные состояния невыразимыми — если status === 'loading', поля data и error не существуют.

🍎 Swift enum with associated values
enum LoadingState<T> { case idle case loading case success(T) case error(Error) } // Использование switch state { case .idle: showPlaceholder() case .loading: showSpinner() case .success(let data): showData(data) case .error(let error): showError(error) }
⚛️ TypeScript discriminated union
type LoadingState<T> = | { status: 'idle' } | { status: 'loading' } | { status: 'success'; data: T } | { status: 'error'; error: Error }; // Использование function renderState(state: LoadingState<Team[]>) { switch (state.status) { case 'idle': return <Placeholder />; case 'loading': return <Spinner />; case 'success': return <TeamList teams={state.data} />; // TypeScript знает что data есть! case 'error': return <Error message={state.error.message} />; } }

TypeScript выполняет exhaustive checking — если ты не обработал все варианты в switch, компилятор выдаст ошибку. Для этого добавь default: const _exhaustive: never = state; return _exhaustive; в конец switch. Тип never означает "этот код никогда не должен выполняться". Если добавишь новый вариант в union и забудешь обработать — TypeScript поймает это при компиляции. В Swift это работает из коробки для enum.

TypeScript автоматически сужает тип внутри switch case (type narrowing). Как pattern matching в Swift. В case 'success' TypeScript знает, что state.data существует и имеет тип T. В case 'error' — что state.error существует. Обращение к state.data в case 'loading' вызовет ошибку компиляции — невозможное состояние невыразимо.
React Query уже использует этот паттерн внутри: useQuery возвращает объект с полями status: 'pending' | 'error' | 'success'. Вместо проверки isLoading && !error && data можно проверять status === 'success' — и TypeScript автоматически типизирует data как non-null. Используй это вместо отдельных boolean-проверок.

🧪 Testing: Jest + React Native Testing Library

Философия тестирования в React принципиально отличается от iOS. В XCTest ты часто тестируешь реализацию: проверяешь, что label содержит текст, что кнопка вызывает метод, что ViewModel имеет определённое свойство. React Native Testing Library (RNTL) продвигает другой подход: тестируй поведение, а не реализацию. Тест должен взаимодействовать с компонентом так, как пользователь — нажимать, вводить текст, читать результат.

Практическое отличие: в iOS тесте ты обращаешься к sut.nameLabel.text — деталь реализации (label может быть заменён на другой элемент). В RNTL ты вызываешь screen.getByText('Real Madrid') — проверяешь, что текст видим пользователю, не важно в каком элементе. Это делает тесты устойчивыми к рефакторингу: замена Text на Animated.Text не сломает тест.

🍎 XCTest
import XCTest @testable import MyApp class TeamRowTests: XCTestCase { func testDisplaysTeamName() { let team = Team(id: "1", name: "Real Madrid") let sut = TeamRow(team: team) XCTAssertEqual(sut.nameLabel.text, "Real Madrid") } func testFavoriteButtonTapped() { var tapped = false let sut = TeamRow( team: mockTeam, onFavorite: { tapped = true } ) sut.favoriteButton.sendActions(for: .touchUpInside) XCTAssertTrue(tapped) } }
⚛️ Jest + RNTL
import { render, fireEvent, screen } from '@testing-library/react-native'; import { TeamRow } from './TeamRow'; describe('TeamRow', () => { const mockTeam = { id: '1', name: 'Real Madrid', country: 'Spain' }; it('displays team name', () => { render(<TeamRow team={mockTeam} onPress={jest.fn()} />); expect(screen.getByText('Real Madrid')).toBeTruthy(); }); it('calls onPress when tapped', () => { const onPress = jest.fn(); render(<TeamRow team={mockTeam} onPress={onPress} />); fireEvent.press(screen.getByText('Real Madrid')); expect(onPress).toHaveBeenCalledWith(mockTeam); }); });

Что тестировать: (1) рендеринг — правильные данные отображаются, (2) взаимодействие — нажатия вызывают правильные callback-и, (3) условный рендеринг — loading/error/empty показывают правильный UI, (4) accessibility — элементы имеют правильные роли и метки. Чего НЕ тестировать: стили (они визуальные), внутреннее состояние (оно деталь), порядок вызовов хуков (это ограничение React).

В iOS юнит-тесты запускаются в Simulator — медленно. Jest запускает тесты в Node.js — быстро (секунды, не минуты). RNTL не рендерит на экране: он создаёт виртуальное дерево компонентов и проверяет его структуру. Это аналог snapshot testing, но с семантическими запросами вместо сравнения пикселей. Скорость тестов в React — главное преимущество для TDD.
jest.fn() — это мок-функция (аналог Mock в XCTest). Она запоминает все вызовы: expect(onPress).toHaveBeenCalledTimes(1), expect(onPress).toHaveBeenCalledWith(mockTeam). Не забывай передавать jest.fn() для обязательных callback props — иначе компонент выбросит ошибку при попытке вызвать undefined.

🪝 Testing Custom Hooks

Тестирование хуков — уникальная задача, которой нет в iOS. Хук — это функция, но она не может вызываться вне React-компонента (из-за правил хуков). Поэтому RNTL предоставляет renderHook — утилиту, которая создаёт минимальный компонент-обёртку, вызывает хук, и возвращает доступ к его результату через result.current.

Если хук зависит от Context (React Query, Auth, и т.д.), нужно обернуть его в соответствующие Provider-ы через параметр wrapper. Это как dependency injection в iOS тестах: ты создаёшь тестовое окружение с моками. retry: false в QueryClient — чтобы тесты не делали повторных попыток при ошибках, иначе они будут медленными.

import { renderHook, waitFor } from '@testing-library/react-native'; import { QueryClient, QueryClientProvider } from '@tanstack/react-query'; import { useTeams } from './useTeams'; // Wrapper для React Query const createWrapper = () => { const queryClient = new QueryClient({ defaultOptions: { queries: { retry: false } } }); return ({ children }) => ( <QueryClientProvider client={queryClient}> {children} </QueryClientProvider> ); }; describe('useTeams', () => { it('fetches teams on mount', async () => { // Mock API jest.spyOn(api, 'fetchTeams').mockResolvedValue([ { id: '1', name: 'Real Madrid' }, { id: '2', name: 'Barcelona' }, ]); const { result } = renderHook(() => useTeams(), { wrapper: createWrapper(), }); // Initial state expect(result.current.isLoading).toBe(true); expect(result.current.teams).toEqual([]); // After fetch await waitFor(() => { expect(result.current.isLoading).toBe(false); }); expect(result.current.teams).toHaveLength(2); expect(result.current.teams[0].name).toBe('Real Madrid'); }); });

waitFor — это аналог XCTestExpectation + waitForExpectations(timeout:) в XCTest. Он повторяет проверку до тех пор, пока она не пройдёт (или не наступит таймаут). Это нужно для асинхронных операций: загрузка данных не мгновенна, и нужно дождаться, пока хук обновит своё состояние. Без waitFor тест проверит начальное состояние (loading: true) и завершится до получения данных.

jest.spyOn(api, 'fetchTeams').mockResolvedValue([...]) — мокает API-вызов. mockResolvedValue для Promise (async), mockReturnValue для синхронных функций. В iOS ты бы создал протокол APIClient и инжектил мок через init. В JS мокание модулей через jest.spyOn проще — не нужны протоколы и фабрики. Но помни: мок нужно сбрасывать между тестами через jest.restoreAllMocks() в afterEach.
Если хук обновляет state (вызывает setState), и ты проверяешь результат вне waitFor или act(), React выдаст предупреждение "An update was not wrapped in act(...)". act() — это обёртка, которая гарантирует, что все обновления состояния обработаны перед проверкой. waitFor уже включает act() внутри, поэтому для async хуков используй waitFor, а для синхронных действий — act(() => { result.current.toggleFavorite('1'); }).
Тестирование хуков отдельно от компонентов — это аналог тестирования ViewModel в изоляции от View. Ты проверяешь логику (загрузка, обработка ошибок, вычисления), не зависимая от UI. Компоненты тестируешь через render + fireEvent — это integration test. Такое разделение даёт высокое покрытие с минимальным количеством тестов.

📡 Real-time в мобильных приложениях

Real-time коммуникация -- это фундамент ride-sharing приложений. В iOS ты привык работать с URLSession, Combine и CoreLocation напрямую, и у тебя был полный контроль над жизненным циклом соединения. В React Native подход другой: ты работаешь через JavaScript-обёртки над нативными модулями, и управление ресурсами происходит через хуки (useEffect с cleanup). Главная ментальная модель, которую нужно принять -- вместо императивного "создай, настрой, запусти, останови" ты описываешь декларативно "когда этот компонент активен, поддерживай соединение".

Существует три основных подхода к real-time обновлениям, и выбор зависит от задачи. Polling (периодические HTTP-запросы) -- самый простой, подходит для данных, которые обновляются раз в 30-60 секунд (например, статус заказа). Server-Sent Events (SSE) -- однонаправленный поток от сервера к клиенту, хорош для уведомлений и лент. WebSocket -- полнодуплексное соединение, необходимо когда нужен обмен данными в обе стороны в реальном времени (чат, live tracking). Для Onward Rides WebSocket -- очевидный выбор: водитель отправляет координаты, пассажир получает обновления, оба могут обмениваться сообщениями.

iOS React Native ─────────────────────────────────────────────── URLSessionWebSocketTask → WebSocket API / Socket.io CoreLocation → expo-location MapKit → react-native-maps Push Notifications → expo-notifications Background Tasks → expo-task-manager Combine Publishers → RxJS / Event Emitters

В контексте Onward Rides real-time архитектура выглядит так: когда пассажир заказывает поездку, приложение открывает WebSocket-соединение и подписывается на обновления конкретного ride. Сервер пушит координаты водителя каждые 2-5 секунд, обновления ETA и смену статусов. Параллельно Push Notifications используются как fallback -- если приложение уходит в background, пользователь получит нотификацию "Водитель приехал". Это сочетание WebSocket + Push -- стандартный паттерн для ride-sharing (так работают Uber, Lyft, Bolt).

Для ride-sharing приложения (Onward) ключевые real-time фичи:

  • 🚗 Live location tracking (driver position)
  • 📍 ETA updates
  • 🔔 Push notifications (ride assigned, driver arrived)
  • 💬 In-app messaging
В iOS ты мог использовать Combine Publisher для стриминга данных через цепочку операторов (map, filter, debounce). В React Native эквивалент -- это комбинация WebSocket + useState + useEffect. Реактивность встроена в React: когда setLocation обновляет стейт, все зависимые компоненты автоматически перерисовываются. Тебе не нужен отдельный reactive framework.
В iOS background location + WebSocket работает "из коробки" с правильными capabilities. В React Native background-режим значительно ограничен: WebSocket может разрываться при переходе в background, а expo-task-manager для фоновых задач требует специальной настройки и работает не так предсказуемо, как BGTaskScheduler в iOS.

🔌 WebSocket = Persistent Connection

WebSocket -- это протокол, который устанавливает постоянное двунаправленное TCP-соединение между клиентом и сервером. В отличие от HTTP (request-response), WebSocket держит канал открытым и позволяет серверу пушить данные клиенту в любой момент. Это критически важно для ride tracking, где задержка даже в 1-2 секунды заметна пользователю.

Жизненный цикл WebSocket-соединения: Handshake (HTTP Upgrade request) → Open (соединение установлено) → Messages (обмен данными) → Close (одна из сторон закрывает). В production-приложении нужно обрабатывать все фазы: показывать "Connecting..." при handshake, обрабатывать ошибки при обрыве, и реализовывать reconnection strategy (exponential backoff: 1s → 2s → 4s → 8s → max 30s). Также важен heartbeat/ping-pong механизм -- периодические ping-сообщения подтверждают, что соединение живо, и помогают обнаружить "zombie connections" (когда TCP-соединение формально открыто, но данные не проходят).

function useRideTracking(rideId: string) { const [location, setLocation] = useState<Location | null>(null); const [eta, setEta] = useState<number | null>(null); const wsRef = useRef<WebSocket | null>(null); useEffect(() => { if (!rideId) return; const ws = new WebSocket(`wss://api.app.com/rides/${rideId}`); wsRef.current = ws; ws.onopen = () => console.log('WS connected'); ws.onmessage = (event) => { const update = JSON.parse(event.data); if (update.type === 'location') { setLocation(update.location); } else if (update.type === 'eta') { setEta(update.eta); } }; ws.onerror = (e) => console.error('WS error:', e); ws.onclose = () => console.log('WS closed'); return () => ws.close(); // Cleanup! }, [rideId]); return { location, eta }; }

Ключевые различия в поведении: в Swift ты должен рекурсивно вызывать receiveMessage() после каждого полученного сообщения -- URLSessionWebSocketTask не делает это автоматически. В React Native WebSocket API основан на событиях (onmessage вызывается для каждого сообщения автоматически). Также обрати внимание на управление памятью: в Swift тебе нужен [weak self] чтобы избежать retain cycle. В React -- cleanup функция в useEffect (return () => ws.close()) обеспечивает закрытие соединения при размонтировании компонента или при смене rideId.

URLSessionWebSocketTask имеет встроенный метод sendPing() для heartbeat. В React Native WebSocket API нативного ping нет -- нужно реализовывать через обычные сообщения: setInterval(() => ws.send(JSON.stringify({type: 'ping'})), 30000). Также в iOS URLSessionWebSocketTask поддерживает и .string и .data типы сообщений, а в RN WebSocket всё приходит как строка в event.data.
В React Native WebSocket не reconnect'ится автоматически при обрыве. Если пользователь вошёл в тоннель метро и вышел, соединение будет потеряно. Нужно реализовывать reconnection вручную: отслеживать onclose, запускать таймер с exponential backoff, и пробовать переподключиться. Именно поэтому для production рекомендуется Socket.io (следующий слайд) -- он решает это из коробки.

🔄 Socket.io — удобнее чем raw WebSocket

Socket.io -- это библиотека поверх WebSocket, которая решает все боли production real-time приложений. Почему не raw WebSocket? Потому что в реальном мире тебе нужны: автоматический reconnect с exponential backoff, fallback на HTTP long-polling если WebSocket недоступен (корпоративные прокси!), rooms (подписка на конкретную поездку), namespaces (разделение трафика /rides, /chat, /notifications), и подтверждение доставки (acknowledgements). Всё это Socket.io даёт из коробки.

Для iOS-разработчика ключевое отличие в паттерне: вместо delegate-based подхода (CLLocationManagerDelegate, URLSessionWebSocketDelegate) Socket.io использует event-driven паттерн. Ты подписываешься на именованные события: socket.on('ride:location', handler). Это ближе к NotificationCenter.addObserver, но типизировано и с гарантией доставки. Паттерн emit/on -- это по сути pub/sub, и он очень естественно ложится на React-модель.

// hooks/useRideSocket.ts import { io, Socket } from 'socket.io-client'; interface RideUpdate { type: 'location' | 'eta' | 'status'; data: any; } export function useRideSocket(rideId: string) { const [connected, setConnected] = useState(false); const [location, setLocation] = useState<LatLng | null>(null); const [status, setStatus] = useState<RideStatus>('pending'); const socketRef = useRef<Socket | null>(null); useEffect(() => { const socket = io('https://api.onward.app', { query: { rideId }, transports: ['websocket'], // Важно для React Native! }); socketRef.current = socket; socket.on('connect', () => setConnected(true)); socket.on('disconnect', () => setConnected(false)); socket.on('ride:location', (data) => { setLocation({ latitude: data.lat, longitude: data.lng }); }); socket.on('ride:status', (data) => { setStatus(data.status); }); // Cleanup return () => { socket.disconnect(); }; }, [rideId]); // Send message const sendMessage = useCallback((message: string) => { socketRef.current?.emit('ride:message', { rideId, message }); }, [rideId]); return { connected, location, status, sendMessage }; }

Обрати внимание на transports: ['websocket'] -- это критически важно для React Native. По умолчанию Socket.io сначала пробует HTTP long-polling и потом апгрейдится до WebSocket. В браузере это нормально, но в React Native long-polling работает плохо и создаёт лишний overhead. Указав только websocket, ты говоришь: "используй WebSocket сразу". Также заметь, что query: { rideId } передаётся при handshake -- сервер получает rideId и автоматически подписывает клиента на нужный room.

Socket.io автоматически reconnect'ится, имеет rooms/namespaces, и проще в использовании. В iOS аналогом rooms можно считать подписку на конкретный topic в APNS или channel в Firebase. Namespaces -- это как разные URLSession configurations для разных типов трафика. Socket.io объединяет всё в один API с единым соединением (multiplexing).
Socket.io -- это НЕ совместимый с обычным WebSocket протокол. Если сервер использует Socket.io, клиент тоже должен использовать Socket.io. Нельзя подключиться обычным new WebSocket() к Socket.io серверу. Учитывай это при обсуждении архитектуры на интервью -- скажи: "Socket.io добавляет свой protocol layer поверх WebSocket для reliability features".
На интервью часто спрашивают: "Почему Socket.io, а не raw WebSocket?" Ответ: raw WebSocket -- это low-level транспорт (как TCP socket). Socket.io -- это application-level библиотека, которая даёт reconnection, multiplexing (rooms), event-based API, и fallback. Для MVP можно начать с raw WebSocket, но для production ride-sharing -- Socket.io или аналог (например, Ably, Pusher) практически обязателен.

📍 Location Services

Location -- одна из областей, где опыт iOS-разработчика даёт значительное преимущество. Ты уже знаешь все нюансы: разница между WhenInUse и Always, точность vs батарея, фоновые обновления, geofencing. В React Native/Expo всё это тоже есть, но модель разрешений абстрагирована через expo-location, и некоторые вещи работают иначе.

Модель разрешений: в iOS ты вызываешь requestWhenInUseAuthorization() или requestAlwaysAuthorization() и обрабатываешь результат через delegate. В Expo это async/await: вызов requestForegroundPermissionsAsync() показывает системный диалог и возвращает результат. Для фонового доступа нужен отдельный вызов requestBackgroundPermissionsAsync(). Важно: Expo абстрагирует и iOS, и Android разрешения -- на Android модель другая (ACCESS_FINE_LOCATION, ACCESS_COARSE_LOCATION, ACCESS_BACKGROUND_LOCATION), и Expo унифицирует это для тебя.

На практике водительское приложение часто делают native, а пассажирское (которому нужен только foreground tracking) -- на React Native.

📍 Continuous Location Updates

Continuous tracking -- это когда ты подписываешься на поток обновлений локации, как startUpdatingLocation() в CoreLocation. В Expo это реализуется через watchPositionAsync, который возвращает объект subscription. Это subscription-based паттерн: ты начинаешь "наблюдение", получаешь callback при каждом обновлении, и в конце отменяешь подписку. Концептуально это как Combine sink -- ты подписался, получаешь значения, и можешь отменить через cancellable.

Два ключевых параметра для баланса точности и батареи: distanceInterval (минимальное расстояние в метрах между обновлениями) и timeInterval (минимальное время между обновлениями в миллисекундах). Для ride tracking водителя оптимально distanceInterval: 10 метров + timeInterval: 5000ms. Для навигации с пошаговыми инструкциями -- distanceInterval: 5 + BestForNavigation accuracy. Для "показать где я на карте" -- distanceInterval: 50 + Balanced accuracy. Чем точнее -- тем больше расход батареи, это такой же tradeoff как в iOS.

// hooks/useLocationTracking.ts import * as Location from 'expo-location'; export function useLocationTracking(enabled: boolean) { const [location, setLocation] = useState<LatLng | null>(null); const [heading, setHeading] = useState<number | null>(null); useEffect(() => { if (!enabled) return; let locationSubscription: Location.LocationSubscription; let headingSubscription: Location.LocationSubscription; (async () => { const { status } = await Location.requestForegroundPermissionsAsync(); if (status !== 'granted') return; // Watch position (как startUpdatingLocation) locationSubscription = await Location.watchPositionAsync( { accuracy: Location.Accuracy.BestForNavigation, distanceInterval: 10, // meters timeInterval: 5000, // ms }, (loc) => { setLocation({ latitude: loc.coords.latitude, longitude: loc.coords.longitude, }); } ); // Watch heading (compass) headingSubscription = await Location.watchHeadingAsync((h) => { setHeading(h.trueHeading); }); })(); return () => { locationSubscription?.remove(); headingSubscription?.remove(); }; }, [enabled]); return { location, heading }; }

Обрати внимание на паттерн if (!enabled) return; в начале useEffect. Параметр enabled в dependency array [enabled] означает: когда enabled меняется с false на true -- useEffect запускается и создаёт подписки. Когда enabled меняется с true на false -- сначала вызывается cleanup (remove подписок), а потом useEffect запускается заново, но сразу выходит через return. Это элегантный способ управлять "включено/выключено" декларативно -- аналог того, как в iOS ты бы вызывал startUpdatingLocation/stopUpdatingLocation.

В CoreLocation heading -- это CLHeading с magneticHeading и trueHeading, и ты получаешь его через delegate. В Expo watchHeadingAsync -- отдельная подписка (не часть location). trueHeading работает только когда есть location fix (нужен GPS для коррекции магнитного склонения). На практике для карты с иконкой машины тебе нужен heading чтобы показать направление движения -- без него иконка будет "смотреть" всегда на север.
Не забывай cleanup subscription! Иначе battery drain + memory leak. Типичная ошибка: async функция внутри useEffect создаёт подписку, но если компонент размонтировался до завершения await, cleanup уже вызвался, а subscription ещё не присвоена переменной. Решение: используй флаг isMounted или AbortController, и проверяй перед созданием подписки.
Каждый вызов setLocation вызывает re-render компонента. Если updates приходят каждые 2 секунды -- это 30 re-renders в минуту. Для одного компонента это нормально, но если location пробрасывается через props в глубокое дерево компонентов -- используй React.memo на дочерних компонентах или выноси location в Zustand store и подписывайся точечно. В iOS аналогичная проблема решается через @Published + ObservableObject.

🗺️ Maps: react-native-maps

react-native-maps -- это мост к нативным картам: на iOS под капотом работает Apple Maps (MKMapView), на Android -- Google Maps. Это значит, что ты получаешь нативную производительность и привычный look & feel. Для iOS-разработчика переход будет естественным: react-native-maps по сути оборачивает MKMapView в React-компонент с декларативным API.

Основные props MapView: region (контролируемая позиция камеры с latitude, longitude, latitudeDelta, longitudeDelta) vs initialRegion (начальная позиция, карта потом управляется пользователем). ⚡ Performance: React Native Specifics

Производительность в React Native устроена принципиально иначе, чем в нативном iOS. В Swift/UIKit всё работает в одном процессе: UI-рендеринг, бизнес-логика, обработка данных -- всё на стороне native с прямым доступом к GPU. В React Native существует JS thread (где работает вся твоя логика, React reconciliation, обработка данных) и UI thread (где происходит нативный рендеринг). Между ними -- мост (bridge), через который данные сериализуются в JSON и передаются асинхронно. Это главный источник performance bottleneck'ов.

Хорошая новость: в 2024-2025 React Native перешёл на новую архитектуру (New Architecture). Вместо bridge теперь используются Fabric (новый rendering engine с синхронным UI) и TurboModules (ленивая загрузка нативных модулей + JSI для прямого вызова C++ из JS без JSON-сериализации). Это значительно уменьшает overhead: если в старой архитектуре каждый вызов нативного метода проходил через async JSON bridge, то теперь это синхронный вызов через JSI (JavaScript Interface). На практике это означает: анимации плавнее, жесты отзывчивее, startup быстрее.

Метрики производительности: 1. JS THREAD FPS Цель: 60 fps Проблемы: тяжёлые вычисления, большие списки 2. UI THREAD FPS Цель: 60 fps Проблемы: тяжёлые анимации, layout thrashing 3. RAM USAGE Проблемы: утечки памяти, незакрытые subscriptions 4. BUNDLE SIZE Проблемы: большие зависимости, неиспользуемый код

Для профилирования используй: Flipper -- десктопный инструмент для инспекции React Native приложений (network, layout, database, performance). React DevTools -- для визуализации re-renders (Profiler tab показывает какие компоненты перерисовались и почему). Perf Monitor -- встроенный overlay (доступен через Dev Menu), показывает JS FPS и UI FPS в реальном времени. В iOS ты привык к Instruments (Time Profiler, Allocations), и Flipper -- это ближайший аналог для RN.

В React Native есть 2 потока: JS thread (логика) и UI thread (native рендеринг). Блокировка любого = лаги. Правило: если анимация дёргается при скролле -- проблема в JS thread (тяжёлые вычисления блокируют обработку событий). Если UI не откликается на тап -- проблема в UI thread (тяжёлый layout). Reanimated 3 решает проблему анимаций, вынося их полностью на UI thread через worklets.
В iOS тебе не нужно думать о "двух потоках" -- главный поток один (main thread), и ты просто выносишь тяжёлую работу в background через GCD/async-await. В RN аналог -- это InteractionManager.runAfterInteractions() (аналог DispatchQueue.main.async с приоритетом ниже UI) и использование Web Workers или нативных модулей для CPU-intensive задач.
60 fps = 16.6ms на кадр. В Swift ты имеешь весь этот бюджет на native code. В RN этот бюджет делится между JS thread и UI thread + overhead на коммуникацию. На практике у тебя ~10ms на JS обработку. Это значит: JSON.parse 500 объектов, сортировка массива из 1000 элементов, или 5 вложенных .map/.filter -- могут вызвать frame drop. Используй useMemo для кэширования!

🚀 Performance Optimizations Checklist

Каждая из этих оптимизаций решает конкретную проблему. В iOS ты не думаешь о большинстве из них: UITableViewCell переиспользуется автоматически, стили компилируются, функции не пересоздаются при каждом рендере. В React мире каждый render компонента -- это вызов функции, и всё внутри пересоздаётся заново (объекты, функции, массивы). Поэтому memoization -- это не преждевременная оптимизация, а необходимость для списков и сложных UI.

// ✅ 1. Memoize components const TeamRow = React.memo(function TeamRow({ team, onPress }) { return (...); }); // ✅ 2. Memoize callbacks const handlePress = useCallback((id) => { navigation.push('Details', { id }); }, [navigation]); // ✅ 3. Memoize expensive computations const sortedTeams = useMemo(() => { return [...teams].sort((a, b) => a.name.localeCompare(b.name)); }, [teams]); // ✅ 4. Use StyleSheet.create (not inline styles) const styles = StyleSheet.create({ container: { flex: 1 }, }); // ❌ BAD: <View style={{ flex: 1 }} /> // ✅ 5. FlatList optimizations <FlatList data={data} keyExtractor={(item) => item.id} // NOT index! getItemLayout={(_, index) => ({ length: ITEM_HEIGHT, offset: ITEM_HEIGHT * index, index, })} removeClippedSubviews={true} maxToRenderPerBatch={10} windowSize={5} /> // ✅ 6. Avoid anonymous functions in render // ❌ BAD: onPress={() => handlePress(item.id)} // ✅ GOOD: Pass item to memoized component

1. React.memo -- оборачивает компонент и пропускает re-render, если props не изменились (shallow comparison). Без него TeamRow перерисовывается каждый раз, когда родительский компонент рендерится, даже если team не изменился. Импакт: в списке из 100 элементов вместо 100 re-renders будет 0-1. Аналог в iOS: diffable data source в UITableView, который обновляет только изменённые ячейки.

2. useCallback -- мемоизирует функцию, чтобы она не пересоздавалась при каждом рендере. Без useCallback, handlePress -- новый объект при каждом рендере, и React.memo на дочернем компоненте не работает (props "изменились"). 3. useMemo -- кэширует результат вычисления. Сортировка массива O(n log n) выполняется только когда teams реально изменились, а не при каждом рендере. 4. StyleSheet.create -- оптимизирует стили: объект создаётся один раз и передаётся по ID на native сторону. Inline стили ({ flex: 1 }) создают новый объект каждый рендер.

5. FlatList -- ключевые параметры: getItemLayout позволяет FlatList рассчитать позицию без рендеринга (как estimatedRowHeight в UITableView, но точный). windowSize: 5 означает рендерить только 5 экранов вокруг текущей позиции. removeClippedSubviews отсоединяет невидимые элементы от нативного дерева. 6. Anonymous functions -- () => handlePress(item.id) создаёт новую функцию для каждого элемента при каждом рендере. Решение: передай item как prop в мемоизированный компонент, и вызывай onPress(item.id) внутри него.

В SwiftUI оптимизация списков встроена: List с ForEach автоматически переиспользует view. В React Native тебе нужно явно указать FlatList (а не ScrollView + .map), добавить keyExtractor (аналог Identifiable), и обернуть элемент в React.memo (аналог EquatableView). Это больше ручной работы, но даёт тонкий контроль.
Правило измерения: используй React DevTools Profiler (вкладка "Profiler") чтобы увидеть, какие компоненты рендерятся и сколько времени занимают. Не оптимизируй вслепую! React.memo имеет overhead (shallow comparison), и для простых компонентов может быть медленнее, чем просто re-render. Оптимизируй только то, что реально тормозит.

🖼️ Image Optimization

В iOS ты привык к тому, что UIImageView + NSURLCache + NSCache обеспечивают базовое кэширование, а для продвинутых сценариев используются Kingfisher или SDWebImage. В React Native встроенный компонент Image имеет минимальное кэширование (только на iOS, через URL loading system), и на Android кэш не работает надёжно. Поэтому expo-image -- это must-have замена, аналог Kingfisher для React Native.

Почему expo-image лучше стандартного Image: он построен на нативных библиотеках (SDWebImage на iOS, Glide на Android), поддерживает двухуровневый кэш (memory + disk), progressive loading, анимированные переходы (transition), и blurhash -- алгоритм, который кодирует изображение в строку ~30 символов и показывает размытый placeholder мгновенно, пока грузится реальная картинка. Blurhash -- это то, что использует Instagram и Mastodon для мгновенного отображения лент.

// ❌ BAD: Loading full-size images <Image source={{ uri: team.logoUrl }} style={styles.logo} /> // ✅ GOOD: Use expo-image (caching + progressive loading) import { Image } from 'expo-image'; <Image source={team.logoUrl} style={styles.logo} contentFit="cover" placeholder={blurhash} // Low-res placeholder transition={200} cachePolicy="memory-disk" /> // ✅ Use different sizes for different contexts const thumbnailUrl = `${team.logoUrl}?w=100&h=100`; const fullUrl = `${team.logoUrl}?w=400&h=400`; // ✅ Preload images import { Image } from 'expo-image'; useEffect(() => { Image.prefetch([ team1.logoUrl, team2.logoUrl, // ... ]); }, []);

Обрати внимание на contentFit="cover" -- это аналог contentMode = .scaleAspectFill в UIImageView. Другие значения: "contain" (aspectFit), "fill" (scaleToFill), "none" (center без масштабирования). cachePolicy="memory-disk" говорит: кэшируй и в RAM (для мгновенного доступа при скролле), и на диске (для повторного открытия приложения). transition={200} -- fade-in анимация в миллисекундах при загрузке.

expo-image намного лучше встроенного Image. Кэширование, blurhash, transitions. Аналогия: стандартный Image в RN -- это как UIImageView без Kingfisher. expo-image -- это Kingfisher/SDWebImage с полным набором фич. Также expo-image поддерживает SVG, WebP, AVIF и анимированные GIF/APNG из коробки -- не нужны отдельные библиотеки.
В списках (FlatList) изображения могут вызывать серьёзные лаги. Типичная проблема: загрузка изображения декодируется на JS thread, блокируя скролл. expo-image решает это, декодируя на нативном потоке. Также используй разные размеры: в списке загружай thumbnail (100x100), на детальном экране -- full size. Если API поддерживает resize query params (Cloudinary, Imgix, S3 Lambda) -- это огромный выигрыш в трафике и скорости.
Image.prefetch -- это аналог UIImageView.prefetchURLs в Kingfisher. Используй его перед навигацией: например, на экране списка предзагружай изображения для первых 10 элементов, или при переходе на детальный экран предзагрузи full-size версию. На ride-sharing экране можно предзагрузить аватар водителя и фото машины, пока пользователь видит "Searching for driver...".

✨ Animations: Reanimated 3

В React Native есть два animation API: встроенный Animated (работает на JS thread, может лагать) и Reanimated (работает на UI thread, 60fps всегда). Для production-приложений Reanimated -- единственный правильный выбор. Conceptually это похоже на разницу между UIView.animate (UIKit) и Core Animation (CALayer): второй работает на отдельном потоке и не блокируется бизнес-логикой.

Ключевая концепция Reanimated -- worklets. Worklet -- это JavaScript функция, которая компилируется и выполняется на UI thread (через C++ движок Reanimated, не через JS engine). useAnimatedStyle(() => ...) -- это worklet: он вычисляется на UI thread при каждом кадре. useSharedValue -- это "shared" потому что значение доступно И на JS thread, И на UI thread одновременно. Когда ты пишешь scale.value = withTiming(1.2), анимация полностью происходит на UI thread, JS thread даже не участвует.

🍎 SwiftUI Animation
struct PulsingView: View { @State private var scale: CGFloat = 1.0 var body: some View { Circle() .scaleEffect(scale) .animation( .easeInOut(duration: 1) .repeatForever(), value: scale ) .onAppear { scale = 1.2 } } }
⚛️ Reanimated 3
import Animated, { useSharedValue, useAnimatedStyle, withRepeat, withTiming, } from 'react-native-reanimated'; function PulsingView() { const scale = useSharedValue(1); useEffect(() => { scale.value = withRepeat( withTiming(1.2, { duration: 1000 }), -1, // infinite true // reverse ); }, []); const animatedStyle = useAnimatedStyle(() => ({ transform: [{ scale: scale.value }], })); return <Animated.View style={[styles.circle, animatedStyle]} />; }

Сравнение подходов: в SwiftUI ты просто меняешь @State и система анимирует transition автоматически. В Reanimated ты явно описываешь анимацию: withTiming (аналог .easeInOut), withSpring (аналог .spring), withRepeat (аналог .repeatForever). Это более многословно, но даёт больше контроля. Композиция анимаций: withSequence (последовательно), withDelay (с задержкой). Всё это выполняется на UI thread и не вызывает re-render React-компонентов.

Reanimated анимации работают на UI thread -- никаких лагов даже при сложных анимациях. Это единственная библиотека анимаций для RN, которая гарантирует 60fps. Встроенный Animated API работает на JS thread и будет лагать при тяжёлых вычислениях (загрузка данных, парсинг JSON). Для ride-sharing это критично: анимация перемещения иконки водителя на карте должна быть плавной, даже когда приходят WebSocket-сообщения.
В iOS gesture-driven анимации (drag, swipe) делаются через UIPanGestureRecognizer + UIView.animate или через SwiftUI .gesture + .animation. В React Native для этого используется react-native-gesture-handler + Reanimated. Gesture Handler обрабатывает жесты на UI thread (не через JS bridge), а Reanimated анимирует результат тоже на UI thread. Это связка gesture-handler + reanimated -- аналог direct manipulation в UIKit, где gesture recognizer и animation layer работают без main thread.
Частая ошибка iOS-разработчиков: пытаться анимировать через useState (setScale(1.2) внутри useEffect). Это вызовет мгновенное изменение без анимации + re-render всего компонента. Для анимаций всегда используй useSharedValue + useAnimatedStyle. useState -- для данных, useSharedValue -- для визуальных свойств (позиция, масштаб, прозрачность, цвет).

🔋 Battery Optimization

В iOS ты привык к Background Modes, Background App Refresh и строгим ограничениям от системы. В React Native ситуация аналогичная, но с дополнительными нюансами: JS thread потребляет энергию даже когда "ничего не делает" (если есть активные таймеры, подписки или анимации). Главное правило: минимизируй работу JS thread, особенно в фоне.

iOS Concept → React Native Equivalent ────────────────────────────────────────────────────────────── Background App Refresh → expo-background-fetch Background Location → expo-task-manager + expo-location App Lifecycle (Active/ → AppState API Background/Inactive) ('active' | 'background' | 'inactive') NSTimer → setInterval (⚠️ убивает батарею!) Push Notifications → expo-notifications (APNs + FCM) Doze Mode (Android) → WorkManager через expo-task-manager
🍎 iOS AppDelegate
// App lifecycle func applicationDidEnterBackground(_ app: UIApplication) { stopLocationUpdates() invalidateTimers() saveState() } func applicationWillEnterForeground(_ app: UIApplication) { refreshData() resumeLocationUpdates() }
⚛️ React Native AppState
import { AppState } from 'react-native'; function useAppState() { const appState = useRef(AppState.currentState); useEffect(() => { const sub = AppState.addEventListener( 'change', (nextState) => { if (appState.current === 'background' && nextState === 'active') { refreshData(); // вернулись из фона } if (nextState === 'background') { cleanup(); // уходим в фон } appState.current = nextState; }); return () => sub.remove(); }, []); }

Ключевые стратегии оптимизации батареи:

1. Location polling: используй distanceInterval (метры) вместо timeInterval (секунды). Обновление каждые 100м потребляет в 10x меньше, чем каждые 5 секунд. В фоне переключайся на Significant Location Changes (аналог iOS CLLocationManager.startMonitoringSignificantLocationChanges).

2. WebSocket: в фоне закрывай соединение и используй push notifications вместо постоянного WebSocket. Открытое WS-соединение = постоянный wake lock = мёртвая батарея.

3. Hermes engine: использует bytecode compilation (как AoT), что уменьшает startup time на 50% и memory usage на 30%. Меньше CPU = меньше батареи. Включён по умолчанию в Expo SDK 50+.

Главный враг батареи в RN: таймеры и подписки, которые не очищаются при уходе в фон. Всегда слушай AppState и останавливай: setInterval, location tracking, WebSocket, Reanimated анимации. Возобновляй при возврате в 'active'. Это прямой аналог applicationDidEnterBackground/WillEnterForeground в iOS.
На интервью вопрос "как оптимизировать батарею?" — покажи awareness: (1) AppState для lifecycle, (2) cleanup subscriptions в useEffect return, (3) distanceInterval вместо timeInterval для GPS, (4) push вместо polling/WebSocket в фоне, (5) Hermes для меньшего CPU. Этих 5 пунктов достаточно, чтобы показать senior-уровень.

🎯 Live Coding Strategy

Live coding интервью по React Native -- это не "напиши код молча за 30 минут". Это structured conversation, где ты демонстрируешь мышление senior/lead-инженера. Интервьюер оценивает: как ты уточняешь требования (product thinking), как декомпозируешь задачу (architecture), как принимаешь trade-off решения (engineering maturity), и как общаешься при этом (collaboration). Код -- это только 40% оценки.

Для iOS-разработчика переход к live coding на React Native имеет специфику: ты можешь не знать точный API (забыл параметры FlatList, не помнишь синтаксис useCallback). Это нормально! Скажи: "В iOS я бы использовал UITableView с prefetchDataSource, в React Native аналог -- FlatList с windowSize, давайте я напишу структуру и уточню API". Интервьюер ценит понимание концепций больше, чем знание наизусть синтаксиса.

Структура решения (15-30 минут): 1. CLARIFY (2 мин) - Уточни требования - Спроси про edge cases - Определи scope 2. PLAN (3 мин) - Опиши подход словами - Нарисуй component tree - Определи state и props 3. IMPLEMENT (15-20 мин) - Начни с skeleton - Добавляй функционал итеративно - Комментируй свои решения 4. TEST & REFINE (5 мин) - Проверь edge cases - Обсуди оптимизации - Предложи улучшения

CLARIFY -- задай 2-3 вопроса: "Данные приходят с API или локальные?", "Нужна ли пагинация?", "Должен ли работать offline?". Это показывает product thinking. PLAN -- проговори: "Я разобью это на 3 компонента: Screen, List, ListItem. State будет в Screen: data, loading, error. Использую React Query для фетчинга". Это показывает architecture skills. IMPLEMENT -- начни с types/interfaces, потом skeleton компонента, потом заполняй логику. Пиши сверху вниз: сначала main component, потом hooks, потом helpers. TEST -- вслух пройди по edge cases: "Что если data пустой? Что если network error? Что если юзер быстро нажмёт дважды?".

Говори вслух! Интервьюер хочет понять твой мыслительный процесс, не только результат. Конкретные фразы: "Я начну с типов, чтобы определить контракт", "Здесь я использую useMemo потому что сортировка O(n log n) и не должна выполняться при каждом рендере", "В production я бы добавил error boundary, но для MVP пропущу". Каждое решение должно быть озвучено с причиной.
В iOS интервью ты привык работать в Xcode с автодополнением и компилятором. В React Native live coding обычно в CodeSandbox, StackBlitz, или даже Google Docs. Не паникуй без автодополнения -- сфокусируйся на структуре и логике. Мелкие синтаксические ошибки (забыл скобку, неточное имя пропа) допустимы и не влияют на оценку.
Типичная ошибка senior iOS-разработчиков: переусложнение. Тебя просят список с поиском, а ты начинаешь проектировать MVVM + Coordinator + Repository + Protocol. В React Native live coding ожидают: functional component + hooks + 1 кастомный hook. Начни с самого простого рабочего решения, потом предложи: "В production я бы вынес это в отдельный hook и добавил error handling".

📋 Common Interview Patterns

Эти 4 паттерна покрывают ~80% задач на React Native интервью. Зная их наизусть, ты можешь быстро написать skeleton и сосредоточиться на деталях. Каждый паттерн -- это готовый "рецепт", как Design Patterns в iOS (MVC, Delegate, Observer), но для React UI.

// Pattern 1: List with search/filter function SearchableList({ data }) { const [search, setSearch] = useState(''); const filtered = useMemo(() => data.filter(item => item.name.toLowerCase().includes(search.toLowerCase())), [data, search] ); return (<FlatList data={filtered} ... />); } // Pattern 2: Form with validation function Form() { const [values, setValues] = useState({ name: '', email: '' }); const [errors, setErrors] = useState({}); const validate = () => { ... }; const handleSubmit = () => { if (validate()) submit(values); }; } // Pattern 3: Loading/Error/Data states function DataScreen() { const { data, isLoading, error } = useQuery(...); if (isLoading) return <Loading />; if (error) return <Error onRetry={refetch} />; return <DataView data={data} />; } // Pattern 4: Infinite scroll function InfiniteList() { const { data, fetchNextPage, hasNextPage } = useInfiniteQuery(...); return ( <FlatList data={data.pages.flat()} onEndReached={() => hasNextPage && fetchNextPage()} onEndReachedThreshold={0.5} /> ); }

Pattern 1: Search/Filter -- самый частый на интервью. Ключевые моменты: useMemo для кэширования фильтрации (без него фильтр пересчитывается при каждом рендере), case-insensitive поиск через toLowerCase(), и debounce (если данные с API). В iOS аналог: UISearchController с updateSearchResults delegate. Edge cases: пустой поиск (показать всё), спецсимволы в поиске, пробелы в начале/конце (trim). Интервьюер ожидает, что ты добавишь debounce если данные с сервера.

Pattern 2: Form -- тестирует умение работать с controlled inputs. Каждое поле контролируется через useState, валидация -- отдельная функция. В iOS аналог: UITextField с delegate + валидация перед submit. Edge cases: disable submit при невалидной форме, очистка ошибки при изменении поля, keyboard dismiss при submit. Интервьюер может спросить: "а как бы ты сделал с 20 полями?" -- ответ: react-hook-form или Formik (аналог библиотек валидации форм).

Pattern 3: Loading/Error/Data -- три состояния экрана. Это фундаментальный паттерн, который интервьюер ожидает увидеть ВСЕГДА. Аналог в Swift: enum ViewState { case loading, loaded(Data), error(Error) }. В React Query это встроено. Если не используешь React Query, сделай кастомный хук: const { data, loading, error } = useFetch(url). Pattern 4: Infinite scroll -- пагинация через onEndReached. Аналог: UITableViewDataSourcePrefetching. onEndReachedThreshold: 0.5 значит "начни загрузку когда осталось 50% viewport до конца". Edge cases: loading indicator внизу списка, "нет больше данных", предотвращение двойного fetch.

В iOS ты привык к UITableViewDelegate/DataSource с множеством методов. В React Native все эти patterns укладываются в один functional component с hooks. Интервьюер ожидает, что ты знаешь когда какой pattern применить: показать список -- Pattern 1 или 4, заполнить данные -- Pattern 2, загрузить экран -- Pattern 3. Часто задача комбинирует несколько patterns.
Интервьюер оценивает не только знание паттернов, но и то, что ты НАЧИНАЕШЬ с правильного pattern. Если задача -- "сделай экран со списком пользователей", начни с Pattern 3 (loading/error/data), внутри data -- Pattern 1 (FlatList с поиском). Если данных много -- упомяни Pattern 4 (pagination). Покажи, что ты думаешь о UX: "пользователь не должен видеть пустой экран, сначала loading, потом данные или ошибка с retry".

❓ Вопросы на интервью (будь готов)

Эти вопросы задают на 90% React Native интервью. Для каждого важно не просто знать ответ, но понимать, ЧТО интервьюер проверяет. Ниже -- модельные ответы senior-уровня, с акцентом на глубину понимания.

  • Как работает Virtual DOM в React?
    Diff алгоритм, reconciliation, batching updates

    Модельный ответ: "React поддерживает виртуальное дерево компонентов в памяти (Virtual DOM). При изменении state React создаёт новое виртуальное дерево и сравнивает с предыдущим (reconciliation). Diff-алгоритм работает за O(n) благодаря двум допущениям: элементы разного типа дают разные деревья, и key позволяет идентифицировать элементы в списках. Найденные различия batching'ом применяются к реальному UI. В React Native вместо DOM обновляется нативное дерево views." Интервьюер проверяет: понимаешь ли ты, ПОЧЕМУ React быстрый и зачем нужны keys.

  • Разница между useState и useRef?
    useState триггерит re-render, useRef -- нет

    Модельный ответ: "useState хранит значение И вызывает re-render при изменении -- для данных, которые влияют на UI (текст, флаги, массивы). useRef хранит мутабельное значение (.current) без re-render -- для ссылок на нативные элементы (MapView ref), таймеров (setTimeout ID), предыдущих значений, и WebSocket-соединений. Аналогия в iOS: useState = @State/@Published (меняет UI), useRef = обычное свойство класса (не меняет UI)." Интервьюер проверяет: знаешь ли ты когда НЕ нужен re-render.

  • Когда использовать useMemo vs useCallback?
    useMemo -- для значений, useCallback -- для функций. Оба для предотвращения re-render детей

    Модельный ответ: "useMemo кэширует результат вычисления: useMemo(() => expensiveSort(data), [data]). useCallback кэширует саму функцию: useCallback((id) => navigate(id), [navigate]). Оба нужны для стабильности ссылок: если передать не-мемоизированную функцию в React.memo компонент, он будет re-render'иться каждый раз (потому что функция -- новый объект). Но не злоупотребляй: useMemo/useCallback сами имеют overhead, используй только когда есть measured performance issue или когда передаёшь в React.memo компонент." Интервьюер проверяет: не слепо ли ты оптимизируешь.

  • Как избежать prop drilling?
    Context, Zustand, Redux, или composition pattern

    Модельный ответ: "Prop drilling -- это когда пропсы прокидываются через 3-4 уровня компонентов. Решения по возрастанию сложности: 1) Composition -- передай готовый JSX, а не данные (аналог slot pattern). 2) Context -- для редко меняющихся данных (тема, пользователь). 3) Zustand -- для часто меняющихся данных (стейт формы, данные с сервера). 4) Redux -- для больших приложений с complex state. Для Onward я бы использовал Zustand для ride state и Context для user auth." Интервьюер проверяет: умеешь ли ты выбрать правильный инструмент, а не самый модный.

  • Как оптимизировать FlatList?
    getItemLayout, windowSize, React.memo для items, keyExtractor

    Модельный ответ: "Базовые оптимизации: React.memo для renderItem компонента, keyExtractor с уникальным ID (не index!), getItemLayout если высота фиксирована (FlatList не нужно рендерить для измерения). Продвинутые: windowSize (по умолчанию 21, можно уменьшить до 5 для экономии памяти), maxToRenderPerBatch (сколько элементов рендерить за один batch), removeClippedSubviews (отсоединяет невидимые view). Для больших списков (1000+): используй FlashList от Shopify -- drop-in замена FlatList, в 5-10x быстрее." Интервьюер проверяет: знаешь ли ты реальные production-оптимизации.

  • Как тестировать React Native компоненты?
    Jest + React Native Testing Library, test user behavior not implementation

📋 Swift → React Native Cheatsheet

Эта таблица -- твой quick reference. Распечатай или держи открытой во время практики. Главное -- не точный синтаксис, а ментальная модель: каждый концепт в Swift имеет аналог в React Native, просто выражен иначе. Ниже таблица с краткими пояснениями самых важных различий.

| Swift/iOS | React Native | Notes | |--------------------------|---------------------------|--------------------------| | @State | useState | Local state | | @Binding | props + callback | Parent controls state | | @EnvironmentObject | useContext | Global state | | ObservableObject | Zustand store | Shared state | | Combine | React Query / RxJS | Reactive data | | onAppear | useEffect([], []) | Mount lifecycle | | onDisappear | useEffect cleanup | Unmount lifecycle | | UITableView | FlatList | Virtualized list | | UINavigationController | Stack Navigator | Push/pop navigation | | UITabBarController | Tab Navigator | Tab bar | | CLLocationManager | expo-location | Location services | | MKMapView | react-native-maps | Maps | | XCTest | Jest + RNTL | Testing | | Codable | TypeScript interfaces | Data modeling | | enum with cases | discriminated union | State machines |

@State → useState: оба хранят локальное состояние компонента и вызывают обновление UI. Разница: @State -- свойство struct, useState -- вызов функции, возвращающий [value, setter]. В Swift: @State var count = 0; count = 1. В RN: const [count, setCount] = useState(0); setCount(1). Нельзя мутировать напрямую -- только через setter.

@Binding → props + callback: самое неочевидное для iOS-разработчика. В SwiftUI @Binding даёт child-компоненту двунаправленный доступ к state родителя. В React нет Binding -- вместо этого parent передаёт значение как prop и callback для изменения: <Input value={name} onChange={setName} />. Это "controlled component" паттерн. Codable → TypeScript interfaces: в Swift Codable обеспечивает runtime encoding/decoding с проверкой типов. TypeScript interfaces -- это только compile-time проверка, в runtime JSON.parse не валидирует типы! Для runtime валидации используй Zod.

onAppear/onDisappear → useEffect: useEffect(() => { /* onAppear */ return () => { /* onDisappear */ }; }, []). Пустой dependency array [] означает "выполни один раз при mount". Cleanup function -- аналог onDisappear. Если dependency array содержит переменные [rideId] -- эффект перезапускается при их изменении. enum → discriminated union: Swift enum MyState { case loading; case loaded(Data); case error(Error) } становится type MyState = { type: 'loading' } | { type: 'loaded'; data: Data } | { type: 'error'; error: Error }. Pattern matching через switch(state.type).

Самая частая ошибка при переходе с iOS: пытаться создать class-based ViewModel с delegate/protocol pattern. В React всё -- функции. "ViewModel" -- это custom hook (useRideData). "Delegate" -- это callback prop. "Protocol" -- это TypeScript interface. "Singleton" -- это Zustand store или Context. Думай функциями, не классами.
Три вещи, которые iOS-разработчики чаще всего забывают в React Native: 1) cleanup в useEffect (аналог deinit/removeObserver -- без него memory leaks), 2) dependency array (забыл переменную -- stale closure, лишняя переменная -- бесконечный loop), 3) immutability (нельзя array.push -- нужно [...array, newItem], нельзя obj.key = value -- нужно {...obj, key: value}). Запомни эти три и 80% ошибок новичка исчезнут.

Шаг 2: Загрузка данных (useTeams)

// hooks/useTeams.ts export function useTeams(): UseTeamsResult { const [teams, setTeams] = useState<Team[]>([]); // @State var teams: [Team] = [] const [loading, setLoading] = useState(true); // @State var loading = true const [error, setError] = useState<string | null>(null); const load = useCallback(async () => { setLoading(true); setError(null); try { const data = await fetchAllTeams(); // URLSession.shared.data(from:) setTeams(data); } catch (e) { setError(e instanceof Error ? e.message : "Failed to load teams"); } finally { setLoading(false); } }, []); useEffect(() => { load(); }, [load]); // .onAppear { load() } return { teams, loading, error, retry: load }; }
Паттерн useCallback + useEffect -- стандартный способ загрузки данных в React. Хук useCallback мемоизирует load(), а useEffect вызывает его при mount (пустой массив зависимостей). Retry -- просто повторный вызов load().

Шаг 3: API Layer (fetch + parallel)

259 команд на 26 страницах. Стратегия: загрузить первую страницу, узнать total_pages, параллельно загрузить остальные.

// utils/api.ts export async function fetchAllTeams(): Promise<Team[]> { const firstPage = await fetchTeamsPage(1); const { total_pages, data } = firstPage; if (total_pages <= 1) return data; // Параллельная загрузка всех страниц const remaining = Array.from( { length: total_pages - 1 }, (_, i) => fetchTeamsPage(i + 2) ); const pages = await Promise.all(remaining); // DispatchGroup / TaskGroup const allTeams = [...data, ...pages.flatMap(p => p.data)]; return allTeams.sort( (a, b) => b.estimated_value_numeric - a.estimated_value_numeric ); }

Шаг 6: Фильтрация (useFilters + useMemo)

Клиентская фильтрация с debounce поиска. useMemo пересчитывает только когда меняются зависимости -- как @State + computed property.

// hooks/useFilters.ts export function useFilters(teams: Team[]): UseFiltersResult { const [filters, setFilters] = useState<FilterState>(DEFAULT_FILTERS); const [searchText, setSearchTextRaw] = useState(""); const debounceRef = useRef<ReturnType<typeof setTimeout>>(); // Debounced search (300ms) const setSearchText = useCallback((text: string) => { setSearchTextRaw(text); // мгновенно обновить TextInput if (debounceRef.current) clearTimeout(debounceRef.current); debounceRef.current = setTimeout(() => { setFilters(prev => ({ ...prev, searchQuery: text })); }, 300); // отложить фильтрацию }, []); // Мемоизированная фильтрация -- пересчёт ТОЛЬКО при изменении teams или filters const filteredTeams = useMemo(() => { const query = filters.searchQuery.toLowerCase(); return teams.filter(team => team.name.toLowerCase().includes(query) && team.estimated_value_numeric >= filters.minValue && team.number_of_league_titles_won >= filters.minTitles ); }, [teams, filters]); return { filteredTeams, filters, searchText, ... }; }

Шаг 13: Trade-offs (что спросят на review)

Assessment оценивает не только код, но и мышление. Будь готов объяснить:

"Почему fetch all upfront, а не pagination?"
259 команд = ~100KB. API не поддерживает серверную фильтрацию. При клиентской фильтрации нужны все данные. Pagination добавила бы сложность без выгоды. При 10K+ элементов -- да, нужен серверный фильтр.
"Почему chips, а не slider для value?"
Диапазон $3M-$2.5B нелинейный. На линейном слайдере 90% значений заняли бы 2% полоски. Chips интуитивнее и accessible (44pt targets, VoiceOver friendly).
"Почему нет Redux/Zustand?"
Один экран, 3 фильтра, один массив. useState + useMemo достаточно. Добавлять state management library -- over-engineering. Показывает, что ты знаешь, когда НЕ использовать инструмент.
"Почему distance on-demand?"
Geocoding 259 стадионов при mount: ~4 минуты при 1 req/sec (Nominatim policy). Плохой UX. Per-button-press + cache -- правильный баланс.

Section 2: Code Walkthrough (5 min)

Answers

Q: "Walk me through your solution — what are you most proud of?"

Most proud of the data loading strategy. The API has 259 teams across 26 pages. I fetch page 1, get total_pages, then Promise.all the remaining 25 pages in parallel. Loads in ~1-2s and enables instant client-side filtering with zero additional network calls. For a dataset this size, it's the right tradeoff. The useTeams hook is the single source of truth — fetch, filter, sort, error handling — like a ViewModel in iOS.

Q: "What would you change with more time?"

Three things: (1) Team detail screen with expo-router dynamic routes, (2) Unit tests for filtering logic and Haversine formula, (3) Skeleton loading instead of spinner for perceived performance.

Q: "Trade-offs you'd flag upfront?"

keyExtractor uses item.name which isn't unique — teams can appear in multiple leagues. Would use item.id or composite key. Also, all data in memory — fine for 259, won't scale to 10K. And filters apply immediately on chip tap — for larger data, a modal with deferred "Apply" would reduce re-renders.

Section 2b: Architecture Questions

Answers Architecture

Q: "How did you structure your state management?"

Pure React primitives — useState + useMemo + useCallback. The useTeams hook owns all state: raw teams array, loading/error, and filters. filteredTeams is a useMemo that recomputes only when teams or filters change. No Redux or Zustand — with one screen and simple data flow, that'd be over-engineering. If this grew to multiple screens sharing state, I'd introduce Zustand.

Q: "Why this component breakdown?"

Single responsibility each: SearchBar (controlled input), FilterControls (collapsible chips, owns expand state), TeamCard (wrapped in React.memo, pure display), DistanceButton (3-state UI: idle → loading → result/error via useStadiumDistance). Hooks separated too: useDebounce is generic/reusable, useLocation handles permissions, useStadiumDistance combines geocoding + Haversine.

Q: "How does filtering work — client-side or API-driven?"

100% client-side. After loading all 259 teams, useMemo chains: name substring match (lowercased) → minimum value threshold → minimum titles threshold → sort descending by valuation. Debounced search (300ms) prevents useMemo recalculating on every keystroke.

Q: "How would this hold up with 10,000 teams?"

It wouldn't — three changes: (1) Server-side filtering — pass search/filter as query params, (2) Cursor-based pagination — FlatList onEndReached fetches next page, (3) Debounced API calls instead of in-memory filter. The migration is clean — useTeams already abstracts the data source, so swapping from in-memory to API-driven is isolated to that one hook.

Backend/API + Edge Cases + Outro

Answers Architecture

Q: "GraphQL query for teams with filtering?"

Connection-style: filter { search, minValue }, first/after for cursor pagination. Return edges → node { id, name, league, estimatedValue, stadiumName } + cursor, plus pageInfo { hasNextPage, endCursor }.

Q: "Offset or cursor-based pagination?"

Cursor-based. Offset breaks on insert/delete between pages — duplicates or skips. Cursor points to specific item, stable for infinite scroll. Maps to onEndReached → fetch with after: lastCursor.

Q: "Auth tokens in Apollo/fetch?"

Apollo link chain: authLink reads token from expo-secure-store, sets Authorization: Bearer. errorLink catches 401s, refreshes token, retries transparently.

Q: "What if geocoding API fails?"

Already handled: 3-state DistanceButton (idle/loading/result-or-error). Nominatim timeout → "Unavailable". mountedRef guard. Module-level Map cache for retaps.
