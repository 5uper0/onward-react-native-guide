🗺️ Navigation в React Native: Overview

Самое важное отличие от iOS: в React Native нет встроенной навигации. В UIKit/SwiftUI навигация — часть фреймворка (UINavigationController, NavigationStack). В RN навигация — это сторонняя библиотека, которая эмулирует нативное поведение через JavaScript. Под капотом React Navigation использует нативные экраны (react-native-screens), но управление стеком происходит в JS. Это значит, что переходы между экранами — это не настоящий push/pop ViewController, а управление деревом React-компонентов с нативными анимациями.

Это фундаментальное архитектурное отличие. В iOS ты привык, что UINavigationController владеет стеком ViewController-ов, каждый со своим жизненным циклом (viewDidLoad, viewWillAppear). В RN "экран" — это просто React-компонент, который монтируется/размонтируется. Вместо viewWillAppear ты используешь хук useFocusEffect, вместо viewDidLoad — useEffect при монтировании.

iOS: React Native: ──────────────────────────────────────────── UINavigationController → Stack Navigator UITabBarController → Tab Navigator UISplitViewController → Drawer Navigator present(vc, animated:) → Stack.Screen (modal) push/pop → navigation.navigate/goBack Coordinator pattern → Expo Router (file-based)

В React Native есть два подхода:

  • React Navigation — классический, императивный. Ты описываешь навигационное дерево в коде (как Coordinator pattern). Гибко, но много boilerplate. Используется в большинстве production-приложений.
  • Expo Router — file-based, как Next.js (рекомендуется). Файловая структура определяет routes автоматически. Проще для новых проектов, встроенный deep linking, типизированные маршруты. Основан на React Navigation, поэтому все его возможности доступны.

Выбор между ними: если начинаешь новый проект на Expo — используй Expo Router. Если интегрируешь RN в существующее нативное приложение (brownfield) — React Navigation напрямую. Expo Router — это обёртка над React Navigation, поэтому переход от одного к другому не требует переписывания всего.

Expo Router = SwiftUI NavigationStack + файловая структура. Создал файл — получил route. В отличие от Coordinator pattern, где ты вручную описываешь каждый переход, здесь маршруты генерируются автоматически из структуры папок. Это устраняет целый класс багов, связанных с рассинхронизацией маршрутов и кода.
В iOS при push/pop ViewController-ы сохраняют состояние в стеке. В RN экраны по умолчанию тоже остаются в памяти при push, но при navigate к уже существующему экрану — он может быть переиспользован, а не создан заново. Это поведение отличается от iOS, где каждый push создаёт новый экземпляр.

📁 Expo Router: Структура файлов = Routes

File-based routing — это конвенция из мира веб-разработки (Next.js, Nuxt, SvelteKit), перенесённая в мобильную разработку. Идея проста: структура папок и файлов в директории app/ автоматически определяет навигационные маршруты приложения. Не нужен Coordinator, не нужен Router — файловая система и есть твой роутер.

Для iOS-разработчика это радикальное упрощение. Вспомни, сколько кода уходит на Coordinator pattern: протоколы, фабрики, методы навигации, связывание ViewController-ов. В Expo Router всего этого нет. Создал файл teams.tsx — маршрут /teams готов. Создал папку team/ с файлом [id].tsx — динамический маршрут /team/:id готов. Вся навигационная логика выводится автоматически.

🍎 iOS Coordinator
// AppCoordinator.swift func start() { let home = HomeVC() let teams = TeamsVC() let profile = ProfileVC() tabBar.viewControllers = [ UINavigationController(root: home), UINavigationController(root: teams), UINavigationController(root: profile) ] } // Переход на детали команды func showTeamDetail(_ id: String) { let detail = TeamDetailVC(id: id) navigationController.push(detail) }
⚛️ Expo Router (file-based)
app/ ├── _layout.tsx // Root layout ├── (tabs)/ // Tab group │ ├── _layout.tsx // Tab bar config │ ├── index.tsx // "/" (Home tab) │ ├── teams.tsx // "/teams" tab │ └── profile.tsx // "/profile" tab ├── team/ │ └── [id].tsx // "/team/123" (dynamic) └── modal.tsx // "/modal" (modal screen) // Навигация: router.push('/team/123'); router.back();

Обрати внимание на ключевые отличия в поведении: в iOS Coordinator явно создаёт экземпляры ViewController-ов и управляет их жизненным циклом. В Expo Router экран создаётся автоматически при навигации на соответствующий URL. Параметры передаются через URL (как в вебе), а не через инициализатор. Это делает deep linking бесплатным — любой маршрут доступен по URL, что в iOS требует отдельной настройки Universal Links.

Имя файла = URL path. [id].tsx — dynamic segment (как :id в web). Круглые скобки (tabs) создают "группу" — она не появляется в URL. Файл index.tsx — это корневой маршрут группы (как index.html в вебе). Если ты привык к Coordinator pattern, представь, что файловая система — это и есть твой Coordinator, только декларативный.
Распространённая ошибка iOS-разработчиков: пытаться передать сложные объекты через навигационные параметры (как через segue prepare). В Expo Router параметры — это строки в URL. Передавай только ID, а данные загружай на целевом экране через React Query или Zustand. Это отличается от iOS, где ты часто передаёшь целый model object при push.

📦 _layout.tsx — Root Layout

Файлы _layout.tsx — это ключевая концепция Expo Router. Каждый layout оборачивает все дочерние маршруты в своей директории. Это как UINavigationController, который "владеет" всеми экранами внутри себя. Layout определяет тип навигатора (Stack, Tabs, Drawer) и общие настройки для всех дочерних экранов.

Layouts вкладываются друг в друга: root layout оборачивает всё приложение, layout в (tabs)/ оборачивает только вкладки. Это создаёт иерархию навигаторов — точно как в iOS, где UITabBarController содержит UINavigationController-ы. Разница в том, что в iOS ты строишь эту иерархию императивно в коде, а здесь она выводится из структуры папок.

// app/_layout.tsx import { Stack } from 'expo-router'; export default function RootLayout() { return ( <Stack screenOptions={{ headerStyle: { backgroundColor: '#1a1a2e' }, headerTintColor: '#fff', headerTitleStyle: { fontWeight: 'bold' }, }} > <Stack.Screen name="(tabs)" options={{ headerShown: false }} /> <Stack.Screen name="team/[id]" options={{ title: 'Team Details' }} /> <Stack.Screen name="modal" options={{ presentation: 'modal', title: 'Settings' }} /> </Stack> ); }

В этом коде screenOptions — это глобальные настройки, аналог UINavigationBar.appearance(). Они применяются ко всем экранам в этом Stack. Каждый Stack.Screen регистрирует маршрут и его опции. Обрати внимание на presentation: 'modal' — это аналог present(vc, animated: true) в iOS. Stack автоматически добавляет кнопку "назад" и жест свайпа, как UINavigationController.

Stack — это UINavigationController. screenOptions — глобальные настройки navigation bar. Важное отличие: в iOS ты можешь настроить navigation bar из самого ViewController (в viewWillAppear). В Expo Router ты тоже можешь переопределить options прямо из экрана через компонент <Stack.Screen options={{...}} />, размещённый внутри компонента экрана.
Root Layout — это точка входа приложения. Здесь размещают глобальные providers (QueryClient, Context, темы). Если ты забудешь обернуть Stack в необходимый Provider, все дочерние экраны потеряют доступ к контексту. Это аналог того, как в iOS SceneDelegate конфигурирует root window и root ViewController.

📑 Tab Bar Layout

Tab Bar в Expo Router — это layout-файл внутри группы (tabs). Круглые скобки в имени папки создают "группу маршрутов" — они не влияют на URL. То есть файл app/(tabs)/teams.tsx доступен по адресу /teams, а не /(tabs)/teams. Это чисто организационная структура для группировки экранов под одним навигатором.

Каждый файл внутри (tabs)/ автоматически становится вкладкой. Порядок вкладок определяется порядком Tabs.Screen в layout. Если файл существует в папке, но не описан в layout — он всё равно будет доступен как маршрут, но появится как дополнительная вкладка (что обычно нежелательно).

🍎 UITabBarController
let tabBar = UITabBarController() let home = HomeVC() home.tabBarItem = UITabBarItem( title: "Home", image: UIImage(systemName: "house"), tag: 0 ) tabBar.viewControllers = [home, teams, profile] tabBar.tabBar.tintColor = .blue
⚛️ Expo Router Tabs
// app/(tabs)/_layout.tsx import { Tabs } from 'expo-router'; import { Ionicons } from '@expo/vector-icons'; export default function TabLayout() { return ( <Tabs screenOptions={{ tabBarActiveTintColor: '#3b82f6', }} > <Tabs.Screen name="index" options={{ title: 'Home', tabBarIcon: ({ color }) => ( <Ionicons name="home" color={color} /> ), }} /> <Tabs.Screen name="teams" ... /> <Tabs.Screen name="profile" ... /> </Tabs> ); }

Ключевые отличия от UITabBarController: в iOS каждая вкладка обычно содержит свой UINavigationController. В Expo Router ты можешь вложить Stack внутрь каждой вкладки, но часто этого не нужно — навигация из вкладки просто пушит экран поверх всего tab bar (через root Stack). Иконки — это React-компоненты (не UIImage), что даёт больше гибкости: можно использовать анимированные иконки, бейджи, кастомные SVG.

В iOS для кастомного tab bar ты создаёшь подкласс UITabBarController или подменяешь UITabBar. В Expo Router можно полностью заменить tab bar через tabBar prop: <Tabs tabBar={(props) => <MyCustomTabBar {...props} />}>. Это значительно проще, чем наследование от UITabBar.
Частая ошибка: создать файл в (tabs)/ и забыть описать его в layout. Expo Router создаст для него вкладку с дефолтными настройками. Чтобы скрыть маршрут из tab bar, используй options={{ href: null }} — экран останется доступным программно, но не покажется как вкладка.

🔗 Navigation Hooks

В iOS для навигации ты обращаешься к navigationController — объекту, который управляет стеком. В React Native навигация доступна через хуки — функции, которые дают доступ к навигатору из любого компонента. Это важное отличие: не нужно передавать ссылку на навигатор через dependency injection или делегаты — любой компонент внутри навигатора может навигировать.

Expo Router предоставляет три ключевых хука: useRouter() для навигации, useLocalSearchParams() для чтения параметров текущего маршрута, и usePathname() для определения текущего URL. Эти хуки типизированы — TypeScript подскажет, какие параметры доступны на каждом маршруте.

import { useRouter, useLocalSearchParams, usePathname } from 'expo-router'; function TeamListScreen() { const router = useRouter(); const handleTeamPress = (teamId: string) => { // Push new screen (как navigationController.push) router.push(`/team/${teamId}`); // Или с параметрами router.push({ pathname: '/team/[id]', params: { id: teamId, tab: 'players' } }); }; const handleSettings = () => { // Modal (как present(vc, animated:)) router.push('/modal'); }; const handleBack = () => { router.back(); // Как navigationController.popViewController }; } // В team/[id].tsx function TeamDetailScreen() { const { id } = useLocalSearchParams<{ id: string }>(); // id = "123" если URL был /team/123 }

Разница между router.push() и router.replace(): push добавляет экран в стек (как pushViewController), replace заменяет текущий (как setViewControllers с последним элементом). Есть также router.navigate() — умная навигация, которая не создаёт дубликат, если экран уже в стеке. Это отличается от iOS, где каждый push всегда создаёт новый ViewController.

useLocalSearchParams — как получить параметры из segue в iOS. Но есть важное отличие: параметры всегда строковые (это URL). Если передал число, получишь строку. Привычка из Swift, где параметр может быть любого типа, здесь не работает. Используй parseInt(id) или валидацию через Zod, если нужны числа.
Есть тонкая разница между useLocalSearchParams и useGlobalSearchParams: local возвращает параметры, актуальные для данного экрана (даже если URL изменился), global — всегда текущие параметры URL. На практике почти всегда используй local, иначе компонент будет перерисовываться при навигации на другие экраны.
Deep linking работает автоматически: поскольку навигация основана на URL, любой маршрут вроде /team/123 доступен из push-уведомлений или внешних ссылок без дополнительной настройки. В iOS для этого нужно настраивать Universal Links, обрабатывать URL в SceneDelegate, и маршрутизировать вручную.

🗄️ State Management: Когда useState мало

Управление состоянием — это, пожалуй, самое большое концептуальное отличие React от iOS. В UIKit ты привык, что каждый ViewController владеет своим состоянием, и передача данных между экранами происходит через делегаты, замыкания, NotificationCenter или shared singleton. В SwiftUI появились @State, @Binding, @EnvironmentObject — но даже там нет такого чёткого разделения типов состояния, как в React-экосистеме.

В React есть фундаментальное правило: данные текут только вниз (unidirectional data flow). Родительский компонент передаёт данные детям через props. Ребёнок не может напрямую изменить состояние родителя — только вызвать callback, переданный через props. Это отличается от iOS, где delegate pattern позволяет ребёнку вызвать метод родителя напрямую, а @Binding создаёт двустороннюю связь. В React двусторонней привязки нет — есть только передача данных вниз и callback-ов вверх.

Типы состояния: 1. LOCAL STATE (useState) Скоуп: один компонент Пример: isExpanded, inputValue iOS: @State в SwiftUI, свойства ViewController 2. LIFTED STATE (props drilling) Скоуп: несколько компонентов в дереве Пример: filter в родителе, используется в детях iOS: @Binding, delegate pattern 3. GLOBAL STATE (Context/Redux/Zustand) Скоуп: всё приложение Пример: user, theme, favorites iOS: @EnvironmentObject, Singleton, AppDelegate 4. SERVER STATE (React Query/SWR) Скоуп: кэшированные данные с бэкенда Пример: teams, players, matches iOS: URLSession + ViewModel + кэш (нет стандарта)

Ключевой принцип: "поднимай состояние" (lift state up). Если два соседних компонента используют одни и те же данные, перенеси состояние в их общего родителя. Если состояние нужно далеко по дереву — используй Context или Zustand. Если данные приходят с сервера — используй React Query. Неправильный выбор типа состояния — главная причина проблем с производительностью и запутанного кода в React-приложениях.

В iOS это: @State = useState, @Binding = callback props, @EnvironmentObject = Context, ViewModel + Combine = React Query. Главное отличие: в React чёткое разделение client state и server state. В iOS ViewModel обычно хранит и то, и другое — это источник сложности.
В iOS ты привык к reference-type моделям (class) и мутации по месту. В React всё состояние immutable — ты не мутируешь массив, а создаёшь новый. setItems([...items, newItem]) вместо items.append(newItem). Это нужно для того, чтобы React мог определить, что состояние изменилось, и перерисовать компонент.

🌐 React Context = @EnvironmentObject

React Context — это механизм "проброса" данных через дерево компонентов без передачи через props на каждом уровне. Концептуально это аналог @EnvironmentObject в SwiftUI: ты размещаешь Provider наверху дерева, и любой дочерний компонент может "подписаться" на эти данные через useContext.

Но есть критическое отличие от SwiftUI: в SwiftUI, @EnvironmentObject работает с ObservableObject, и SwiftUI умно перерисовывает только те View, которые используют изменённые @Published свойства. В React Context нет такой гранулярности: когда значение Context изменяется, ВСЕ компоненты, использующие useContext для этого контекста, перерисовываются. Даже если они используют только часть данных, которая не изменилась.

🍎 SwiftUI Environment
// Создание class UserStore: ObservableObject { @Published var user: User? @Published var favorites: Set<String> = [] func toggleFavorite(_ id: String) { if favorites.contains(id) { favorites.remove(id) } else { favorites.insert(id) } } } // Провайдер @main struct MyApp: App { @StateObject var userStore = UserStore() var body: some Scene { WindowGroup { ContentView() .environmentObject(userStore) } } } // Использование struct TeamRow: View { @EnvironmentObject var store: UserStore var body: some View { ... } }
⚛️ React Context
// Создание контекста interface FavoritesContextType { favorites: Set<string>; toggleFavorite: (id: string) => void; } const FavoritesContext = createContext<FavoritesContextType>(null!); // Провайдер function FavoritesProvider({ children }) { const [favorites, setFavorites] = useState(new Set<string>()); const toggleFavorite = useCallback((id: string) => { setFavorites(prev => { const next = new Set(prev); if (next.has(id)) next.delete(id); else next.add(id); return next; }); }, []); return ( <FavoritesContext.Provider value={{ favorites, toggleFavorite }}> {children} </FavoritesContext.Provider> ); } // Использование function TeamRow() { const { favorites, toggleFavorite } = useContext(FavoritesContext); }

Обрати внимание на useCallback в провайдере. Без него функция toggleFavorite пересоздавалась бы при каждом рендере провайдера, и это считалось бы "новым значением" контекста, провоцируя перерисовку всех потребителей. useCallback мемоизирует функцию — она создаётся один раз и не меняется. В SwiftUI такой проблемы нет, потому что методы ObservableObject не пересоздаются.

Context подходит для данных, которые меняются редко: тема, авторизация, настройки. НЕ используй Context для часто обновляемых данных (позиция скролла, ввод текста, анимации) — каждое обновление перерисует всё поддерево. Для таких случаев используй Zustand с селекторами или локальный state. Это главная ловушка Context, в которую попадают разработчики с опытом SwiftUI, где @EnvironmentObject более оптимизирован.

🪝 Custom Hook для Context

Обёртка useContext в кастомный хук — это не просто удобство, а best practice в React-разработке. Этот паттерн решает три проблемы: (1) скрывает детали реализации (потребитель не знает, что под капотом Context), (2) добавляет проверку на наличие Provider, (3) обеспечивает единую точку рефакторинга — если захочешь заменить Context на Zustand, изменишь только этот хук.

В iOS эквивалент — это протокол, за которым скрывается конкретная реализация. Компонент вызывает useFavorites() и получает интерфейс, не зная, хранятся ли данные в Context, Zustand, или AsyncStorage. Это инверсия зависимости (Dependency Inversion) на уровне React-хуков.

// contexts/FavoritesContext.tsx import { createContext, useContext, useState, useCallback, ReactNode } from 'react'; interface FavoritesContextType { favorites: Set<string>; isFavorite: (id: string) => boolean; toggleFavorite: (id: string) => void; } const FavoritesContext = createContext<FavoritesContextType | null>(null); export function FavoritesProvider({ children }: { children: ReactNode }) { const [favorites, setFavorites] = useState<Set<string>>(new Set()); const isFavorite = useCallback((id: string) => favorites.has(id), [favorites]); const toggleFavorite = useCallback((id: string) => { setFavorites(prev => { const next = new Set(prev); next.has(id) ? next.delete(id) : next.add(id); return next; }); }, []); return ( <FavoritesContext.Provider value={{ favorites, isFavorite, toggleFavorite }}> {children} </FavoritesContext.Provider> ); } // Custom hook для удобства export function useFavorites() { const context = useContext(FavoritesContext); if (!context) { throw new Error('useFavorites must be used within FavoritesProvider'); } return context; }

Проверка if (!context) — это эквивалент fatalError() в Swift: если компонент пытается использовать useFavorites() вне FavoritesProvider, приложение падает с понятным сообщением. Без этой проверки ты получишь загадочный Cannot read property 'favorites' of null — и будешь долго искать причину. В SwiftUI @EnvironmentObject делает то же самое — крэшится, если объект не предоставлен.

Всегда проверяй null в custom hook! Иначе получишь cryptic errors. Типичная ошибка: используешь хук в компоненте, который рендерится ДО монтирования Provider. Например, в _layout.tsx до обёртки в Provider. Порядок имеет значение — Provider должен быть выше в дереве, чем любой потребитель.
Обрати внимание на паттерн: Context + тип + Provider + custom hook — это всегда один файл. Этот файл экспортирует только FavoritesProvider и useFavorites. Сам FavoritesContext не экспортируется — он деталь реализации. Это как private/public в Swift: потребителю не нужен доступ к сырому Context.

Подключение Provider

Provider tree ("дерево провайдеров") — это паттерн, который может удивить iOS-разработчика. В SwiftUI ты цепляешь .environmentObject() к View, и все дочерние View получают доступ. В React аналогичная концепция, но провайдеры вкладываются друг в друга как матрёшки. Root layout — это место, где ты выстраиваешь эту "матрёшку".

Порядок вложенности важен: внутренний провайдер может использовать данные из внешнего, но не наоборот. Например, если AuthProvider зависит от QueryClient (для загрузки профиля пользователя), то QueryClientProvider должен быть снаружи. Типичный порядок: ErrorBoundary > QueryClient > Auth > Theme > Navigation.

// app/_layout.tsx import { Stack } from 'expo-router'; import { FavoritesProvider } from '@/contexts/FavoritesContext'; import { QueryClient, QueryClientProvider } from '@tanstack/react-query'; const queryClient = new QueryClient(); export default function RootLayout() { return ( <QueryClientProvider client={queryClient}> <FavoritesProvider> <Stack> <Stack.Screen name="(tabs)" options={{ headerShown: false }} /> <Stack.Screen name="team/[id]" /> </Stack> </FavoritesProvider> </QueryClientProvider> ); } // В любом компоненте внутри: function TeamRow({ team }) { const { isFavorite, toggleFavorite } = useFavorites(); return ( <View style={styles.row}> <Text>{team.name}</Text> <TouchableOpacity onPress={() => toggleFavorite(team.id)}> <Ionicons name={isFavorite(team.id) ? "heart" : "heart-outline"} color="red" /> </TouchableOpacity> </View> ); }

В этом примере QueryClientProvider оборачивает всё — значит React Query доступен везде, включая внутри FavoritesProvider. FavoritesProvider оборачивает Stack — значит все экраны навигации имеют доступ к favorites. Если бы FavoritesProvider был снаружи QueryClientProvider, то favorites-логика не могла бы использовать React Query для синхронизации с сервером.

В iOS ты часто используешь AppDelegate или SceneDelegate для инициализации глобальных сервисов. В React root layout выполняет ту же роль. Если у тебя много провайдеров (5+), создай отдельный компонент Providers, который их все оборачивает — это снижает "лесенку" вложенности и делает root layout чище.
Распространённая ошибка: создание new QueryClient() внутри компонента (в теле функции). При каждом рендере будет создаваться новый экземпляр, что сбрасывает весь кэш. Объявляй QueryClient вне компонента (на уровне модуля) или используй useState / useRef для однократного создания.

🐻 Zustand — простой global state

Zustand (немецкое слово "состояние") — это минималистичная библиотека для глобального состояния. Если Context — это встроенный инструмент React с известными ограничениями (перерисовка всех потребителей), то Zustand решает эти проблемы и добавляет удобства. В мире iOS ближайший аналог — это синглтон с @Published свойствами в Combine, но без boilerplate.

Почему Zustand, а не Redux или MobX? Redux требует actions, reducers, middleware — огромный boilerplate для простых задач. MobX использует observable/proxy-паттерн, похожий на SwiftUI, но добавляет "магию", которую трудно отлаживать. Zustand — золотая середина: минимальный API (create — и всё), нет Provider-обёрток, нативная поддержка TypeScript, встроенная персистенция. Для большинства мобильных приложений это оптимальный выбор.

Context хорош, но Zustand ещё проще и быстрее.

// stores/useFavoritesStore.ts import { create } from 'zustand'; import { persist, createJSONStorage } from 'zustand/middleware'; import AsyncStorage from '@react-native-async-storage/async-storage'; interface FavoritesStore { favorites: string[]; toggleFavorite: (id: string) => void; isFavorite: (id: string) => boolean; } export const useFavoritesStore = create<FavoritesStore>()( persist( (set, get) => ({ favorites: [], toggleFavorite: (id) => set((state) => ({ favorites: state.favorites.includes(id) ? state.favorites.filter(f => f !== id) : [...state.favorites, id] })), isFavorite: (id) => get().favorites.includes(id), }), { name: 'favorites-storage', storage: createJSONStorage(() => AsyncStorage), } ) ); // Использование (без Provider!) function TeamRow({ team }) { const { isFavorite, toggleFavorite } = useFavoritesStore(); // ... }

Ключевое преимущество Zustand перед Context — селекторы. Ты можешь подписаться только на часть store: const favorites = useFavoritesStore(state => state.favorites). Компонент перерисуется только когда favorites изменится, а не при любом изменении store. В iOS это как @Published в Combine — подписка на конкретное свойство, а не весь объект. В Context такой гранулярности нет.

Zustand = useState + persist + devtools. Без boilerplate. Без Provider wrapper. Ещё одно преимущество: store можно использовать вне React-компонентов (в утилитах, middleware). Вызови useFavoritesStore.getState() из обычной функции — это невозможно с Context, который работает только внутри компонентов.
Встроенная персистенция через persist middleware — это аналог UserDefaults в iOS. Данные автоматически сериализуются в AsyncStorage и восстанавливаются при запуске. Не нужно вручную писать save/load логику. Для чувствительных данных (токены) используй expo-secure-store вместо AsyncStorage — аналог Keychain в iOS.

💾 Storage в React Native: Полный гайд

В iOS у тебя UserDefaults, Keychain, CoreData, FileManager — и ты точно знаешь, когда что использовать. В React Native ситуация аналогичная, но с другими именами. Главное правило: выбор storage зависит от типа данных, объёма и требований безопасности.

AsyncStorage — дефолтный выбор для большинства случаев. Асинхронный key-value store, данные хранятся как строки (нужен JSON.stringify/parse для объектов). Ограничение: 6MB на Android. Zustand persist middleware использует его по умолчанию.

MMKV — если нужна скорость. Синхронный (не нужен await!), в 30x быстрее AsyncStorage на чтении. Создан WeChat для миллиардов пользователей. Используй как drop-in замену AsyncStorage в Zustand: persist(store, { storage: createMMKVStorage() }).

Главное отличие от iOS: в React Native нет синхронного key-value store из коробки (AsyncStorage — async). Это значит, что при старте приложения ты не можешь мгновенно прочитать настройки — нужен loading state. MMKV решает эту проблему, но требует нативной зависимости. На интервью: начни с AsyncStorage (встроенный), упомяни MMKV как production-оптимизацию.
На интервью часто спрашивают: "Где хранить auth token?" Правильный ответ: expo-secure-store (аналог Keychain). НИКОГДА не AsyncStorage — данные не зашифрованы, на Android доступны через root. Для offline-first приложения с большим объёмом данных: SQLite или WatermelonDB. Для простых настроек/фильтров: AsyncStorage или MMKV.

🔄 React Query = Server State Management

React Query (TanStack Query) — это, вероятно, самая важная библиотека для React-разработки после самого React. Она решает проблему, которую каждый iOS-разработчик решал десятки раз: загрузка данных с сервера, кэширование, обработка ошибок, повторные попытки, обновление в фоне, отмена запросов. В iOS для этого ты пишешь ViewModel с Combine, URLSession, кэш-слой — React Query делает это в 5 строк.

Ключевая идея: серверные данные (teams, players, matches) — это не "состояние приложения", а "кэш удалённых данных". Они принадлежат серверу, а у тебя только копия. React Query управляет этой копией: знает, когда она "свежая" (staleTime), когда нужно обновить (refetch), когда показать кэшированное значение пока грузится новое (stale-while-revalidate). Этот паттерн устраняет 90% кода, который ты писал в ViewModel.

🍎 Swift ViewModel

Сравни объём кода: Swift ViewModel — 20+ строк ручного управления состоянием (и это ещё без кэширования и дедупликации!). React Query — одна декларация. queryKey: ['teams'] — уникальный ключ кэша. Если два экрана одновременно запросят ['teams'], React Query отправит только один HTTP-запрос и раздаст результат обоим. staleTime — как долго данные считаются "свежими": 5 минут не будет повторного запроса. Это как NSCache + URLSessionConfiguration.requestCachePolicy, но декларативно.

React Query — это как URLSession + NSCache + Combine publisher + error handling + retry logic + request deduplication — всё в одном. В iOS ты бы строил Repository layer, кэш-менеджер, и координировал это через Combine. Здесь это один вызов useQuery. Stale-while-revalidate паттерн означает: показать кэшированные данные мгновенно, одновременно загрузить свежие в фоне — пользователь не видит спиннер при повторном заходе на экран.
Не храни серверные данные в Zustand или Context! Типичная ошибка при переходе с iOS: создать глобальный store для teams/players, загрузить данные при старте, и вручную обновлять. React Query делает это лучше: автоматический refetch при возвращении на экран, при восстановлении сети, по таймеру. Zustand — для client state (favorites, настройки). React Query — для server state (API-данные).

✏️ Mutations в React Query

Если useQuery — это чтение данных (GET), то useMutation — это запись (POST, PUT, DELETE). В iOS ты бы написал метод в ViewModel, который отправляет запрос, обрабатывает ответ, обновляет локальное состояние, показывает спиннер и обрабатывает ошибки. Mutation делает то же самое декларативно: ты описываешь "что мутировать" и "что делать при успехе/ошибке".

Ключевая концепция — cache invalidation. После мутации (например, добавления игрока) кэш становится устаревшим. Вместо того чтобы вручную обновлять локальные данные (как ты бы сделал с @Published var players в iOS), ты вызываешь invalidateQueries — React Query автоматически перезагрузит данные с сервера. Это проще и надёжнее, чем ручная синхронизация.

// Mutation = POST/PUT/DELETE (изменение данных) function TeamDetailScreen({ teamId }) { const queryClient = useQueryClient(); // Получение данных const { data: team } = useQuery({ queryKey: ['team', teamId], queryFn: () => api.getTeam(teamId), }); // Mutation для добавления игрока const addPlayerMutation = useMutation({ mutationFn: (player: NewPlayer) => api.addPlayer(teamId, player), onSuccess: () => { // Инвалидировать кэш — данные устарели queryClient.invalidateQueries({ queryKey: ['team', teamId] }); // React Query автоматически перезагрузит }, onError: (error) => { Alert.alert('Error', error.message); }, }); const handleAddPlayer = (player: NewPlayer) => { addPlayerMutation.mutate(player); }; return ( <View> {addPlayerMutation.isPending && <ActivityIndicator />} <Button title="Add Player" onPress={() => handleAddPlayer({ name: 'New Player' })} disabled={addPlayerMutation.isPending} /> </View> ); }

Для ещё лучшего UX существуют optimistic updates — ты обновляешь UI сразу, не дожидаясь ответа сервера. Если сервер вернул ошибку — откатываешь изменения. React Query поддерживает это через onMutate (обновить кэш оптимистично), onError (откатить), onSettled (синхронизировать). В iOS аналог — Core Data с pending changes, которые потом синхронизируются.

invalidateQueries — говорит React Query "эти данные устарели, перезагрузи". Можно инвалидировать по частичному ключу: invalidateQueries({ queryKey: ['team'] }) инвалидирует ВСЕ team-запросы (список, детали). Это как NSNotification, которая говорит всем подписчикам "данные изменились" — но автоматически, без ручных Observer-ов.
Не вызывай mutate() внутри рендера! Mutation — это side effect, его нужно вызывать в обработчиках событий (onPress, onSubmit). Если нужна автоматическая мутация при изменении данных, используй useEffect с mutate. Также помни: mutate не возвращает Promise. Если нужен await — используй mutateAsync.

🏗️ Architecture: iOS patterns → React

Одна из главных ловушек для iOS-разработчика — попытка перенести MVVM/VIPER/Clean Architecture буквально в React. Это не работает, потому что React построен на другой парадигме. В iOS ты разделяешь View, ViewModel, Model как отдельные классы с чёткими зависимостями. В React основной примитив декомпозиции — хук (hook), а не класс. Хуки — это функции, которые инкапсулируют логику и могут использовать другие хуки. Они заменяют и ViewModel, и Use Case, и Repository — но в виде composable функций, а не объектов.

Почему MVVM не переводится напрямую? В iOS ViewModel — это ObservableObject, который живёт дольше View (через @StateObject). В React "ViewModel" — это custom hook, который вызывается внутри компонента и привязан к его жизненному циклу. Нет отдельного объекта-ViewModel — есть набор хуков, которые компонент вызывает для получения данных и действий. Это более плоская архитектура: нет слоёв абстракции ради абстракции.

iOS Pattern React Equivalent ───────────────────────────────────────────── MVC → Component + hooks MVVM → Component + custom hooks Coordinator → Expo Router Repository → API layer + React Query Use Cases → Custom hooks Dependency Injection → Context / Zustand Singleton Services → Global stores

На практике архитектура React-приложения сводится к трём уровням: (1) Components — чистый UI, рендерит JSX на основе props, (2) Hooks — бизнес-логика, загрузка данных, вычисления, (3) Services/Stores — API-клиент, глобальное состояние, утилиты. Это проще, чем VIPER с 5 слоями, но достаточно для масштабирования. Большинство продовых React Native приложений используют именно эту структуру.

В React нет строгого MVC/MVVM. Есть компоненты + hooks + state management. Вместо архитектурных паттернов с классами — функциональная композиция: маленькие хуки собираются в большие, компоненты используют хуки. Если ты захочешь Clean Architecture — custom hook = Use Case, API-модуль = Repository, Context/Zustand = Dependency Injection container.
Опытные iOS-разработчики часто создают избыточные абстракции в React: отдельный ViewModel-класс, протоколы для каждого сервиса, фабрики. Это overeengineering. В React custom hook УЖЕ является ViewModel. useTeamDetail(id) — это и ViewModel, и Use Case одновременно. Не создавай дополнительные классы-обёртки, если хук справляется.

📁 Рекомендуемая структура проекта

Структура проекта в React Native отличается от iOS-проектов. В Xcode ты привык к группам по слоям: Models/, Views/, ViewModels/, Services/. В React-мире тоже есть такое деление, но с другими именами. Важный принцип — colocation (размещение рядом): файлы, которые используются вместе, хранятся в одной папке. Hook useTeams лежит рядом с типами Team и API-вызовами для teams.

Папка app/ — единственная "магическая" директория: Expo Router сканирует её для построения маршрутов. Всё остальное — обычные папки, организованные по конвенции. Нет файла вроде .xcodeproj, который определяет структуру — ты сам решаешь, как организовать код.

src/ ├── app/ # Expo Router (screens) │ ├── _layout.tsx │ ├── (tabs)/ │ │ ├── _layout.tsx │ │ ├── index.tsx │ │ └── ... │ └── team/[id].tsx │ ├── components/ # Переиспользуемые UI компоненты │ ├── TeamRow.tsx │ ├── LoadingSpinner.tsx │ └── ErrorView.tsx │ ├── hooks/ # Custom hooks (бизнес-логика) │ ├── useTeams.ts │ ├── useTeamDetail.ts │ └── useAuth.ts │ ├── stores/ # Zustand stores (global state) │ ├── useFavoritesStore.ts │ └── useAuthStore.ts │ ├── services/ # API, networking │ ├── api.ts │ └── types.ts │ ├── utils/ # Helpers │ └── formatters.ts │ └── constants/ # Config, theme ├── colors.ts └── config.ts

Разберём каждую папку: app/ содержит только экраны и layouts — минимум логики, максимум UI. components/ — переиспользуемые UI-элементы (как UIView-подклассы). hooks/ — бизнес-логика, привязанная к домену (как ViewModel-ы). stores/ — глобальное состояние (как Singleton-сервисы). services/ — API-клиент и типы (как Networking layer). utils/ — чистые функции без побочных эффектов. constants/ — конфигурация, цвета, размеры (как Assets.xcassets + Constants.swift).

По мере роста проекта можно перейти к feature-based структуре: features/teams/ содержит компоненты, хуки, типы и сервисы для teams. Это как модули в iOS (SPM packages), но без формальных границ. Когда стоит переходить? Когда в hooks/ скопилось 20+ файлов и становится трудно найти нужный. Начинай с плоской структуры и рефактори по мере необходимости.
Не помещай бизнес-логику в app/ (экраны). Экран должен только вызывать хуки и рендерить UI. Если видишь useState + useEffect + вычисления в файле экрана — пора выносить в custom hook. Правило: файл экрана не должен превышать 50-80 строк. Вся сложность — в хуках.

🧹 Clean Architecture: Custom Hook как Use Case

Это центральный паттерн React-архитектуры, и он элегантно решает задачу Clean Architecture без лишних слоёв. Custom hook объединяет все источники данных (сервер, локальное состояние, глобальный store) и предоставляет компоненту единый интерфейс. В Clean Architecture терминах это Use Case / Interactor — он координирует получение данных из разных репозиториев и предоставляет результат Presenter-у (компоненту).

Преимущество перед MVVM: в iOS ViewModel — это класс, который нужно инициализировать, держать ссылку (@StateObject), управлять его жизненным циклом. Custom hook — это просто вызов функции. Его можно вызвать условно, можно вызвать несколько раз, можно передать другому хуку. Хуки composable: useTeamDetail внутри использует useQuery, useState и useFavoritesStore — каждый из них может быть заменён независимо.

// hooks/useTeamDetail.ts // Это как Use Case / Interactor в Clean Architecture export function useTeamDetail(teamId: string) { // Server state (React Query) const { data: team, isLoading, error } = useQuery({ queryKey: ['team', teamId], queryFn: () => api.getTeam(teamId), enabled: !!teamId, // Не запускать если нет ID }); // Local state const [isEditing, setIsEditing] = useState(false); // Global state (Zustand) const { isFavorite, toggleFavorite } = useFavoritesStore(); // Derived state const favorite = useMemo(() => isFavorite(teamId), [teamId, isFavorite]); // Actions const handleFavoriteToggle = useCallback(() => { toggleFavorite(teamId); }, [teamId, toggleFavorite]); // Return ViewModel-like interface return { // State team, isLoading, error, isEditing, favorite, // Actions setIsEditing, toggleFavorite: handleFavoriteToggle, }; } // Использование в компоненте — чистый UI! function TeamDetailScreen() { const { id } = useLocalSearchParams(); const { team, isLoading, favorite, toggleFavorite } = useTeamDetail(id); if (isLoading) return <Loading />; return <TeamDetailView team={team} favorite={favorite} onToggle={toggleFavorite} />; }

Обрати внимание на тестируемость: useTeamDetail можно протестировать отдельно от UI через renderHook. API мокается, Zustand store мокается — и ты проверяешь всю логику без рендеринга компонентов. В iOS для этого ты бы создал протоколы для каждой зависимости ViewModel и инжектил моки. Здесь зависимости (api, store) можно мокнуть через jest.mock на уровне модулей — проще и менее церемониально.

Dependency injection в React происходит через модульную систему (import) и React Context, а не через инициализаторы. Для тестов ты мокаешь модули через jest.mock('./api'). Для runtime DI — передаёшь зависимости через Context. Не нужны фабрики, контейнеры, Swinject — модульная система JS и есть твой DI-контейнер.
Правило хорошего хука: он возвращает объект с данными (state) и действиями (actions), как ViewModel. Компонент деструктурирует результат и рендерит UI. Если хук возвращает больше 7-8 полей — пора разбить на два хука. Композиция хуков — главный инструмент управления сложностью в React.

🛡️ Error Boundaries — ловим crashes

В iOS необработанная ошибка в UI-коде приводит к крэшу всего приложения. В React аналогично: если компонент выбросит исключение при рендере, всё приложение "белеет" (white screen of death). Error Boundary — это компонент-обёртка, который ловит ошибки в дочерних компонентах и показывает fallback UI вместо крэша. Это как do-catch для целого поддерева компонентов.

Важная деталь: Error Boundary обязан быть class-компонентом. Это единственное место в современном React, где нужен class вместо function. Хуков для Error Boundary не существует (пока). Метод getDerivedStateFromError вызывается при ошибке рендера, а componentDidCatch — для логирования (как Crashlytics/Sentry). На практике большинство проектов используют библиотеку react-error-boundary, которая предоставляет готовый компонент с хуками.

// components/ErrorBoundary.tsx import { Component, ReactNode } from 'react'; interface Props { children: ReactNode; fallback?: ReactNode; } interface State { hasError: boolean; error?: Error; } class ErrorBoundary extends Component<Props, State> { state: State = { hasError: false }; static getDerivedStateFromError(error: Error): State { return { hasError: true, error }; } componentDidCatch(error: Error, info: React.ErrorInfo) { // Логирование в Sentry, Crashlytics и т.д. console.error('ErrorBoundary caught:', error, info); } render() { if (this.state.hasError) { return this.props.fallback || ( <View style={styles.container}> <Text>Something went wrong</Text> <Button title="Try again" onPress={() => this.setState({ hasError: false })} /> </View> ); } return this.props.children; } } // Использование в _layout.tsx export default function RootLayout() { return ( <ErrorBoundary> <QueryClientProvider client={queryClient}> <Stack /> </QueryClientProvider> </ErrorBoundary> ); }
Error Boundary — это try/catch для React компонентов. Ловит ошибки рендеринга. В iOS аналог — это NSSetUncaughtExceptionHandler + fallback UI. Стратегия: размести ErrorBoundary на корневом уровне (ловит всё) и на уровне отдельных экранов/фич (изоляция ошибки в одном разделе не убивает всё приложение).
Error Boundary НЕ ловит: (1) ошибки в event handlers (onPress, onChange) — для них используй try/catch, (2) асинхронные ошибки (setTimeout, fetch) — для них React Query.onError или собственный try/catch, (3) ошибки на сервере (SSR). Error Boundary ловит только ошибки в render-фазе. Это важное ограничение — многие ожидают, что он ловит ВСЁ, как глобальный exception handler.
Хорошая стратегия: Error Boundary + React Query error handling + try/catch в event handlers. Вместе они покрывают все случаи. React Query уже встроенно обрабатывает сетевые ошибки (retry, error state). Error Boundary ловит ошибки рендера. Try/catch — для остального. Это более гранулярно, чем глобальный crash handler в iOS.

📝 TypeScript: Типы для React Native

Если ты привык к Swift, TypeScript покажется знакомым, но с важными отличиями. Swift — это language с типами первого класса: struct, class, enum, protocol. TypeScript — это надстройка над JavaScript, и типы существуют только при компиляции (стираются в runtime). Это значит: нет is / as для runtime type checking, нет Codable для автоматической десериализации. Но TypeScript даёт то же, что Swift — автодополнение, ошибки при компиляции и документацию через типы.

В React Native типизация особенно важна в четырёх местах: (1) модели данных (Team, Player), (2) props компонентов, (3) параметры навигации, (4) возвращаемые значения хуков. Ниже — конвенции для каждого.

// types/team.ts export interface Team { id: string; name: string; country: string; founded: number; stadium: string; players?: Player[]; } export interface Player { id: string; name: string; position: 'GK' | 'DF' | 'MF' | 'FW'; number: number; } // API responses export interface ApiResponse<T> { data: T; meta?: { total: number; page: number; }; } // Component props export interface TeamRowProps { team: Team; onPress: (team: Team) => void; showFavorite?: boolean; } // Screen params (Expo Router) export type TeamDetailParams = { id: string; tab?: 'info' | 'players' | 'matches'; }; // Hook return types export interface UseTeamsResult { teams: Team[]; isLoading: boolean; error: Error | null; refetch: () => void; }

Обрати внимание на interface vs type: в TypeScript они почти взаимозаменяемы. Конвенция: interface для объектных форм (props, модели), type для union-ов и сложных типов. ApiResponse<T> — generic, как Result<T, Error> в Swift. position: 'GK' | 'DF' | 'MF' | 'FW' — это string literal union, аналог enum Position: String в Swift, но без отдельного типа.

В Swift ты привык к Codable для маппинга JSON → struct. В TypeScript нет runtime проверки типов: если API вернул неправильный JSON, TypeScript НЕ выдаст ошибку — данные просто будут не того типа. Для runtime-валидации используй библиотеку Zod: const TeamSchema = z.object({ id: z.string(), ... }). Это как Codable, но явный и с кастомными ошибками.
Частая ошибка: определять generic компоненты без правильной типизации событий. Нативные события в React Native типизированы: onPress: (event: GestureResponderEvent) => void, onChangeText: (text: string) => void. Не используй any для event handlers — это убивает весь смысл TypeScript. Также: для платформо-специфичного кода используй Platform.OS с типами 'ios' | 'android'.

🎯 Discriminated Unions — типизация состояний

Discriminated unions — это ответ TypeScript на Swift enums with associated values. В Swift enum LoadingState<T> с case-ами — это единый тип, где каждый вариант может нести свои данные. В TypeScript аналогичная концепция, но реализована через union объектных типов с общим "дискриминантом" — полем, по которому TypeScript различает варианты.

Почему это важно для React? Потому что типичный анти-паттерн — хранить loading, error и data как отдельные boolean/nullable поля: isLoading: boolean, error: Error | null, data: T | null. Это позволяет невозможные состояния: isLoading: true, error: someError — одновременно загрузка и ошибка. Discriminated union делает невозможные состояния невыразимыми — если status === 'loading', поля data и error не существуют.

🍎 Swift enum with associated values
enum LoadingState<T> { case idle case loading case success(T) case error(Error) } // Использование switch state { case .idle: showPlaceholder() case .loading: showSpinner() case .success(let data): showData(data) case .error(let error): showError(error) }
⚛️ TypeScript discriminated union
type LoadingState<T> = | { status: 'idle' } | { status: 'loading' } | { status: 'success'; data: T } | { status: 'error'; error: Error }; // Использование function renderState(state: LoadingState<Team[]>) { switch (state.status) { case 'idle': return <Placeholder />; case 'loading': return <Spinner />; case 'success': return <TeamList teams={state.data} />; // TypeScript знает что data есть! case 'error': return <Error message={state.error.message} />; } }

TypeScript выполняет exhaustive checking — если ты не обработал все варианты в switch, компилятор выдаст ошибку. Для этого добавь default: const _exhaustive: never = state; return _exhaustive; в конец switch. Тип never означает "этот код никогда не должен выполняться". Если добавишь новый вариант в union и забудешь обработать — TypeScript поймает это при компиляции. В Swift это работает из коробки для enum.

TypeScript автоматически сужает тип внутри switch case (type narrowing). Как pattern matching в Swift. В case 'success' TypeScript знает, что state.data существует и имеет тип T. В case 'error' — что state.error существует. Обращение к state.data в case 'loading' вызовет ошибку компиляции — невозможное состояние невыразимо.
React Query уже использует этот паттерн внутри: useQuery возвращает объект с полями status: 'pending' | 'error' | 'success'. Вместо проверки isLoading && !error && data можно проверять status === 'success' — и TypeScript автоматически типизирует data как non-null. Используй это вместо отдельных boolean-проверок.

🧪 Testing: Jest + React Native Testing Library

Философия тестирования в React принципиально отличается от iOS. В XCTest ты часто тестируешь реализацию: проверяешь, что label содержит текст, что кнопка вызывает метод, что ViewModel имеет определённое свойство. React Native Testing Library (RNTL) продвигает другой подход: тестируй поведение, а не реализацию. Тест должен взаимодействовать с компонентом так, как пользователь — нажимать, вводить текст, читать результат.

Практическое отличие: в iOS тесте ты обращаешься к sut.nameLabel.text — деталь реализации (label может быть заменён на другой элемент). В RNTL ты вызываешь screen.getByText('Real Madrid') — проверяешь, что текст видим пользователю, не важно в каком элементе. Это делает тесты устойчивыми к рефакторингу: замена Text на Animated.Text не сломает тест.

🍎 XCTest
import XCTest @testable import MyApp class TeamRowTests: XCTestCase { func testDisplaysTeamName() { let team = Team(id: "1", name: "Real Madrid") let sut = TeamRow(team: team) XCTAssertEqual(sut.nameLabel.text, "Real Madrid") } func testFavoriteButtonTapped() { var tapped = false let sut = TeamRow( team: mockTeam, onFavorite: { tapped = true } ) sut.favoriteButton.sendActions(for: .touchUpInside) XCTAssertTrue(tapped) } }
⚛️ Jest + RNTL
import { render, fireEvent, screen } from '@testing-library/react-native'; import { TeamRow } from './TeamRow'; describe('TeamRow', () => { const mockTeam = { id: '1', name: 'Real Madrid', country: 'Spain' }; it('displays team name', () => { render(<TeamRow team={mockTeam} onPress={jest.fn()} />); expect(screen.getByText('Real Madrid')).toBeTruthy(); }); it('calls onPress when tapped', () => { const onPress = jest.fn(); render(<TeamRow team={mockTeam} onPress={onPress} />); fireEvent.press(screen.getByText('Real Madrid')); expect(onPress).toHaveBeenCalledWith(mockTeam); }); });

Что тестировать: (1) рендеринг — правильные данные отображаются, (2) взаимодействие — нажатия вызывают правильные callback-и, (3) условный рендеринг — loading/error/empty показывают правильный UI, (4) accessibility — элементы имеют правильные роли и метки. Чего НЕ тестировать: стили (они визуальные), внутреннее состояние (оно деталь), порядок вызовов хуков (это ограничение React).

В iOS юнит-тесты запускаются в Simulator — медленно. Jest запускает тесты в Node.js — быстро (секунды, не минуты). RNTL не рендерит на экране: он создаёт виртуальное дерево компонентов и проверяет его структуру. Это аналог snapshot testing, но с семантическими запросами вместо сравнения пикселей. Скорость тестов в React — главное преимущество для TDD.
jest.fn() — это мок-функция (аналог Mock в XCTest). Она запоминает все вызовы: expect(onPress).toHaveBeenCalledTimes(1), expect(onPress).toHaveBeenCalledWith(mockTeam). Не забывай передавать jest.fn() для обязательных callback props — иначе компонент выбросит ошибку при попытке вызвать undefined.

🪝 Testing Custom Hooks

Тестирование хуков — уникальная задача, которой нет в iOS. Хук — это функция, но она не может вызываться вне React-компонента (из-за правил хуков). Поэтому RNTL предоставляет renderHook — утилиту, которая создаёт минимальный компонент-обёртку, вызывает хук, и возвращает доступ к его результату через result.current.

Если хук зависит от Context (React Query, Auth, и т.д.), нужно обернуть его в соответствующие Provider-ы через параметр wrapper. Это как dependency injection в iOS тестах: ты создаёшь тестовое окружение с моками. retry: false в QueryClient — чтобы тесты не делали повторных попыток при ошибках, иначе они будут медленными.

import { renderHook, waitFor } from '@testing-library/react-native'; import { QueryClient, QueryClientProvider } from '@tanstack/react-query'; import { useTeams } from './useTeams'; // Wrapper для React Query const createWrapper = () => { const queryClient = new QueryClient({ defaultOptions: { queries: { retry: false } } }); return ({ children }) => ( <QueryClientProvider client={queryClient}> {children} </QueryClientProvider> ); }; describe('useTeams', () => { it('fetches teams on mount', async () => { // Mock API jest.spyOn(api, 'fetchTeams').mockResolvedValue([ { id: '1', name: 'Real Madrid' }, { id: '2', name: 'Barcelona' }, ]); const { result } = renderHook(() => useTeams(), { wrapper: createWrapper(), }); // Initial state expect(result.current.isLoading).toBe(true); expect(result.current.teams).toEqual([]); // After fetch await waitFor(() => { expect(result.current.isLoading).toBe(false); }); expect(result.current.teams).toHaveLength(2); expect(result.current.teams[0].name).toBe('Real Madrid'); }); });

waitFor — это аналог XCTestExpectation + waitForExpectations(timeout:) в XCTest. Он повторяет проверку до тех пор, пока она не пройдёт (или не наступит таймаут). Это нужно для асинхронных операций: загрузка данных не мгновенна, и нужно дождаться, пока хук обновит своё состояние. Без waitFor тест проверит начальное состояние (loading: true) и завершится до получения данных.

jest.spyOn(api, 'fetchTeams').mockResolvedValue([...]) — мокает API-вызов. mockResolvedValue для Promise (async), mockReturnValue для синхронных функций. В iOS ты бы создал протокол APIClient и инжектил мок через init. В JS мокание модулей через jest.spyOn проще — не нужны протоколы и фабрики. Но помни: мок нужно сбрасывать между тестами через jest.restoreAllMocks() в afterEach.
Если хук обновляет state (вызывает setState), и ты проверяешь результат вне waitFor или act(), React выдаст предупреждение "An update was not wrapped in act(...)". act() — это обёртка, которая гарантирует, что все обновления состояния обработаны перед проверкой. waitFor уже включает act() внутри, поэтому для async хуков используй waitFor, а для синхронных действий — act(() => { result.current.toggleFavorite('1'); }).
Тестирование хуков отдельно от компонентов — это аналог тестирования ViewModel в изоляции от View. Ты проверяешь логику (загрузка, обработка ошибок, вычисления), не зависимая от UI. Компоненты тестируешь через render + fireEvent — это integration test. Такое разделение даёт высокое покрытие с минимальным количеством тестов.
